# Week7.5 LoRA Ranking Training Workflow

week7.5 的训练型 framework 不是为了把项目重新拆成很多新方向，而是为了在已经完成 week5、week6、week7 的前提下，给 version pony 增加一个真正可训练、可评估、可进入现有 baseline 坐标系的主方法对象。这个阶段的边界非常明确：只训练 `Beauty + candidate ranking + Qwen3-8B + LoRA` 这一条主线，不扩成三任务联合训练，不提前打开四域矩阵，也不把 pointwise 和 pairwise 从既有角色中挪走。pointwise 继续承担 diagnosis、uncertainty elicitation 和 calibration 的 supporting layer；pairwise 继续承担 mechanism line 和局部偏好解释；只有 candidate ranking 是 week7.5 第一轮训练的主战场。

当前训练入口是 `main_lora_train_rank.py`，主配置是 `configs/lora/qwen3_rank_beauty_framework_v1.yaml`。这条训练路径已经被刻意拆成三种层级。第一层是 `--startup_check`，它不进入真正训练，而是先完成训练输入、验证输入、评估输入、prompt、base model config、adapter 输出目录和 framework 输出目录的完整校验，同时写出 `outputs/summary/week7_5_startup_check.json` 和 `outputs/summary/week7_5_dataset_preview.csv`。第二层是 `--dry_run`，它继续沿用 startup check 之后的骨架流程，但只落训练状态、训练摘要、example preview 和 framework manifest，不宣称本地已经训练出了真实 adapter。第三层才是常规训练模式，也就是直接运行 `python main_lora_train_rank.py --config ...`，由此进入真正的 LoRA ranking training。这样组织的目的，是把“能不能安全启动训练”和“训练本身是否已经真实完成”分开，而不是让这两件事在一个入口里混淆。

训练数据并没有重新发明新的项目格式，而是直接复用当前 Beauty 主域 candidate ranking 的 processed 数据。`src/training/rank_dataset.py` 负责把 ranking 样本转成 supervised examples：prompt 仍然来自现有 `prompts/candidate_ranking.txt` 和 prompt builder，target ranking 则由 `positive_item_id` 和 `candidate_item_ids` 构造成一个最小可行的 training target。这个设计故意保持克制。week7.5 的第一版 framework 不是为了立即上复杂多头结构、多任务联合 loss 或 uncertainty-aware neural objective，而是为了先把 trainable ranking framework 站住。换句话说，现有 pointwise calibrated uncertainty 和 structured risk strongest hand-crafted baseline 会继续作为支持信号和 compare 对照存在，而不是在这一轮就把训练主循环改造成三任务大杂烩。

评估入口由 `main_eval_lora_rank.py` 独立承担，它和训练入口严格分离。这一点吸收的是 DecodingMatters 那类工作在工程组织上的优点，而不是照搬方法本身。训练阶段负责生成 adapter 与训练状态，评估阶段负责用 `base + adapter` 在标准 Beauty ranking eval split 上产出 `rank_predictions.jsonl`、`ranking_metrics.csv` 和 `framework_eval_summary.csv`。这种分离非常重要，因为它让后续服务器 demo 的流程变成一条清晰的 closure path：训练负责产出 adapter，评估负责产出 framework result，compare 负责把 framework 放回现有 baseline 坐标系。如果之后需要重评估、改 compare、或者只替换 eval split，就不必再次触碰训练脚本。

从 day3 开始，week7.5 的本地工程已经不再是三个孤立入口，而是一条真正的 closure path。当前约定的标准顺序是 `startup_check -> train -> eval -> compare`。对应的关键产物包括：`framework_run_manifest.json`、`training_summary.csv`、`framework_eval_summary.csv`、`week7_5_framework_compare.csv` 和 `week7_5_framework_compare.md`。这些文件的作用并不相同。manifest 记录 framework 自己的运行状态链条，training summary 记录训练过程与 adapter 产物，eval summary 记录 framework 在 candidate ranking 主任务上的指标出口，而 compare 则把 framework 放到 direct ranking、structured risk strongest hand-crafted baseline 和 literature-aligned baselines 的共同坐标系中。这样一来，后续服务器只要沿着既定路径回填真实结果，本地工程就不需要再补接线。

day4 在这条路径上进一步收紧了 compare 层的角色定义。当前 framework compare 已经不只是一个单独的 csv，而是开始显式对齐现有三层 baseline 坐标系。具体来说，`direct ranking` 被固定为 `same_task_reference`，`structured risk current best family` 被固定为 `strongest_handcrafted_baseline`，`literature-aligned ranking baselines` 保持外部或 task-aligned reference 的身份，而 `LoRA-adapted framework` 则作为 `trainable_framework_main_candidate` 进入 compare。对应地，summary 层除了 `week7_5_framework_compare.csv` 和 `week7_5_framework_compare.md`，现在还会额外生成 `week7_5_baseline_matrix.csv` 和 `week7_5_baseline_matrix.md`，把 framework compare 正式桥接进 `week7_day4_baseline_matrix.csv` 那条已有的 baseline matrix 语义里。这意味着后续服务器真实结果一旦回来，不仅会自然落进 framework compare，也会更自然地进入更大的 baseline matrix，而不是再次开一张孤立 compare 表。

因此，后续服务器 demo 的最小真实闭环应当严格沿着下面这条路径执行，而不是临时重新拼命令。第一步：`python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml --startup_check`。第二步：`python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml`。第三步：`python main_eval_lora_rank.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml --overwrite`。第四步：`python main_compare_framework.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml`。如果后续需要按 batch 方式组织服务器 demo，则继续使用 `configs/batch/week7_5_rank_train.yaml` 作为 handoff 配置。到了这个阶段，week7.5 的重点已经不再是补脚手架，而是等待真实服务器结果回填到已经成形的 artifact chain 和 baseline bridge 中。

week7.5 第五天在这条路径后面再补上一层 speed upgrade 约束：即便后续为了进入 week8 需要处理 ranking 吞吐，也不能通过删 pointwise、pairwise、calibration 或 baseline 来换速度。因此在服务器第一次真实闭环之后，还应继续执行 `python main_prepare_speed_upgrade.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml`，把 ranking bottleneck、输出收紧、batch_generate 校验、减少重复加载、shard/resume 和后续 serving 预留这几件事统一写入 `outputs/summary/week7_5_speed_upgrade_plan.csv` 与 `outputs/summary/week7_5_speed_upgrade.md`。这样一来，week7.5 的闭环就不只是一条训练链，而是一条同时包含 framework compare、baseline bridge 和执行升级的完整入场路径。
