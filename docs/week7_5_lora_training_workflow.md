# Week7.5 LoRA Ranking Training Workflow

week7.5 第二天的重点，不是宣布“以后可以训练”，而是把 candidate ranking 主任务上的 LoRA training 变成一个真正可启动、可记录、可接后续服务器 demo 的工程入口。当前项目已经拥有 pointwise diagnosis、pairwise mechanism 和 candidate ranking main decision 这三层任务结构，也已经拥有 Qwen3-8B 本地后端、batch/registry、medium-scale server 结果和三层 baseline 坐标系。因此这一阶段不再去开新的任务边界，而是把 trainable framework 明确压在 candidate ranking 主任务上，并且让训练逻辑、评估逻辑、状态记录和 compare 入口都落到统一路径下。这里最重要的一点是，pointwise 和 pairwise 并没有被删除；它们只是继续留在 supporting layer，而不是被拉进 day2 的训练主循环。

当前训练入口是 `main_lora_train_rank.py`，主配置是 `configs/lora/qwen3_rank_beauty_framework_v1.yaml`。这个入口现在支持两种启动模式。第一种是 `--startup_check`，它不会进入真正训练，而是先校验 train/valid/eval 路径、prompt 路径、基础模型路径、target modules、输出目录、样本加载和 prompt 构造是否成立，并把结果写到 `outputs/summary/week7_5_startup_check.json`。同时，它还会把 train split 和 valid split 的数据摘要写入 `outputs/summary/week7_5_dataset_preview.csv`，让后续上服务器之前就能知道当前 Beauty ranking 数据在样本数和候选规模上的基本情况。第二种是常规训练模式，即直接执行 `python main_lora_train_rank.py --config ...`。在这种模式下，入口会先重复 startup check 所需的准备工作，再进入真正的 LoRA training。无论是否真正训练，训练状态都会被写入 `outputs/summary/week7_5_train_status.csv`，而训练摘要则会写到 `artifacts/logs/qwen3_rank_beauty_framework_v1/training_summary.csv`。

LoRA 训练数据并没有重新发明新格式，而是直接复用当前 Beauty processed 目录下的 ranking 数据。`src/training/rank_dataset.py` 负责把现有 ranking 样本转成 supervised examples：prompt 仍然来自现有 `prompts/candidate_ranking.txt` 和现有 prompt builder，target ranking 则由 `positive_item_id` 和 `candidate_item_ids` 构造而成。当前第一版 target 逻辑非常克制，只要求 positive item 在训练目标里排到最前，然后保持其他候选的稳定顺序，不额外引入 pointwise/pairwise 联合标签或复杂多头 supervision。这个选择不是因为项目不做 uncertainty-aware 研究，而是因为 week7.5 的任务是先让 trainable ranking framework 站住，再去讨论如何把 uncertainty 更深地融进训练。现有 pointwise calibrated uncertainty 路径和 structured risk strongest hand-crafted baseline 已经通过 config 保留在 `support_signals` 里，它们会在后续 compare 阶段与 trainable framework 发生关系，而不是在 day2 里直接把训练循环搞成多任务大杂烩。

评估入口则由 `main_eval_lora_rank.py` 单独承担。它和训练入口分离，这是 week7.5 明确吸收 DecodingMatters 工程组织方式的地方之一。训练完成后，评估入口会读取 base model config，再把 `adapter_path` 注入运行时配置，用 `base + adapter` 在标准 Beauty ranking eval split 上生成 framework predictions，并直接写出 `outputs/beauty_qwen3_rank_framework_v1/predictions/rank_predictions.jsonl` 与 `outputs/beauty_qwen3_rank_framework_v1/tables/ranking_metrics.csv`。这种组织方式的优点是非常明确：训练阶段负责产出 adapter，评估阶段负责产出 framework result，compare 阶段负责把 framework 放回现有 baseline 坐标系。这样以后如果需要重评估、重比较、或更换 eval split，而不重新训练，就不必再去碰训练脚本。

day2 还刻意补了一个后续服务器 demo 友好的 batch 配置草图 `configs/batch/week7_5_rank_train.yaml`。这份 batch config 并不意味着现在就要在本地直接跑完训练，而是把后续最小闭环的三步明确写成统一口径：第一步 `startup_check`，第二步 `train`，第三步 `eval`。等你之后统一把 week7.5 的内容带到服务器上时，不需要再重新组织这三步，也不用再临时思考命令应该怎么写。换句话说，day2 的价值不是“本地已经训练出结果”，而是“训练流程已经正式进入项目工程结构，并且具备可启动、可记录、可批处理、可后续 compare 的形态”。

week7.5 第三天进一步把这条路径收紧成真正的最小闭环接口，而不是继续停留在“可以训练”的口头层面。当前的闭环顺序已经固定成四段。第一段仍然是 `startup_check`，它负责回答“能不能安全启动”；第二段是 `train`，它负责回答“adapter 能不能生成并落到标准目录”；第三段是 `eval`，它负责回答“base + adapter` 能不能在标准 ranking 指标下产出 framework metrics”；第四段是 `compare`，它负责回答“framework 是否已经进入 direct ranking、structured risk strongest hand-crafted baseline 与 literature-aligned ranking baseline 的统一坐标系”。这意味着后续服务器 demo 不再是三四个互相松散的脚本，而是一条有明确阶段产物的 closure path。对应该路径，项目当前会产出 `framework_run_manifest.json`、`framework_eval_summary.csv`、`week7_5_framework_compare.csv` 和 `week7_5_framework_compare.md`。前两者负责把 framework 自己的训练与评估证据固定下来，后两者负责把 framework 放回 compare 坐标系。

因此，后续上服务器时的最小真实闭环应该严格顺着下面这条路径跑，而不是临时插命令。第一步：`python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml --startup_check`。第二步：`python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml`。第三步：`python main_eval_lora_rank.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml --overwrite`。第四步：`python main_compare_framework.py --config configs/lora/qwen3_rank_beauty_framework_v1.yaml`。同时，`configs/batch/week7_5_rank_train.yaml` 现在已经把这四段都写进 batch 级 handoff 草图里，所以后续服务器 demo 的重点不再是回头补本地工程，而是沿着既定路径真正把 adapter、framework metrics 和 compare 结果跑出来。
