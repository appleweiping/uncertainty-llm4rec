# Week7.5 Framework Scope

Week7.5 不是对 week7 的返工，也不是提前把 week8 的多域统一矩阵拉进来。它的任务，是在现有 version pony 多任务结构、structured risk current best family、Qwen3 本地执行后端、batch/registry 和 baseline matrix 都已经站住之后，正式插入一个训练型 framework 定型阶段。当前项目并不缺 pointwise、pairwise 和 candidate ranking 这三层任务的结构，也不缺 Qwen3-8B 本地后端和 Beauty medium-scale 的真实执行证据。真正还没有落下来的，是一个可训练、可评估、可放进现有 baseline compare 坐标系的主 framework。因此 week7.5 day1 的工程目标，不是继续搭零碎脚手架，而是把 Beauty 单域、candidate ranking 主任务、Qwen3-8B + LoRA 的训练与评估入口正式拆开，并把它与现有 strongest hand-crafted baseline、已有三层 baseline 体系和后续服务器执行路径接起来。

这一阶段为什么先做 candidate ranking trainable framework，而不是 pointwise 或三任务联合训练，需要在工程边界上写死。因为 version pony 从 week5 开始就已经把三层任务的角色定义得很清楚：pointwise 是 diagnosis layer，承担 uncertainty elicitation、校准和误差分析职责；pairwise 是 mechanism line，承担局部偏好与机制解释职责；candidate ranking 才是 main decision layer。真正值得训练的第一轮主方法，必须先服务于 main decision layer，而不是在支持层先开一个训练分支，更不能把 pointwise、pairwise 和 ranking 三条线同时拉进训练主循环，把复杂度、数据接口和验证成本一下子推爆。week7.5 因而明确采用“ranking 主任务训练，pointwise/pairwise 支持信号保留”的方案：训练对象只固定为 candidate ranking，pointwise calibrated uncertainty 和 pairwise mechanism evidence 继续作为 supporting layer 存在，但不进入 day1 的训练主循环。

week7.5 也不会偷偷降级 structured risk current best family。相反，它在这一阶段应当被正式重命名为 training-stage strongest hand-crafted baseline。这个定位非常重要，因为它决定了 trainable framework 第一轮到底要回答什么问题。week6 和 week7 的真实结果已经说明，structured risk 在 Beauty + Qwen3 本地 medium setting 下并不是一个临时过渡公式，而是当前 strongest hand-crafted uncertainty-aware main line。week7.5 的 LoRA framework 不是为了推翻这个结论，而是为了继续问一个更强的问题：在 strongest hand-crafted baseline 已经站住的情况下，引入 trainable relevance adaptation 之后，是否还能进一步提升 candidate ranking 主任务表现，而且这个提升是否值得其训练复杂度。只有把 structured risk 写成必须正面对比的强基线，而不是“旧方法遗留物”，后续 week8 才会有稳的叙事中心。

week5、week6、week7 累积下来的 baseline 体系在 week7.5 里也不会被推翻，而是会被重新定位为 trainable framework 的正式对照层。第一层是 uncertainty-source baseline，例如 raw verbalized、calibrated、self-consistency、fused，这一层回答的是 uncertainty signal 到底如何定义、哪些 source 更稳。第二层是 decision-formulation baseline，例如 direct candidate ranking、plain pairwise aggregation，以及 structured risk current best family 和 retained exploratory families，这一层回答的是在同任务 setting 下 uncertainty-aware decision 是否有意义。第三层是 week7 已经接进来的 literature-aligned baseline，这一层回答的是项目的方法是否已经能在同候选集、同划分、同指标下与接近文献范式的外部对照比较。week7.5 的 trainable framework 不会替代这三层 baseline，而是要正式接进这张已经存在的 compare 坐标系，成为它们之上的一个新的 trainable 主线候选。

这也是为什么 week7.5 特别强调 train / evaluate 分离的工程形态，而不是再写一个“训练完顺手评估”的混合脚本。学长给出的 DecodingMatters 参考，真正值得吸收的不是它的方法本身，而是它在训练组织上的规范性：训练入口和评估入口分离，配置驱动，adapter 产物、训练日志、评估结果和 compare summary 各自有清楚目录，不把训练阶段逻辑和评估阶段逻辑揉成一坨。version pony 在 week5 到 week7 已经把 inference/eval/batch 做出了比较清楚的工程边界；week7.5 的任务，是让 trainable framework 也遵守同样的组织原则，而不是发明一个与现有工程生态脱节的新脚本壳子。因此 day1 新增的 LoRA ranking config、训练入口、评估入口、adapter 输出目录和 framework compare schema，既是技术骨架，也是后续避免项目再次发散的边界声明。

当前这版 week7.5 day1 工程骨架遵守三个刻意的限制。第一，它只固定 Beauty 单域，不扩四域，因为现在最关键的是先把主 framework 定型，而不是拿一个未定型方法去做横向放大。第二，它只固定 Qwen3-8B 本地 backbone，不再横向多开其他 backbone，因为当前阶段最需要的是把 trainable framework 跑通，而不是把训练线和 generalization 线混在一起。第三，它先把 uncertainty 作为 support signal 和 compare axis 保留下来，而不是在第一版 framework 里急于把 uncertainty fully neuralized。这样做并不是保守，而是为了让 week7.5 结束时，项目能稳稳拿到一个“可训练、可评估、可对照”的中心方法对象，再把它带进 week8 的 task、model、domain、robustness 五维统一矩阵。
