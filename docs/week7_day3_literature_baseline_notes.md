# Week7 Literature-Aligned Baseline Notes

Week7 starts moving the project from internal method comparison toward literature-aligned defense. The current goal is deliberately narrow: add a first runnable baseline group that shares the same candidate sets, splits, and metrics as the project tasks. This is not a claim that OpenP5, EvaLLMasRecommender, or Exp3rt has been fully reproduced. The alignment is at the task-organization level: unified schema, same evaluation boundary, candidate ranking as a listwise task, and pairwise preference as a mechanism-level task.

The first ranking group includes candidate-order, popularity-prior, longtail-prior, and history-overlap rankers. Candidate-order is the candidate-list prior. Popularity and longtail priors expose common recommendation controls and exposure tradeoffs. The history-overlap ranker is a lightweight content-similarity baseline: it scores each candidate by token overlap between the user history and candidate text, without uncertainty and without LLM generation. This makes it a useful first task-aligned baseline because it can run under the same candidate set and ranking metrics before heavier model baselines are added.

The first pairwise group includes history-overlap and popularity-prior pairwise preference baselines. They use the same generated pairwise samples as the LLM pairwise task and output the same `preferred_item_pred` schema, so `main_eval_pairwise.py` style metrics remain applicable. Their role is to test whether a very simple non-generative preference heuristic already explains part of the pairwise signal.

The local model route is kept explicit but not expanded into training here. The summary schema includes `model_family` and `adapter_path`; the current value is base-only Llama 3.1 8B Instruct, and `adapter_path` is intentionally empty. If Week8 or reviewer-gap experiments require adaptation, the same baseline interface can represent `base+adapter` without changing the task schema. LoRA and vLLM remain later execution-layer extensions, not Week7 baseline-search goals.

The first summary is written to `outputs/summary/week7_day3_literature_baseline_summary.csv`. Because the compact ranking candidate set currently contains fewer than ten items per event, HR@10 is not the most discriminative ranking signal. Interpret NDCG@10 and MRR first, and treat HR@10 mainly as a schema sanity signal in this compact stage.
