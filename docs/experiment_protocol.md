# Experiment Protocol

This protocol defines how real TRUCE-Rec experiments should be run after the
smoke-only phases. It is not a record of completed experiments and contains no
paper results.

## Datasets

Planned dataset tiers:

- `tiny`: committed fixture data for tests and smoke runs only.
- MovieLens-style: local sanity and pilot recommendation experiments.
- Amazon Reviews-style multi-domain: primary full-data setting after raw files,
  licenses, and preprocessing are confirmed.
- Generic CSV/JSONL: adapter path for additional domains.

No dataset is downloaded by this protocol.

## Split strategy

Supported split strategies:

- leave-one-out next-item split;
- temporal split with train/validation/test boundaries;
- user-stratified split if later adapters require it.

The split used by each run must be recorded in `resolved_config.yaml`.
Interactions after the example target timestamp must never be used as history,
features, training evidence, popularity counts, or prompt context.
Real experiment templates must include an explicit `split_strategy` section with
the strategy name, timestamp handling, future-interaction policy, history policy,
and repeat-target or no-repeat-target slicing policy. If target-excluding
examples are constructed for a prompt or grounding diagnostic, that protocol must
be reported separately and not used as main recommendation accuracy.

## Candidate protocol

Candidate construction must be one of:

- full ranking over the catalog when feasible;
- sampled negatives with a fixed seed and documented sampling pool;
- candidate set loaded from file;
- candidate set generated by a retriever and saved as an artifact.

Comparable baselines and OursMethod must use comparable candidate sets. If a
method requires a different candidate protocol, it must be reported separately.
Real templates must declare the candidate-set saved path or explain how the full
catalog candidate set is reconstructed from `resolved_config.yaml` and the
catalog snapshot. Candidate sets that exclude the held-out target are diagnostic
only; they can test prompt/grounding behavior, but their target correctness must
not be interpreted as recommendation accuracy.

## Negative sampling protocol

Sampled negatives must:

- exclude the target item;
- exclude forbidden future items when the protocol requires it;
- use the configured seed;
- record sample size, sampling pool, and replacement policy;
- be saved or reproducible from the resolved config.

Negative sampling must not use target labels except to remove the positive item
from negatives.

## Data protocol

Every real experiment config must declare raw data paths or `TBD`, processed
data paths, interaction schema, item metadata schema, user/item ID mapping
policy, timestamp handling, candidate-set path, train-only popularity source,
domain field if multi-domain, and repeat-target/no-repeat-target slicing policy.
Popularity buckets, novelty, and long-tail statistics must be computed from the
training split only.

## Full-ranking vs sampled-ranking

Full-ranking results and sampled-ranking results are not directly comparable
unless the paper table explicitly separates them. Full ranking is preferred for
smaller catalogs. Sampled ranking is allowed for large catalogs when full
ranking is infeasible and all compared methods share the same candidate sets.

## Prompt protocol

Prompts may include:

- user history titles available before the target;
- visible non-target candidate titles when the method is candidate-based;
- catalog metadata allowed by config.

Prompts must not include:

- target item title;
- target item ID;
- future interactions;
- target-derived explanations, labels, or correctness signals.

Prompt template IDs and hashes must be saved with predictions.

## LLM provider protocol

Provider modes:

- MockLLM for tests and smoke runs;
- OpenAI-compatible API provider only when config is explicit and the user
  confirms the run;
- local HF causal LM only from an existing local model path unless downloads are
  explicitly allowed;
- LoRA/QLoRA inference or training only after server resources are confirmed.

Raw requests, raw responses, token usage, latency, retries, cache status, and
provider config snapshots should be saved when available.

## Random seeds

Real experiments should use multiple seeds, recorded in config and artifacts.
Smoke configs currently use seed `13`. Real multi-seed templates should not be
converted into paper evidence until all intended seeds finish successfully.

## Metrics

Shared evaluator metrics:

- Recall@K, HitRate@K, MRR@K, NDCG@K, MAP@K if available;
- validity, hallucination, parse success, candidate adherence;
- confidence and calibration metrics;
- coverage, diversity, novelty, long-tail metrics;
- popularity-stratified metrics;
- latency, token, cost, and throughput where available.

## Significance testing

Significance tests should be run only on completed real runs with comparable
splits, candidate sets, metrics, and seeds. Report the test used, paired unit,
number of seeds or examples, correction policy if multiple tests are run, and
the exact metrics files used as input.

## Cost and latency tracking

Each run should save `cost_latency.json` when available. API runs must record
prompt tokens, completion tokens, total tokens, estimated cost when configured,
latency mean/p50/p95, retry counts, and cache-hit rate.

## Output artifact structure

Each run should save:

```text
outputs/runs/<run_id>/
  resolved_config.yaml
  environment.json
  git_info.json if available
  logs.txt
  predictions.jsonl
  metrics.json
  metrics.csv
  cost_latency.json
  checkpoints/
  artifacts/
```

LLM runs should also preserve raw request/response artifacts when the provider
layer exposes them.
The current runner records git commit and branch inside `environment.json`;
standalone `git_info.json` is optional unless a deployment wrapper adds it.

## Leakage avoidance

Mandatory safeguards:

- target item title must never appear in prompts;
- target item ID must never appear in prompts;
- future interactions must not be used;
- train popularity only for popularity buckets, long-tail metrics, and novelty;
- all methods share evaluator and prediction schema;
- baselines and OursMethod use comparable candidate sets;
- confidence and uncertainty policy must not inspect target correctness;
- grounding uses catalog records only.
