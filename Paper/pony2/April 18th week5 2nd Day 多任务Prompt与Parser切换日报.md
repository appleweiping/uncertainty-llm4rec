今天的工作重点，不是继续扩样本，也不是急着把三条任务线全部推到全量推理，而是先把 version pony 在语言接口层真正立住。week5 第一天我已经把 pointwise、candidate ranking、pairwise preference 三层任务的数据接口统一起来了，但那仍然只是“样本可生成”。如果第二天不把 prompt 设计和 parser 边界钉死，那么 ranking 和 pairwise 在后续进入真实 LLM 推理时，很容易因为输出格式漂移、候选集外 item 幻觉、A/B 偏好表达不稳定而导致整条链条在最脆弱的地方失控。也正因为如此，我今天坚持沿用旧项目已经验证过的工程原则，不另起炉灶，而是在 `src/llm/prompt_builder.py` 与 `src/llm/parser.py` 这两个旧核心模块上做系统扩展。这样做的意义在于，pointwise 不会被抛弃，它仍然保留为 diagnostic layer；但 ranking 与 pairwise 也不再停留在概念层，而是和旧主链共享同一套 prompt-builder / parser 工程心脏。

在 prompt 层，我完成了两个新任务接口的正式接入。第一是 candidate ranking prompt，它现在能够把同一用户的历史行为、候选 item 列表、每个候选的 `item_id`、标题、文本和 popularity group 组织成一条明确的 ranking 输入，并且显式要求模型只在给定候选集合内部排序，返回完整候选序列和 top-k 子序列两个层次的结构化字段。第二是 pairwise preference prompt，它要求模型只在两个显式候选之间进行二选一判断，并返回 `preferred_item`、`confidence` 与简短理由。这里我特意没有追求华丽 prompt，而是优先保证任务边界的清晰性，因为顶会级实验里最怕的不是模型偶尔答错，而是 task formulation 自己就模糊。只要 prompt 允许模型在候选集外自由发挥，后面的 ranking evaluation、pairwise aggregation、uncertainty transfer 就都会失去干净前提。因此我今天在 `prompts/candidate_ranking.txt` 和 `prompts/pairwise_preference.txt` 里都额外加入了约束语句，明确要求不要输出 markdown fence，要求返回合法 JSON，并且不得越界到候选集之外。

在 parser 层，我做的不是“写两个能凑合用的新函数”，而是给多任务接口补一套真正能承受输出漂移的恢复逻辑。`parse_candidate_ranking_response()` 现在已经支持三类核心情况：第一类是严格 JSON；第二类是被 markdown code fence 包裹的 JSON；第三类是只返回 top-k 而没有完整 `ranked_item_ids` 的部分结构化输出。除此之外，我额外补了一个非常关键的恢复能力：如果模型没有返回完整 JSON，而是在自然语言或半结构化文本里提到了合法候选 id，parser 现在会按文本中真实出现的先后顺序恢复 ranking，而不是按候选原始顺序硬套。这一点很重要，因为 ranking 输出最危险的错误不是“格式稍乱”，而是“格式一乱就被我们自己解析成伪正确顺序”。另一方面，`parse_pairwise_preference_response()` 现在不仅能吃标准 JSON 与 A/B shorthand，还能够从自然语言表达里恢复偏好，例如“我会选某个 item”“我更偏向 candidate B”“winner is A”这类常见输出，并把其中的 `confidence` 统一归一化到 0-1 区间。与此同时，我保留了显式失败路径：如果输出中偏好项是模糊词、候选外 id，或者前后冲突，parser 不会假装成功，而是返回 `parse_success=False` 和 `ambiguous_preference=True`。

为了让今天的工作不是停留在“代码看起来更完整”，我专门补了一个小样本但有工程意义的验证入口 `main_validate_multitask_prompt_parser.py`。这不是正式推理脚本，而是 prompt / parser 闭环自测脚本。它直接读取昨天已经生成好的 `ranking_valid.jsonl` 和 `pairwise_valid.jsonl` 的真实样本，在真实样本字段之上构造受控响应，覆盖 strict JSON、fenced JSON、top-k only、自然语言包裹输出、以及显式失败样本几类关键格式。我要特别说明的是，我今天没有在这个阶段直接调用线上或本地 LLM 做全量试跑，因为那属于 week5 第三天的 inference 层任务；第二天的边界应该是“先把语言接口层做扎实”，而不是在任务边界还没稳定时就把问题提前推进到 backend。也就是说，今天的验证是 controlled parser self-check，而不是最终模型效果实验。这种划分是刻意的，因为只有把问题拆干净，后面每一步的失败原因才可定位。

从实际产出来看，今天的目标已经达到了一个可接受而且可信的水平。`outputs/summary/week5_day2_parse_success.csv` 里，candidate ranking 在 12 条真实 ranking 样本上的 parse success rate 是 `0.9167`，pairwise preference 在 24 条真实 pairwise 样本上的 parse success rate 是 `0.9583`。ranking 侧唯一保留的失败样本来自我主动设计的候选集外 item 越界案例，因此 `out_of_candidate_item_ratio=0.0833` 与 `illegal_output_ratio=0.0833` 在当前受控自测里是符合预期的；pairwise 侧的唯一失败同样是我主动保留的模糊偏好案例，因此 `ambiguous_preference_ratio=0.0417` 也正好说明失败检测没有被掩盖。更关键的是，这次 pairwise 的自然语言样例不再是假装成 JSON 的“伪自然语言成功例”，而是真实句式的自由文本输出，parser 已经能够从中恢复 `preferred_item`、`confidence` 和理由片段。对我来说，这比把 success rate 再堆高一点更重要，因为它说明 parser 真正开始对付 LLM 常见漂移，而不是只对付理想输入。

我还把 `outputs/summary/week5_day2_prompt_examples.md` 重写成了代表性样例文档，而不是机械堆积一大串 prompt 和响应。里面现在保留的是 ranking 的 strict JSON、top-k only、out-of-candidate failure，以及 pairwise 的 explicit item-id JSON、fenced A/B shorthand、free-form natural language、ambiguous failure 这几类最有说明力的样本。这样做的目的很明确：一方面它是工程日志，后面如果 parser 出现回归，我能快速定位是哪个格式分支坏了；另一方面它也天然适合后续进入 appendix，因为它展示的不是“我们有很多例子”，而是“我们已经明确知道新任务最容易在哪些语言接口形态下漂移，以及系统如何处理这些漂移”。这类文档在顶会项目里价值很高，因为它能把工程严谨性转化成可叙述的研究证据。

除了代码和输出文件本身，我今天还同步更新了 `README.md`，但刻意避免把它写成“Week5 Day2 完成了什么”的流水账。README 现在更高层地说明了一件事：项目已经不只是从 pointwise 往后接 calibration 与 reranking，而是在保留 pointwise diagnostic layer 的前提下，开始拥有 candidate ranking 与 pairwise preference 的任务级语言接口；同时，这些接口不是零散脚本，而是依托共享的 `prompt_builder` 与 `parser` 进入统一框架。这样写既能反映研究主线已经在升级，也不会泄露过细实现细节，更符合科研项目应有的对外表述方式。

如果从 version pony 的 Part5 主线来评价今天的完成度，我认为第二天的验收已经成立。第一，candidate ranking 与 pairwise preference 已经有了明确、可复用、可配置的 prompt 模板；第二，它们的响应不再只是“希望模型按格式回答”，而是已经有了可恢复、可拒绝、可统计的 parser；第三，我已经拿到了结构化 summary 与代表性例子文档，能够定量回答 parse success、非法输出比例、候选集外 item 出现率与模糊偏好比例这几个关键问题。也就是说，多任务框架今天不再只是“样本能构造出来”，而是第一次真正进入了“新任务可以被 LLM 稳定消费并转化成结构化结果”的推理准备状态。

当然，今天也有两个我刻意保留的边界。第一个边界是，我现在还没有把这套 prompt / parser 正式接到 `main_rank.py`、`main_pairwise.py` 的完整推理链路里，因为那是 week5 第三天的任务，应该在 inference 层统一做，而不是今天为了追求“看起来更完整”提前混在一起。第二个边界是，我现在的 parse 成功统计来自 controlled response set，而不是实际线上模型批量输出，因此它说明的是语言接口的工程健壮性，不说明最终真实 LLM 在 ranking/pairwise 上的任务表现。这个边界我必须主动写清楚，否则后面会把 parser 自测误说成任务效果实验，叙事会失真。正因为我把这两个边界写死了，明天进入三条推理主线时，所有问题才有更干净的归因空间。

所以，今天结束时我的结论是：version pony 已经完成了多任务语言接口层的第一轮正式切换。pointwise、candidate ranking、pairwise preference 三层任务不再只是共享一个研究口号，而是开始共享一套可生成 prompt、可恢复输出、可记录失败、可量化成功率的接口体系。对 Part5 来说，这意味着多任务框架已经从“数据层统一”推进到了“语言交互层统一”；对后面的 week5 第三天和第四天来说，这意味着我已经有资格把三条推理线与三套评估线真正接起来，而不是继续在脆弱的输入输出接口上补丁式前进。
