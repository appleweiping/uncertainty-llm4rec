**April 19th week7 1st Day 服务器本地主实验后端与职责分工落地日报**

今天正式进入 week7 的执行层升级。和 week5、week6 不同，今天的重点不是继续发明新的 ranking family，也不是回头修改 Part5 已经收敛出的 structured risk 主线，而是把项目从 API 驱动的小规模实验模式，推进到“API 观测型后端 + 服务器本地 Hugging Face 主实验后端”的双层执行框架。当前服务器已经切换为 root 登录的 AutoDL/SeeTaCloud GPU 实例，本地主实验模型仍然固定为 Llama 3.1 8B Instruct。这个选择的意义不是替换 Part5 的方法结论，而是为 week7 之后的强 baseline、中等规模实验、可恢复批处理和 week8 统一矩阵提供真正可控的执行底座。今天我没有把服务器密码、远程桌面密码、临时 token 或 SSH 访问链接写进任何仓库文件，也不会把这些敏感信息放进 README、配置或日报；顶会级工程首先要能复现，也要能守住凭据边界。

代码层面，今天首先完成了后端抽象的低侵入升级。`src/llm/base.py` 在原有 `generate()` 基础上补入了统一的 `batch_generate()` 默认接口，使 API 后端和本地 HF 后端都能被三条任务线用同一形状调用。随后新增 `src/llm/local_hf_backend.py`，把服务器本地模型加载从空壳推进成可承担主实验的 Transformers 后端。这个后端支持 `model_name_or_path`、`tokenizer_name_or_path`、`device_map`、`dtype`、`batch_size`、`max_new_tokens`、`temperature`、`top_p`、`local_files_only`、`trust_remote_code`、4-bit/8-bit 预留、`adapter_path` 预留和 chat template 格式化。它的定位很明确：先跑 base-only Llama 3.1 8B Instruct，后续如果 week8 或 reviewer-gap 需要，再自然扩展到 LoRA adapter；当前不训练 LoRA，也不启动 vLLM。`src/llm/__init__.py` 同步注册了 `local_hf`、`hf` 和 `transformers` 三个 backend name，使 `main_infer.py`、`main_rank.py`、`main_pairwise.py` 不需要重写任务逻辑，只要通过 `model_config` 切换后端即可。

配置层面，今天新增并规范化了服务器主模型配置。`configs/model/llama31_8b_instruct_local.yaml` 和 `configs/model/local_llm.yaml` 都指向 Llama 3.1 8B Instruct 的服务器本地模型路径，并显式写出 device、device_map、dtype、batch_size、local_files_only 与 generation 参数。这里的模型路径写成服务器侧默认目录，后续如果实际挂载点不同，只需要改 `model_name_or_path` 与 `tokenizer_name_or_path`，不需要改推理入口。为了让三条任务线都能做最小验证，我还新增了 `configs/exp/beauty_llama31_local_pointwise_smoke.yaml`、`configs/exp/beauty_llama31_local_rank_smoke.yaml` 和 `configs/exp/beauty_llama31_local_pairwise_smoke.yaml`。这三份配置都只控制在极小样本，用于服务器上确认 schema、parser、latency 和输出目录，不承担中等规模结果 claim。

为了把 day1 的验收变成可执行检查，而不是停留在口头说明，我新增了 `main_backend_check.py`。这个入口会对 pointwise_yesno、candidate_ranking、pairwise_preference 三类任务构造最小 prompt，调用同一个 `model_config` 后端，并输出 `outputs/summary/week7_day1_backend_check.csv`，字段包括 task、backend、model、load_success、avg_latency、parse_success、parse_mode、error_message 和 raw_response_preview。本地已经完成 dry-run，验证配置、入口和表头可以正常生成；dry-run 不加载模型，因此 load_success 被明确记录为 `not_loaded_dry_run`，不会被误写成服务器真实加载成功。考虑到当前 Codex 环境中 SSH/scp 被沙箱禁用且没有安全的非交互密码注入方式，我没有把密码拼进命令去强连服务器，而是做了不含凭据的端口连通性探测，确认 SSH 端口可达。为了让服务器真实检查可以顺利补上，我新增了 `scripts/week7_day1_server_backend_check.sh`，服务器仓库目录下运行即可执行非 dry-run backend check；设置 `RUN_SMOKE=1` 后还可以连续运行三条 smoke config。

文档层面，今天新增 `docs/week7_local_backend_and_server_workflow.md` 和 `docs/week7_day1_backend_notes.md`，把模型放服务器、代码本地维护、服务器 pull 代码执行、base model 常驻、后续 LoRA 只保存 adapter、必要时再考虑 vLLM 这条工程分工写清楚。README 也同步更新为 week7 的执行层状态，但保持项目级口吻，没有写入任何临时访问凭据或过细命令细节。`Paper/pony2/version pony week7 工作计划.md` 也补入当前实际采用的服务器与模型配置，以及 day1 的真实边界：今天建立后端抽象、配置、workflow、check 和 smoke 入口，不做 medium-scale，不做 LoRA，不启动 vLLM，不扩 week8 矩阵，也不回头改 Part5 的 structured risk current best family。

今天的验证结果需要诚实区分。本地侧已经通过 Python AST syntax check，`main_backend_check.py --dry_run` 已经生成 `outputs/summary/week7_day1_backend_check.csv`，说明 day1 的配置和入口可以被调用。服务器侧目前已经确认 SSH 端口可达，但因为当前执行环境不能安全地进行交互式 SSH 登录，也不能把密码写进命令或脚本，所以 Llama 3.1 8B Instruct 的真实 GPU 加载、三任务非 dry-run backend check 和三条 smoke inference 仍然需要在服务器可交互登录后补跑。这不是结果缺口的掩饰，而是凭据安全与真实执行边界的正常记录。后续一旦进入服务器，只需要在仓库目录运行 `bash scripts/week7_day1_server_backend_check.sh "$PWD"`；如果模型路径、torch、transformers 和依赖都正确，再设置 `RUN_SMOKE=1` 运行三条最小任务验证。

阶段性结论是，week7 day1 已经完成主实验执行层的第一次工程重构。项目不再只有 API 后端，也不再把本地模型写成模糊愿景；现在已经有统一后端抽象、服务器本地 HF backend、Llama 3.1 8B Instruct 主模型配置、三任务 smoke config、backend check 入口、服务器运行脚本和 workflow 文档。今天还没有产出服务器真实推理指标，因此不能把它写成“本地模型已完成实验验证”；但它已经把 week7 后续最关键的执行分工固定下来：API 负责观察和对照，服务器本地 HF 后端负责主实验吞吐，structured risk current best family 继续作为后续验证主线，LoRA 和 vLLM 都作为后续可插拔扩展，而不是 day1 的发散目标。
