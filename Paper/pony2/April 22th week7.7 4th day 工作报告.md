# April 22th week7.7 4th day 工作报告

今天这一版工作的核心不是继续发明新版本，而是把 week7.7 已经跑出来的四域 evidence 正式收口成可以写论文的结果叙事，并且开始认真处理我一直担心的一个问题：我们后面到底怎么和 NH baseline 里的那些论文比，才既公平、又有说服力、又不把自己的方法定位写偏。到今天为止，我已经不再把当前项目理解成“做一个普通的 LoRA ranking baseline”，而是把它收束成一个更清楚的问题：当大模型或强 teacher 可以提供 uncertainty-aware correction 时，小模型什么时候应该学习这些 correction，什么时候应该保留 direct ranking 的稳定性，以及这种 conditional uncertainty distillation 能否在不同域中形成可解释的收益边界。这个问题比单纯追一个全域通杀的版本更接近论文贡献，因为它不只给出一个分数最高的 variant，也解释了为什么某些域里蒸馏会显著成功，某些域里 teacher gap 太小、过度蒸馏反而会伤害 small model。

今天首先完成的是 GitHub 侧的阶段性冻结。我把本地已经从服务器同步回来的真实结果重新复核后，正式生成了 `Paper/pony2/results/week7_7_four_domain_srpd_final_matrix.csv` 和 `Paper/pony2/results/week7_7_four_domain_srpd_final_matrix.md`，并提交到了 GitHub。这个动作的意义不只是“保存文件”，而是把四域结果从终端输出、聊天记录和服务器临时目录里移出来，变成一个可以被论文、报告和后续 compare 共同引用的固定证据源。当前这张表明确记录了四个域的最终状态：Beauty full973 中 `SRPD-v2` 最强，`NDCG@10 = 0.6353658182685563, MRR = 0.518448098663926`；Books small500 中同样是 `SRPD-v2` 最强，`NDCG@10 = 0.7057074243106245, MRR = 0.6117666666666667`；Electronics small500 中 `SRPD-v5` 最强，`NDCG@10 = 0.6621001800786173, MRR = 0.5528333333333333`；Movies small500 中则是 `structured risk/direct` 线仍然最强，`structured risk` 的 `NDCG@10 = 0.5731826885570888, MRR = 0.43910000000000005`，而 SRPD family 内部最好的修复版本是 `SRPD-v4`。这组结果使我能够诚实地写出一个条件式结论，而不是强行包装成“某一个 SRPD 版本在所有域都赢”。

从方法角色上看，今天的四域矩阵把 `SRPD-v2 / v4 / v5` 的位置完全拉清楚了。`SRPD-v2` 是 positive-domain winner，它在 Beauty 和 Books 两个域里都显著优于 direct 和 structured risk，这说明当 teacher pattern 有足够可蒸馏性时，uncertainty-weighted SFT 能够让 Qwen3 small model 学到比直接 ranking 更好的决策偏好。Beauty 上 direct 是 `0.6140307799644382 / 0.4899990211912103`，v2 提升到 `0.6353658182685563 / 0.518448098663926`；Books 上 direct 是 `0.6398271002354915 / 0.5239000000000001`，v2 提升到 `0.7057074243106245 / 0.6117666666666667`。这两个域的结果给论文主线提供了最强的正例：不确定性不是单纯被当成评估指标，而是被真正写进训练样本权重和 teacher target 后，转化成了 small model ranking performance 的收益。相反，Electronics 和 Movies 暴露了另一面：当 teacher-direct gap 很小，或者 teacher correction 本身不够稳定时，直接把 uncertainty-aware teacher 全量蒸馏给 student 会产生负迁移。Electronics 上 v2 掉到 `0.6429579068580471 / 0.5278333333333333`，低于 direct 的 `0.6583007651522705 / 0.5471666666666666`；Movies 上 v2 更明显地掉到 `0.5373477157607484 / 0.39223333333333327`，低于 direct 的 `0.5730601060765828 / 0.43896666666666667`。这不是代码 bug 式的失败，而是方法边界：uncertainty-aware distillation 不是无条件有用，它需要 teacher correction 有足够信息增益。

因此，`SRPD-v4` 和 `SRPD-v5` 的价值不应该被写成“新版本又调参”，而应该写成 failed-domain repair family。`SRPD-v4` 的 gap-gated uncertainty weighting 试图回答一个很明确的问题：当 structured-risk teacher 和 direct ranking 几乎一致时，我不应该继续把 uncertainty weight 放大成强监督，而应该让这些低 gap 样本回落到接近 v1 的稳定训练；只有当 teacher 确实产生有意义 correction 时，才激活更强的 uncertainty weighting。这个设计在 Movies 上是 SRPD family 内部最好的修复结果，从 v2 的 `0.5373477157607484 / 0.39223333333333327` 拉回到 `0.546388555376638 / 0.40436666666666665`，虽然仍然没追上 direct，但已经说明过度蒸馏的问题被缓解。`SRPD-v5` 则进一步加入 direct-anchor target rule：当 structured-risk teacher 没有把 positive item 排得比 direct 更好时，训练目标回落到 direct order，而不是强迫 student 模仿一个并不更优的 teacher。这个设计在 Electronics 上成为当前最强版本，从 direct 的 `0.6583007651522705 / 0.5471666666666666` 提升到 `0.6621001800786173 / 0.5528333333333333`。但 v5 在 Movies 和 Beauty/Books 上都没有超过对应最优版本，这反过来说明 direct anchor 不是万能解，它是一种更保守、更适合特定失败模式的修复机制。论文里最成熟的表达应该是：v2 是可蒸馏域的主胜利，v4/v5 是低 gap 或 direct-anchor 场景下的条件修复，而不是把它们互相替代。

这也让我能更确定地回答“我们的主题是不是偏了”。我认为没有偏，反而比之前更聚焦。早期我们讨论 uncertainty 时，更多是在 pointwise calibration、ECE、Brier、AUROC 和 coverage 这些诊断指标里证明模型确实存在不确定性；后来 structured risk 把不确定性引入 ranking rerank；再后来 SRPD 把不确定性进一步引入 trainable small model 的 teacher-data construction。到今天四域结果出来以后，主题已经不再是浅层的“uncertainty 有用吗”，而是一个更有论文价值的问题：uncertainty-aware teacher signal 在什么域、什么 teacher gap、什么 target stability 下可以被蒸馏成 small-model gain；当它不能被直接蒸馏时，如何通过 gap gate 或 direct anchor 避免错误 teacher 被学生过度吸收。这个问题仍然完全围绕 uncertainty 展开，只是从 diagnostic uncertainty 进入了 conditional distillation policy。换句话说，我们不是在做普通的 recommender LoRA，而是在做 uncertainty-aware recommendation decision 的可训练化，以及它的跨域边界分析。

今天我也开始认真看 NH baseline 目录里的论文，并且把它们先按“可比较性”而不是“名气”来分类。这个分类很重要，因为如果直接把所有论文的 reported SOTA number 放到我们表里比，会非常危险。很多论文的数据集、候选空间、任务定义、输入模态、模型规模、训练目标都不一样，直接横向比 NDCG 数值没有公平性。比如 EMNLP 2024 的 [Decoding Matters](https://aclanthology.org/2024.emnlp-main.589/) 关注的是 LLM recommendation decoding 里的 amplification bias 和 homogeneity issue，它提出 `D^3` 去修正生成解码过程；EMNLP 2024 的 [ELMRec](https://aclanthology.org/2024.emnlp-main.653/) 和 ACL Findings 2025 的 [AGRec](https://aclanthology.org/2025.findings-acl.369/) 更强调 high-order interaction、graph reasoning 与 autoregressive decoder 的结合；ACL Findings 2025 的 [CoVE](https://aclanthology.org/2025.findings-acl.651/) 关注 item vocabulary expansion 和 item ID grounding；ACL 2024 的 [Aligning Large Language Models for Controllable Recommendations](https://aclanthology.org/2024.acl-long.443/) 更偏 instruction following 与 controllability；ACL Findings 2025 的 [iAgent](https://aclanthology.org/2025.findings-acl.928/) 则是 user-agent-platform paradigm，用 agent shield 去保护用户兴趣；NeurIPS 2024 的 [LLM-ESR](https://proceedings.neurips.cc/paper_files/paper/2024/hash/2f0728449cb3150189d765fc87afc913-Abstract-Conference.html) 主要处理 long-tail sequential recommendation，通过 LLM semantic enhancement 缓解长尾用户和长尾 item；ICLR 2024 的 [UniMP](https://proceedings.iclr.cc/paper_files/paper/2024/hash/297a7a54da76d95f02b04d957a53c7ba-Abstract-Conference.html) 是 multimodal personalization 的统一范式；SIGIR 2024 的 [OpenP5](https://www.researchwithrutgers.com/en/publications/openp5-an-open-source-platform-for-developing-training-and-evalua/) 更像 LLM4Rec 训练评测平台；WWW 2025 的 [SPRec](https://researchportal.hkust.edu.hk/en/publications/sprec-self-play-to-debias-llm-based-recommendation/) 用 self-play 和 DPO 去 debias；KDD 2025 的 [Lost in Sequence / LLM-SRec](https://pure.kaist.ac.kr/en/publications/lost-in-sequence-do-large-language-models-understand-sequential-r/) 强调 LLM 对 sequential signal 的理解不足，并用 CF-SRec representation distillation 修补；SIGIR 2025 的 [PAD](https://sigir2025.dei.unipd.it/detailed-program/paper?paper=3416a75f4cea9109507cacd8e2f2aefc) 则用 pre-train、align、disentangle 的方式把 LLM semantic embedding 和 collaborative signal 结合起来。它们都很相关，但不是每一篇都能被放进同一张 “same setting leaderboard”。

因此，我现在对 NH baseline 的比较策略有一个更清楚的三层方案。第一层是 same-schema executable baselines，也就是我们已经在做的内部主表：direct ranking、structured risk、SRPD-v1/v2/v3/v4/v5，以及已有的 popularity/order/history-overlap 等 lightweight baselines。这一层必须使用同一数据 split、同一 candidate set、同一 top-k metric、同一 Qwen3 small-model 或同一 teacher-output schema，因此它是论文主结果表的核心。第二层是 literature-aligned proxy baselines：对于 D3、SPRec、LLM-ESR、PAD、LLM-SRec 这类和我们问题有真实交集但原 pipeline 不完全一致的方法，我不应该声称“完整复现了原论文”，而应该抽取其可对齐的机制做 same-schema proxy。比如 D3/SPRec 可以作为 debias/homogeneity/fairness-oriented decoding 或 preference regularization reference，比较 coverage、head exposure、longtail coverage、out-of-candidate 和 ranking utility；LLM-ESR/PAD/LLM-SRec 可以作为 “LLM semantic enhancement / distillation into smaller recommender” 的外部同类方向，重点比较我们是否也在用 LLM/teacher signal 帮助小模型，但我们的核心区别是 uncertainty-aware teacher gap 与 conditional distillation，而不是纯 semantic embedding alignment；OpenP5/CoVE/AGRec/ELMRec 更适合作为 LLM4Rec grounding、indexing、graph reasoning 的 related baselines 或 optional implementation targets，除非后面我们真的把它们改写到同一 candidate-ranking schema 下。第三层是 reported literature reference，也就是只在 related work 或 appendix 里说明这些论文在各自 setting 下的成果和方向，而不把它们的原始数字直接和我们的四域数值硬比。这样写才严谨，也能避免审稿人抓住 “different datasets / different candidate spaces / different evaluation protocols” 来质疑。

我现在最担心也最需要提前防住的点，是 outer compare 不能变成一个“看上去很多 baseline、实际上不公平”的表。比如 LLM-ESR 和 PAD 多数是在传统 sequential recommendation backbone 或 embedding enhancement 框架中评估，它们可能用 SASRec/GRU4Rec/BERT4Rec 等 backbone，候选集也可能是全 item ranking 或 sampled ranking；而我们当前的强项是 LLM/teacher-driven candidate ranking rerank 和 small-model distillation，在固定 candidate set 上比较 NDCG/MRR/coverage/head exposure。如果我直接把它们 reported NDCG 放进我们的表，会让表面结果更热闹，但逻辑很脆弱。更好的做法是：主表只放 same-schema 结果；外部方法进入表格时必须满足“能在我们的 candidate set、我们的 splits、我们的 metrics 下产生 ranking”的条件；不能满足这个条件的论文，进入 related work comparison 或 conceptual baseline table，说明它解决的是 graph grounding、item indexing、long-tail semantics、controllability、agent protection、explanation alignment 还是 multimodal personalization。这样我们不是逃避 SOTA，而是在建立一个审稿人能接受的公平边界。

基于今天对 NH 论文的初步梳理，我认为我们的论文定位可以这样收口：现有 LLM4Rec 方法大多从更强表示、更好 item grounding、更强 decoding、更强 agent/control、更强 semantic enhancement 出发，而我们从 uncertainty-aware decision reliability 出发。这个角度和它们不冲突，但有足够清晰的差异。D3 和 SPRec 关心 bias/homogeneity/fairness，但不直接建模 teacher correction 是否值得被 student 学；LLM-ESR、PAD、LLM-SRec 关心 LLM semantic knowledge 或 CF representation 如何增强 sequential recommender，但不直接回答 uncertainty-aware teacher signal 在什么条件下会产生负迁移；Aligning/iAgent/UniMP 更关注 instruction/control/agent/multimodal generalized personalization，也不是我们的 same-schema ranking core。我们的贡献可以写成：我把 uncertainty 从诊断指标推进到 ranking decision，再推进到 trainable distillation；我不仅报告了正例域的收益，也系统记录了失败域，并用 v4/v5 解释失败域修复；最终我给出的是一个 conditional SRPD framework，而不是无条件模仿 teacher 的 SFT pipeline。

今天这份 4th day 报告对论文的直接贡献，是把“结果已经很多”变成“结果可以讲清楚”。现在我能明确写出主结果句：在四域 setting 下，SRPD-v2 在 Beauty full973 和 Books small500 上取得最强 trainable performance，说明 uncertainty-weighted teacher distillation 在可蒸馏语义域中有效；在 Electronics small500 上，SRPD-v5 的 direct-anchor repair 超过 direct/structured risk，说明当 teacher correction 不稳定时保留 direct anchor 可以转化成收益；在 Movies small500 上，structured risk/direct 仍然最强，SRPD-v4 是 family 内部最佳修复但仍未反超 direct，说明 uncertainty-aware distillation 的适用性存在域边界。这个结论比“我们的模型全赢了”更诚实，也更像一个能被顶会接受的科学发现，因为它包含成功、失败、修复和边界。

接下来我认为最合理的推进顺序不是再做 v6，而是进入论文收口和 outer compare 设计。第一步，我要把 NH baseline 进一步整理成一个正式的 literature-to-experiment mapping 表，给每篇论文标注：是否 same-schema 可复现、是否只能做 proxy、是否只进入 related work、它与我们的关系是 distillation、decoding debias、semantic enhancement、graph/index grounding、long-tail fairness、controllability 还是 multimodal personalization。第二步，在这个 mapping 里选出少数真正值得实现的 proxy baseline，而不是把所有论文都硬塞进实验。优先级我会放在和我们最接近的三类：distillation/small model 类，如 SLMRec、LLM-SRec、PAD；debias/homogeneity 类，如 D3、SPRec；long-tail semantic enhancement 类，如 LLM-ESR。第三步，再决定 outer compare 的最终表结构：主表放 same-schema 四域结果，辅助表放 literature-aligned proxy，related-work 表解释不可直接复现的强 baseline 为什么不进入主表。这样做的好处是论文不会虚胖，比较也更防审稿质疑。

所以今天的最终判断是：week7.7 的方法探索阶段应该正式停止扩版本，当前结果已经足够支撑一个清晰的论文故事；接下来最重要的是把这个故事写成严谨的实验设计和 baseline comparison protocol。Beauty 已经是完整主域，Books/Electronics/Movies 已经给出跨域边界，SRPD-v2/v4/v5 的角色也已经从“版本竞争”变成“条件机制”。如果后面还要做实验，我会优先服务于论文比较，而不是服务于继续调方法。换句话说，现在项目最需要的不是更花哨的新方法，而是一张审稿人看了也觉得边界清楚、比较公平、贡献站得住的结果体系。

在这个结果体系里，我还必须把 Week1-Week6 已经做过的 calibration、coverage 和 robustness 资产正式补回当前 Week7.7 主线，而不能让当前 SRPD 结果看起来只是在刷 `NDCG@10 / MRR`。因此我进一步生成了 `Paper/pony2/results/week7_7_four_domain_srpd_full_metric_matrix.csv` 和 `Paper/pony2/results/week7_7_four_domain_srpd_full_metric_matrix.md`，把本地已经同步并复核过的四域 `direct / structured_risk / SRPD-v1/v2/v3/v4/v5` 结果从 utility-only matrix 扩展成 full-metric matrix：除了 `HR@10 / NDCG@10 / MRR`，每一行现在都同时保留 `coverage@10`、`head_exposure_ratio@10`、`longtail_coverage@10`、`parse_success_rate` 和 `out_of_candidate_rate`。这一步很重要，因为 SRPD 的论文定位不是单纯“推荐准确率更高”，而是 uncertainty-aware recommendation decision；如果某个版本提升 NDCG 但显著破坏 coverage、长尾覆盖或解析稳定性，它就不应该被无条件写成 clean win。现在这张 full-metric matrix 让后续论文主表可以同时呈现 utility、coverage/exposure 和 output reliability。

同时，我也把 calibration / uncertainty diagnosis 正式接回 Teacher Reliability Layer。`Paper/pony2/results/week7_7_teacher_reliability_summary.csv` 和 `Paper/pony2/results/week7_7_teacher_reliability_summary.md` 已经从四个 DeepSeek pointwise teacher 诊断文件中抽取了 `accuracy`、`avg_confidence`、`Brier`、`ECE`、`MCE` 和 `AUROC`：Beauty 使用 full5838，Books/Electronics/Movies 使用 full3000。这个表的作用不是再单独证明 pointwise 模型好不好，而是解释为什么不同域上的 SRPD 蒸馏行为不同。Beauty/Books 是 positive-domain distillation evidence，Electronics 是 repair-domain evidence，而 Movies 是 failure-boundary evidence。也就是说，早期做的 ECE、Brier、AUROC 和 confidence correctness 没有被丢掉，它们现在被提升成 teacher correction reliability 的证据，用来解释 teacher signal 什么时候值得被 student 学，什么时候会产生负迁移。

最后，我单独写了 `Paper/pony2/results/week7_7_calibration_coverage_robustness_integration.md` 来记录这些早期资产如何进入当前 SRPD 论文结构。这里的核心判断是：`SRPD-v1` 可以作为 plain structured-risk teacher SFT ablation，`SRPD-v2` 是 uncertainty weighting ablation，`SRPD-v4` 是 gap-gated reliability control ablation，`SRPD-v5` 是 direct-anchor fallback ablation，`SRPD-v6` 则是 DPO-style preference SFT ablation。换句话说，v2-v6 不是一串随意版本，而是一个从 uncertainty weighting、gap gate、direct anchor 到 preference construction 的机制消融路径。早期 `outputs/robustness/` 下的 noise curve 和 robustness summary 也应该被写成现有 robustness infrastructure，后续 Week8/Week9 如果要扩展，就围绕 final SRPD-System 做，而不是从零再发明另一套 robustness 实验。
