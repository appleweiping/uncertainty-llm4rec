**version pony week5 工作计划**

week5 的总目标，是把我当前已经成立的旧主链——也就是 preprocess、build samples、inference、evaluation、calibration、reranking——从“单一 pointwise uncertainty pipeline”正式升级成“多决策任务统一框架”的第一版工程实体。这里最重要的原则，不是推翻旧代码，而是在保留 pointwise 主链的前提下，把 pointwise 重新定位为 uncertainty diagnosis 的基础观测层，然后在其上新增 pairwise preference 和 candidate ranking 两条任务线。也就是说，week5 不是旧系统的重写周，而是整个项目研究对象的重构周。到 week5 结束时，我必须拿到的，不只是几个新增脚本，而是一套可以明确说明三件事的系统结构：第一，pointwise 仍然存在，但已经从最终任务降级为诊断层；第二，pairwise preference 已经作为偏好机制层进入项目；第三，candidate ranking 已经开始承担主决策任务。这三件事如果不能在 week5 内形成可运行、可评估、可聚合的闭环，那 version pony 就仍然停留在文字层，而没有真正进入研究执行层。

week5 第一天，我要完成的是**多任务数据层的正式落地**。这一天的重点不是跑实验，而是把任务边界先在数据结构上钉死。旧版项目里，`main_build_samples.py` 和 `src/data/sample_builder.py` 主要服务的是 pointwise task，也就是从用户历史、正负样本构造出 `train.jsonl`、`valid.jsonl`、`test.jsonl` 这类点式输入。到了 week5，我不会破坏这条主线，因为 pointwise 还要继续承担 diagnostic layer 的职责；但我必须在它之上新增两类多任务派生能力。第一类是 candidate ranking builder，我会新增 `src/data/candidate_ranking_builder.py`，把同一用户的一组候选组织成一条 ranking sample。这个 schema 不能随意写，必须直接为后面的 ranking prompt、ranking 解析、ranking 评估和 ranking rerank 服务，因此每条样本至少要包含 `user_id`、`history`、`candidate_item_ids`、`candidate_titles`、`candidate_texts`、`candidate_popularity_groups`、`positive_item_id`、`timestamp`、`split_name`。第二类是 pairwise preference builder，我会新增 `src/data/pairwise_preference_builder.py`，把 ranking 候选进一步派生成对比较样本。这个 schema 至少要包含 `user_id`、`history`、`item_a_id`、`item_a_text`、`item_b_id`、`item_b_text`、`preferred_item`、`pair_type`、`source_positive_item_id`、`timestamp`。其中 `pair_type` 我一开始就要预留出来，因为后面很可能会从 `positive_vs_negative` 扩展到 `hard_negative_vs_negative`、`head_vs_tail` 等更复杂的 pairwise 结构，如果第一天 schema 不留口，后面所有分析都要返工。

为了让这个多任务数据层真正进入主项目而不是停留在辅助脚本，我还要新增一个明确的主入口 `main_build_multitask_samples.py`。这个脚本不会替代 `main_build_samples.py`，而是建立在 pointwise processed data 和 split 信息之上，统一派生 ranking 与 pairwise 两套输入。与此同时，我还要补齐配置层，至少新增 `configs/task/candidate_ranking.yaml` 和 `configs/task/pairwise_preference.yaml`，并开始建立对应的实验配置模板，例如 `configs/exp/beauty_qwen_pointwise.yaml`、`configs/exp/beauty_qwen_rank.yaml`、`configs/exp/beauty_qwen_pairwise.yaml`。这些配置里必须开始显式包含 `data_config`、`task_type`、`model_config`、`input_path`、`output_dir`、`exp_name`、`max_samples` 等字段，而不能再靠脚本内部写死。第一天的直接输出，不是论文表，而是多任务输入文件和一个结构化的数据汇总表。我至少要拿到 `ranking_valid.jsonl`、`ranking_test.jsonl`、`pairwise_valid.jsonl`、`pairwise_test.jsonl`，并且额外生成 `outputs/summary/week5_day1_multitask_data_summary.csv`，里面要列出每个 split 的样本数、平均候选数、平均 pair 数、pairwise 来源类型分布、正样本覆盖率。第一天的验收标准很明确：第一，ranking 和 pairwise 两类样本都能稳定生成；第二，pointwise、ranking、pairwise 三种任务在同一用户和同一正例来源上能够追踪对齐；第三，pair 数与候选数都处于合理范围，没有在构造层直接失控。第一天的贡献，不是“多了两个文件”，而是我第一次把 Part5 里的三层任务框架落到了数据层，使 version pony 从单任务样本构造正式升级为多任务样本构造系统。第一天结束时，我的结论应当是：我已经完成多决策任务输入接口的统一化，pointwise、pairwise、candidate ranking 三层任务不再停留在概念上，而是进入了可复现的数据生成阶段。

week5 第二天，我要推进的是**多任务 prompt 与 parser 层的正式切换**。如果第一天只是把数据准备好了，那么第二天就是让 LLM 真正能消费这些新任务。这里的关键原则，是不另起炉灶，而是在已有的 `src/llm/prompt_builder.py` 与 `src/llm/parser.py` 上做系统扩展。因为旧版项目已经证明 prompt 构造和 parser 鲁棒性是整条链能否稳定运行的关键，而 ranking 和 pairwise 比 pointwise 更容易出现输出漂移，所以这一天必须非常认真，不能用“先跑通再说”的心态糊过去。我会在 `prompt_builder.py` 里至少新增两个核心函数。第一个是 `build_candidate_ranking_prompt(sample, template, topk=None)`，它负责把 `history` 和多个 `candidate_text` 拼成 ranking prompt，并明确要求模型只在给定 candidates 内排序，不允许生成列表外 item，同时尽可能要求结构化 JSON 输出。第二个是 `build_pairwise_preference_prompt(sample, template)`，它要求模型在 A 与 B 两个候选之间明确给出 `preferred_item`、`confidence` 和简短理由。这两个 prompt 不仅要可运行，还要任务边界极清晰，因为顶会级实验最怕的就是 task formulation 本身模糊。

对应地，我会在 `parser.py` 里新增 `parse_candidate_ranking_response()` 和 `parse_pairwise_preference_response()`。前者至少要支持三种常见返回：纯 JSON、被 markdown code fence 包裹的 JSON、只给出 top-k 而没有完整排序的响应。后者则要能处理 `preferred_item=A/B`、item\_id 显式形式以及带自然语言包裹的情况，并把 confidence 统一规整到 0–1 范围。为了配合这一步，我还必须新增 `prompts/candidate_ranking.txt` 和 `prompts/pairwise_preference.txt` 两个模板文件，并显式规定输出字段名，避免后面 parser 被动追格式。第二天的实验不需要跑全量，但必须做足够扎实的小样本自测。我会用 10–20 条 ranking 样本和 20–50 条 pairwise 样本做 prompt / parse 闭环测试，并生成两类输出。第一类是 `outputs/summary/week5_day2_parse_success.csv`，记录 candidate ranking 与 pairwise 的 parse success rate、平均输出长度、非法输出比例、候选集外 item 出现率等关键统计。第二类是 `outputs/summary/week5_day2_prompt_examples.md`，手工保留成功样例、失败样例和修复说明，这类文档后面非常适合进 appendix 或工程日志。第二天的验收标准是，parse success rate 必须达到可接受水平，至少不能大量出现候选集外 item，也不能大量出现 pairwise 模糊偏好。第二天的贡献，是把 Part5 从“新任务可以构造样本”推进到“新任务可以被 LLM 稳定消费并产出结构化结果”。第二天结束时，我的结论应当是：candidate ranking 与 pairwise preference 已经进入可解析、可量化的推理准备状态，多任务框架的语言接口层完成切换。

week5 第三天，我要推进的是**三条推理主线的正式落地**。这一天开始，我不再满足于 prompt 自测，而要让 pointwise、pairwise、candidate ranking 三类任务都真正形成 inference 闭环。这里我不会继续把所有任务都塞进 `main_infer.py`，因为那样做只会让旧版 pointwise 主链失去清晰性。最稳的做法是保留 `main_infer.py` 专门服务 pointwise diagnostic layer，同时新增 `main_rank.py` 和 `main_pairwise.py` 两个任务入口。这样做的好处很直接：代码职责清晰，任务边界清晰，后面 summary 聚合和方法扩展也更自然。`main_rank.py` 的职责是读取 `ranking_test.jsonl`，调用 `build_candidate_ranking_prompt()`，走统一 backend，解析响应并输出 `rank_predictions.jsonl`。每条 prediction 至少要包含 `user_id`、`candidate_item_ids`、`pred_ranked_item_ids`、`topk_item_ids`、`raw_response`、`latency`、`model_name`、`parse_success`。`main_pairwise.py` 则读取 `pairwise_test.jsonl`，输出 `pairwise_predictions.jsonl`，每条记录至少包含 `user_id`、`item_a_id`、`item_b_id`、`preferred_item_pred`、`confidence`、`raw_response`、`parse_success`。这里必须保留 `raw_response` 和 `parse_success`，因为后面不只是做方法评估，还要做故障排查和 appendix 分析。

为了避免三条任务入口内部复制一堆逻辑，我还会同步整理 `src/llm/inference.py`，让它至少支持按 `task_type` 分发 prompt 构造和 parser，而不是默认 pointwise。第三天的核心实验安排，是 pointwise、pairwise、ranking 三条线一起跑，而不是只跑新任务，因为 week6 做多任务 compare 时，pointwise 仍然是必须保留的观测层。第三天的直接输出必须固定，包括 `outputs/beauty_qwen_pointwise/predictions/test_raw.jsonl`、`outputs/beauty_qwen_rank/predictions/rank_predictions.jsonl`、`outputs/beauty_qwen_pairwise/predictions/pairwise_predictions.jsonl`。除此之外，我还要补一张 `outputs/summary/week5_day3_inference_summary.csv`，统计三类任务的样本数、平均 latency、parse success、平均响应长度、失败比例等。第三天的验收标准是三套推理结果都稳定落盘，没有大面积解析失败、空输出或 schema 缺失。第三天的贡献，是完成多任务推理层的真正闭环。第三天结束时，我的结论应当是：我的项目已经不再只有单一 pointwise inference，而是具备 pointwise、pairwise、candidate ranking 三条并行推理主线，为后续统一评价和 uncertainty 跨任务迁移奠定了基础。

week5 第四天，我要进入**多任务评估层的正式落地**。这里的关键词不是“能跑”，而是“能算”，因为如果没有统一且合理的评估口径，多任务框架就只是多了几类输出文件，而不是真正进入研究空间。Part5 已经非常明确地说明了三层任务应该各自看什么指标。pointwise 继续承担 calibration quality、confidence discrimination 和 exposure-related bias 的职责，因此旧版 `main_eval.py` 仍然要保留并复用。candidate ranking 则必须进入真正的推荐指标空间，因此我会新增 `src/eval/ranking_task_metrics.py`，重点实现 HR@10、NDCG@10、MRR、coverage、head\_exposure\_ratio、longtail\_coverage。pairwise 则要新增 `src/eval/preference_metrics.py`，重点实现 pairwise accuracy、confidence-binned accuracy、preference consistency、必要时的 confidence histogram 统计。在此基础上，我要新增 `main_eval_rank.py` 和 `main_eval_pairwise.py` 两个主入口，让 ranking 与 pairwise 都能像 pointwise 一样稳定落盘出标准表，而不是只停留在 jsonl。

第四天跑完后，我至少要拿到三类核心输出。pointwise 侧要有 `diagnostic_metrics.csv`、reliability diagram、confidence histogram、popularity / exposure 分析图。ranking 侧要有 `ranking_metrics.csv`。pairwise 侧要有 `pairwise_metrics.csv` 和 `preference_confidence_bins.csv`。除此之外，我还必须生成一张统一的多任务摘要表 `outputs/summary/week5_day4_multitask_eval_summary.csv`，它的列头要提前为后面 week6 的 compare 设计好。也就是说，这张表至少要有 `task`、`HR@10`、`NDCG@10`、`MRR`、`pairwise_accuracy`、`ECE`、`Brier`、`coverage`、`head_exposure` 这些统一字段，哪怕某些任务在某些列上为空，也必须保持统一 schema。第四天的验收标准是：三种任务的评价都能稳定跑通，结果不是 NaN，表头统一，而且输出能清楚支撑三层任务的不同定位。第四天的贡献，是把多任务框架从“有结果文件”推进到“有可比较指标”的研究状态。第四天结束时，我的结论应当是：Part5 所要求的多任务评价体系已经完成第一轮最小闭环，pointwise、pairwise、candidate ranking 三个决策粒度第一次进入统一评估空间。

week5 第五天，我要做的是**week5 的论文级收口与多任务第一轮结论整理**。这一天的重点不是再加新功能，而是把前四天已经形成的点位真正收成一个可以支撑论文叙事的结构化总结。为此，我会新增 `src/analysis/aggregate_multitask_results.py`，并配一个入口 `main_compare_multitask.py`。它的职责不是做新算法，而是把 pointwise、pairwise、ranking 三类任务的关键表统一读进来，生成一张真正能够回答 Part5 核心问题的总表。这个总表我会命名为 `outputs/summary/week5_multitask_summary.csv`，每行至少包含 `domain`、`model`、`task`、`samples`、`HR@10`、`NDCG@10`、`MRR`、`pairwise_accuracy`、`ECE`、`Brier`、`coverage`、`head_exposure`、`longtail_coverage` 等字段。pointwise 不适用的 ranking 列可以为空，pairwise 不适用的 pointwise 列也可以为空，但整个表的 schema 必须统一，因为这张表之后要直接服务 week6 乃至论文正文。

除了 csv 之外，我还会同步生成一份自然语言的结论文件 `outputs/summary/week5_multitask_summary.md`。这份文件不是流水账，而要明确回答三个问题。第一，pointwise 是否仍然适合作为 uncertainty diagnosis 层；第二，pairwise 是否确实提供了局部偏好机制分析与中间层表达；第三，candidate ranking 是否已经开始承担真正的主决策角色。第五天的验收标准不是方法一定已经很强，而是我已经能稳定生成跨任务比较表，并且能够用这张表回答 Part5 最核心的问题：uncertainty 在不同决策粒度下到底如何发挥作用，它更适合作为绝对判断的诊断信号、局部偏好的风险估计，还是列表级排序约束。第五天的贡献，是完成 week5 的论文级收口。week5 的总体结论应当写成：我已经把项目从单一 pointwise uncertainty pipeline 升级成了多决策任务统一框架，其中 pointwise 被保留为诊断层，pairwise preference 被落实为偏好机制层，candidate ranking 被落实为主决策层，三层任务在统一数据、统一配置、统一推理和统一评估口径下首次形成了可运行、可比较、可聚合的系统结构。这是 version pony 真正开始成立的第一周，也是后续 week6 方法迁移、week7 强 baseline、week8 统一验证的前提。

如果把 week5 的整体价值再往顶会口径压缩一下，那么这一周最重要的不是“代码量增加了多少”，而是我完成了一次研究对象的正式重构。我不再围绕单一 pointwise task 去做 uncertainty 小修小补，而是把 uncertainty 放进了一个更完整的任务层级里：pointwise 负责观测，pairwise 负责机制，candidate ranking 负责主决策。到 week5 结束时，我应该已经拿到可以写进方法部分的任务定义、可以写进实验部分的输入输出与评估口径、可以写进项目报告的阶段性结果表，以及可以为 week6 的 uncertainty 跨任务迁移提供直接支撑的工程底座。换句话说，week5 要完成的，不是“多写两个脚本”，而是让 version pony 从概念上的顶会框架，第一次变成真正进入执行态的研究系统。

**week5 当前已完成状态补充：Part5 多任务底座已经闭环**

截至本轮最终升级，week5 已经从原先的计划文档转化为 Part5 的真实工程底座。多任务数据层已经完成，pointwise、candidate ranking、pairwise preference 三层任务能够从同一 processed 数据和 split 关系中派生出来，并通过 `main_build_multitask_samples.py` 进入统一输入结构。prompt 与 parser 层已经完成任务边界切换，candidate ranking 与 pairwise preference 不再依赖临时 prompt，而是拥有独立模板、独立解析逻辑和可统计的 parse success 记录。三条 inference 主线也已经落地，`main_infer.py`、`main_rank.py`、`main_pairwise.py` 分别承载 diagnosis、主排序决策和局部偏好机制层，不再把所有任务挤进旧 pointwise 入口。评价层同样已经完成最小闭环，pointwise 继续以 ECE、Brier、reliability、confidence-bias 为核心，candidate ranking 进入 HR@10、NDCG@10、MRR 和 exposure 口径，pairwise 则进入 preference accuracy 与机制一致性口径。最后，week5 的 multitask summary 已经把三层任务压到同一 summary schema 中，构成 week6 uncertainty transfer、decision family compare、pairwise-to-rank 和 Part5 finalization 的直接底座。
