今天这一步，我把 week7.7 从“Beauty 300-sample 的 trainable family 排序确认”正式推进到了“Beauty full teacher strengthening 已经真实落地、full973 SRPD 训练输入已经真实备齐，而且 ranking 执行链已经开始进入可接受吞吐区间”的阶段。对当前项目来说，这一步的价值非常明确：在我已经知道 `SRPD-v2` 是 Beauty 300-sample server run 下 trainable winner、但整体 trainable line 仍然弱于 strongest hand-crafted baseline 的前提下，我不能直接硬开 outer compare，也不能直接把 week8 四域矩阵全烧起来；更合理的做法，是先把 Beauty 单域的 teacher 强度和训练样本规模抬到接近“封王级确认”的状态，再重新检查 `SRPD-v2` 这条 uncertainty-aware trainable 主线在更强 teacher、更多样本下到底能不能继续增强。因此，今天的核心任务不是再跑一轮旧的 `v1/v2/v3`，而是把 Beauty 上的 DeepSeek compact evidence 从旧的 100-sample 层推进到当前真实可用的 full 版本，把这一整套结果沉淀成可以直接喂给 `SRPD-v1/v2/v3 full973` 的 teacher-data，并进一步开始处理 full973 ranking 在服务器上的实际吞吐问题。

这一步里最先需要澄清的事实，是我口头上一直说的“1000”在 Beauty 上并不是一个字面可执行的数字。实际检查本地数据后，我确认 `data/processed/amazon_beauty/ranking_test.jsonl` 当前只有 973 条 ranking events，`data/processed/amazon_beauty/pairwise_coverage_test.jsonl` 当前只有 500 条 pairwise coverage rows，而 `data/processed/amazon_beauty/test.jsonl` 的 pointwise 全量则是 5838 条。因此，week7.7 这一轮真正诚实、可复现、可写进论文的方法口径，不应该再写成含糊的“Beauty 1000”，而应该写成：`pointwise full5838`、`candidate ranking full973`、`pairwise support full500`。这个口径直接决定后面所有 full973 配置、teacher-data summary、以及 server-side full973 SRPD train/eval/compare 的解释边界。今天的第一项实际贡献，就是把这个边界用配置和真实运行结果固定下来，而不是继续停留在口头设想。

围绕这个边界，我完成了 full 版本配置层的落地，包括 `beauty_deepseek_pointwise_full.yaml`、`beauty_deepseek_rank_full973.yaml`、`beauty_deepseek_pairwise_coverage_full500.yaml`，以及与之配套的 `qwen3 local full`、`structured risk full973`、`SRPD-v1/v2/v3 full973` teacher-data config、LoRA config 和 batch config。这一层的作用不是为了“多几个 yaml”，而是为了把当前项目的 full-stage 路线真正变成一条可以从 DeepSeek teacher 强化一路走到 Qwen3 trainable reconfirmation 的正式实验链：DeepSeek full pointwise 负责 uncertainty diagnosis 和 calibrated teacher 信号，DeepSeek full973 ranking 负责 direct teacher decision，DeepSeek structured risk full973 负责 strongest handcrafted teacher，DeepSeek pairwise full500 负责保留 v3 的 mechanism evidence，而 `qwen3_rank_beauty_srpd_v1/v2/v3_full973` 则把这一切对接成可以直接上服务器的训练输入。到这一步为止，week7.7 后半段已经不再是一个模糊计划，而是一套真实工程入口。

今天真正最值钱的工作不只是配好 full 配置，而是把本地 DeepSeek API 调用速度问题真正解决掉。此前 pointwise、pairwise 和 API 侧 ranking 的推理链本质上是串行单请求，而且 `main_infer.py` 与 `main_pairwise.py` 还会在全部结束后才一次性写盘，这就导致两个问题：第一，速度极慢，pointwise full5838 按旧链路几乎是不可接受的几十小时级；第二，任何一次网络抖动都会让长跑任务从 0 开始，非常不适合 full 阶段。我今天针对这条链做的所有修改都严格限定在执行层，不改变 prompt、parser、uncertainty 定义、structured risk 公式、SRPD 训练目标和评价指标。具体来说，我在 `src/llm/api_backend.py` 里给 API backend 加上了受控并发的 `batch_generate`，在 `src/llm/inference.py` 里让 pointwise 与 pairwise 也真正按 backend 的 batch 跑起来，在 `configs/model/deepseek.yaml` 里显式加入了 `runtime.batch_size: 8` 与 `runtime.max_concurrency: 8`，并进一步在 `main_rank.py` 与 `main_pairwise.py` 中加入了 `--resume_partial` 与 `--checkpoint_every_batches` 机制，使得 ranking 与 pairwise 长任务能够批次级断点续跑。同时，我还把 `APIConnectionError`、`APITimeoutError` 这一类连接抖动正式纳入自动重试，而不再把它们直接当成不可恢复失败。这里的研究边界非常清楚：我改的是“怎么把相同方法更稳、更快地跑出来”，不是“为了速度去换一个更容易赢的实验定义”。

这些执行层优化已经得到了真实运行结果的支持。首先，`pointwise full5838` 已经在本地用 DeepSeek API 完整跑通，并完成了后续 eval 与 calibrate。最终原始 pointwise 诊断表显示 `num_samples = 5838`，`accuracy = 0.6565604659129839`，`avg_confidence = 0.7494552929085304`，`Brier = 0.26432216512504286`，`ECE = 0.21877697841726612`，`AUROC = 0.6184730581581528`。更重要的是，整条 full5838 pointwise 推理耗时大约 42 分钟，而根据我在优化前看到的串行速度估算，它原本会是几乎不可接受的超长时间。这说明受控并发对 API teacher strengthening 是真实有效的，而且不会改实验本身。其次，`candidate ranking full973` 也已经用同样的并发链跑完，中间一度出现过一次 DNS/连接级错误，但在我把 API 连接异常也纳入重试、并给 ranking 加了 checkpoint/resume 后，这条链重新跑通并最终完成，耗时大约 15 分钟。最后，`pairwise full500` 也已经跑完，耗时大约 3 分 40 秒。也就是说，今天我不只是证明了“DeepSeek API 理论上能跑 full”，而是已经把本地 DeepSeek full teacher 路线改造成了一条可接受、可恢复、可继续向前推进的真实执行链。

在 full-stage DeepSeek teacher 结果上，最重要的 ranking 结果也已经落地。`beauty_deepseek_rank_full973` 的 direct ranking 结果显示，在 973 条 ranking events 上，`HR@10 = 0.998972250770812`，`NDCG@10 = 0.6351559835664434`，`MRR = 0.5187541891952276`，`parse_success_rate = 0.9948612538540597`。在此基础上，我又用 `beauty_deepseek_pointwise_full` 的 calibrated uncertainty 作为信号，完成了 `beauty_deepseek_rank_full973_structured_risk` 的 structured-risk rerank。其结果是：`HR@10 = 1.0`，`NDCG@10 = 0.6355933223424297`，`MRR = 0.5189962315861597`，`changed_ranking_fraction = 0.0041109969167523125`，`avg_uncertainty_coverage_rate = 0.5005138746145941`。这个结果说明在 Beauty full973 范围里，structured risk 仍然比 direct ranking 略强，也就是说，当前 uncertainty-aware handcrafted mainline 在 full-stage teacher setting 下依然站得住。这一点对于后面重新评估 `SRPD-v2 full973` 是否有机会追上 strongest handcrafted baseline 是一个非常重要的参照系。

在 pairwise mechanism 这一侧，我也没有停在 raw pairwise predictions，而是把 full500 的 weighted pairwise-to-rank 结果完整做了出来。`beauty_deepseek_pairwise_coverage_full500_to_rank` 成功将 500 条 pairwise 结果对齐回 973 条 ranking events，对齐覆盖率为 `500 / 973 = 51.39%`。在这个 overlap subset 上，direct ranking 行给出的 `NDCG@10 = 0.6419210432266884`、`MRR = 0.5279623188405796`，而 `pairwise_to_rank_weighted_win_count` 的结果是 `NDCG@10 = 0.7669470017331042`、`MRR = 0.6983333333333334`，`parse_success_rate = 1.0`，`pairwise_supported_event_fraction = 0.513874614594039`，`pairwise_supported_candidate_fraction = 0.33333333333333326`，`pairwise_avg_reliability_weight = 0.6580272138067402`。这个结果再次验证了我此前的判断：weighted pairwise-to-rank 的确属于 uncertainty-aware mechanism line，而且它在当前 overlap subset 上依旧非常强。但它的强，不等于 trainable final method 已经赢；它更准确的角色，仍然是 supporting mechanism evidence，用来解释为什么 `SRPD-v3` 在设计上值得被保留，而不是直接替代 `SRPD-v2` 成为当前唯一主线。

在 DeepSeek full teacher 真正跑完后，我继续把 `SRPD-v1/v2/v3 full973` 的 teacher-data 直接构建出来，这一步的意义在于把今天这轮本地 DeepSeek full 结果无缝转换成服务器训练的直接输入，而不是等到服务器上再手工拼接。结果非常干净：`qwen3_rank_beauty_srpd_v1_full973`、`qwen3_rank_beauty_srpd_v2_full973`、`qwen3_rank_beauty_srpd_v3_full973` 全部实现了 `973 / 973 matched`，状态均为 `srpd_teacher_data_ready`。其中，`v2` summary 显示 `778 train / 195 valid`，`avg_sample_weight = 1.198151`；`v3` summary 则显示 `778 train / 195 valid`，`avg_sample_weight = 1.265779`，`avg_pairwise_preferences = 0.513875`。这说明 full973 阶段的 trainable line 已经不再缺输入，不再缺 teacher，也不再缺 pairwise support rows。对当前项目状态而言，这一步非常重要，因为它把“本地 DeepSeek full teacher 强化”和“服务器 Qwen3 trainable reconfirmation”正式连成了一条完整链：今天本地已经把该做的 DeepSeek full teacher 全做完了，接下来服务器只需要聚焦在 `Qwen3 SRPD-v1/v2/v3 full973 train/eval/compare` 本身，而不需要再为 teacher 侧补洞。

在今天这轮 full teacher 工作做完之后，我还专门把注意力转回到了服务器端 full973 ranking 的吞吐问题上，因为这一层会直接决定方案 A 的推进成本。实际在服务器上启动 `beauty_qwen3_local_rank_full973` 之后，我观察到当前 `Qwen3-8B local ranking full973` 在 `batch_size = 4` 下大约落在每个 ranking event 4 到 5 秒左右，这意味着整条 full973 direct ranking 仍然是小时级任务。从工程判断上说，这个速度并不离谱，因为这里跑的是本地小模型自回归 listwise generation，不是云端 DeepSeek 服务；而且当前 ranking prompt 仍然要求返回完整 `ranked_item_ids + topk_item_ids + confidence + reason`，同时 `max_new_tokens` 也还处在相对宽松的长生成区间，因此它的吞吐自然会慢于本地 DeepSeek API 那条并发强化链。问题在于，即便这种慢是“合理慢”，它依然会显著拖累 full973 的 compare 节奏，所以我没有把它当作只能接受的自然上限，而是继续只在执行层做了第二轮收紧优化。

这轮服务器端 ranking 提速的核心不是换方法，而是继续压缩输出契约与生成长度。我把 `prompts/candidate_ranking.txt` 改成更短 JSON，只保留必须的 `ranked_item_ids`，不再默认要求 `topk_item_ids`、`confidence` 和 `reason`；同时在 `main_rank.py` 中正式加入了 `max_new_tokens` 配置入口，并把 `beauty_qwen3_local_rank_full973.yaml` 与 `beauty_qwen3_local_rank_medium.yaml` 默认都改成了 `max_new_tokens: 96`、`resume_partial: true`、`checkpoint_every_batches: 1`。与之对应，我也把 `qwen3_rank_beauty_srpd_v1/v2/v3_full973.yaml` 的 eval generation 长度统一压到了 `96`，并在 `main_eval_lora_rank.py` 中加入了与 ranking 主入口一致的 `resume_partial + checkpoint_every_batches` 机制。这里的设计边界也非常明确：我没有改 parser 语义，没有改 ranking 指标逻辑，没有改 uncertainty 主线，只是把“模型必须多说的话”缩减到了“评估真正需要它说的话”。尤其重要的是，我在改完 prompt 之后还重新跑了 `main_validate_multitask_prompt_parser.py` 来确认 parser 仍然能稳定吃下新的更短输出口径，这样后面的速度改进就不会以损失解析稳定性为代价。

关于是否要在这一轮就直接切换到 vLLM 或常驻 serving，我的判断依然是谨慎保留，而不是立刻引入。原因并不是 vLLM 没价值，相反，它很可能会在更大规模阶段进一步改善本地 Qwen3 的吞吐；但对当前 week7.7 来说，最重要的问题仍然是：在同一条方法主线、同一套 rank prompt 语义、同一条 uncertainty-aware compare 口径下，`Qwen3 direct rank full973`、`Qwen3 structured risk full973` 与 `SRPD-v2 full973` 的相对位置到底是什么。如果我现在同时改变输出契约、生成长度、backend execution mode，变量就会过多，后面很难把任何 gain 或 regression 清楚归因。因此，今天的选择是：先用更短 JSON、96 tokens、resume/checkpoint 把 current local-HF backend 这条链压到更合理的执行状态，再在这个新口径下完成方案 A；如果在这一口径下 full973 ranking 仍然慢到不可接受，那么 vLLM 或 resident serving 才应该作为 week7.7 之后的下一层执行模式升级，而不是现在就混进方法确认阶段。

基于今天完成的全部工作，我对项目当前阶段的判断比之前更完整也更稳定。第一，`SRPD-v2` 依然是当前最应该优先继续确认的 trainable 主线；理由不是偏好，而是已有真实结果和 full-stage teacher strengthening 一起指向它：在 Beauty 300-sample server run 中，它已经是 `v1/v2/v3` 中表现最好的 trainable 版本，而今天 full973 teacher-data 构建完成后，它又成为最自然、信息增益最高的 first full973 candidate。第二，当前项目并没有偏离 uncertainty 主线。相反，今天 full-stage 的所有关键推进都进一步说明，当前最有研究价值的方向仍然是 uncertainty-aware ranking，而不是随意换一条更好讲故事的路线：DeepSeek structured risk full973 仍然略胜 direct ranking；`SRPD-v2 full973` 的 sample weights 已经清楚地把 uncertainty 信号写进训练输入；而 `v3` 目前依旧只是保留为 mechanism-enhanced variant，而不是被硬写成 final winner。第三，今天并不意味着项目已经具备立刻打开 outer compare 的资格。虽然 DeepSeek full teacher 强化已经成功，ranking 执行层也已进入第二轮提速，但真正决定 trainable line 是否有资格对外比较的，不是 teacher 侧更强了，而是 full973 下的 `Qwen3 SRPD-v2` 能不能在服务器上逼近甚至超过 strongest handcrafted baseline。因此，最合理的下一步依然是方案 A：优先跑 `Qwen3 direct rank full973`、`Qwen3 structured risk full973`、`SRPD-v2 full973 startup/train/eval/compare`，用最小但最有信息增益的一组 full973 server experiments，确认当前 trainable winner 在更强 teacher、更多样本、收紧执行契约之后到底是否继续成立。只有在这一步真的站住之后，补完 `v1/v3 full973` 和打开 outer compare 才有意义。换句话说，今天这份 3rd day 报告现在真正能收口成一句更准确的话：我不仅已经把 Beauty full teacher strengthening 真正做完了，也已经把 full973 ranking 的执行层提速推进到可用于 server-side winner reconfirmation 的状态，并且我知道为什么当前最值得先确认的是 `SRPD-v2 full973`，而不是把所有变量一起扔进下一轮。  

为了避免后面执行时再回头整理，我把方案 A 在此固定成一套可直接复制到服务器上的最短命令链。第一条是 `python main_rank.py --config configs/exp/beauty_qwen3_local_rank_full973.yaml --overwrite`，它负责在新的短契约、96 tokens 和断点续跑口径下重做 Beauty direct ranking full973；第二条是 `python main_rank_rerank.py --config configs/exp/beauty_qwen3_local_structured_risk_full973.yaml`，它负责在同一新口径下重做当前 strongest handcrafted baseline；第三条是 `python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml --startup_check`，它负责确认 train/valid/eval 输入、Qwen3 本地模型路径、adapter 输出目录和 framework 输出目录都已经对齐；第四条是 `python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml`，它负责正式训练当前最值得优先确认的 uncertainty-aware trainable winner；第五条是 `python main_eval_lora_rank.py --config configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml --overwrite --resume_partial`，它负责在同一套短契约和恢复逻辑下完成 full973 ranking eval；第六条是 `python main_compare_srpd_frameworks.py --configs configs/lora/qwen3_rank_beauty_srpd_v1_full973.yaml configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml configs/lora/qwen3_rank_beauty_srpd_v3_full973.yaml --output_path outputs/summary/week7_7_srpd_framework_compare_full973.csv --markdown_path outputs/summary/week7_7_srpd_framework_compare_full973.md`，它负责把 full973 的 trainable framework 结果正式收回统一 compare 表中。到这一步为止，week7.7 第三天不只是记录了一批“已经发生的工作”，也把“此刻服务器端最应该如何继续推进”确定成了一条足够短、足够稳、足够诚实的执行路径。

在后续真实服务器运行完成之后，我已经拿到了这轮 full973 reconfirmation 最关键的结果，因此这里必须把第三天的阶段结论进一步从“执行方案”升级成“真实 full973 结论”。首先，`beauty_qwen3_local_rank_full973` 已经真实完成，得到 `NDCG@10 = 0.6140307799644382`、`MRR = 0.4899990211912103`、`parse_success_rate = 0.9948612538540597`。其次，`beauty_qwen3_local_pointwise_full` 与 `beauty_qwen3_local_structured_risk_full973` 也已经真实跑通，其中 pointwise full 诊断结果为 `accuracy = 0.7262761219595752`、`ECE = 0.17295135320315183`、`Brier = 0.21858090099349092`，而 structured risk full973 的 rerank 结果为 `NDCG@10 = 0.6140777971356092`、`MRR = 0.4900479616306954`。这说明在当前小模型 full973 口径下，structured risk 相对 direct ranking 的提升仍然存在，但幅度已经很小，因而真正决定 trainable line 是否值得继续推进的，不再是 hand-crafted family 内部的微小差异，而是 SRPD 在同一口径下能否把 teacher 规律稳定蒸馏进 Qwen3-8B small-model。

更重要的是，`SRPD-v1/v2/v3 full973` 这条 trainable line 现在已经完整闭环并给出了真实 family 排序。`SRPD-v1_structured_risk_teacher_sft_full973` 的结果是 `NDCG@10 = 0.6261187576622534`、`MRR = 0.5064576909900651`、`parse_success_rate = 1.0`；`SRPD-v2_uncertainty_weighted_sft_full973` 的结果是 `NDCG@10 = 0.6353658182685563`、`MRR = 0.518448098663926`、`parse_success_rate = 0.998972250770812`；`SRPD-v3_pairwise_preference_enhanced_full973` 的结果则是 `NDCG@10 = 0.623807033106589`、`MRR = 0.503357314148681`、`parse_success_rate = 0.998972250770812`。这组 full973 结果使 trainable family 的内部关系第一次在近千样本、小模型、统一服务器口径下被完全坐实：`v2 > v1 > v3 > direct ranking`。其中，`v1` 已经证明“structured-risk teacher -> Qwen3 LoRA student”本身就是有效的；`v2` 进一步证明 uncertainty-weighted distillation 不是形式升级，而是能带来真实增益；`v3` 则说明当前 pairwise preference enhancement 的注入方式尚未把 mechanism signal 转化为额外收益，因此它仍然保留研究价值，但在当前版本里不能被写成 final small-model winner。

基于这一轮 full973 的真实结果，我对 inner compare、teacher compare 和后续 outer compare 的关系也有了更精确的理解。DeepSeek full973 那一层，仍然应该被写作 teacher / evidence layer：在那里，structured risk 是 strongest handcrafted teacher family，而 weighted pairwise-to-rank 是非常强的 mechanism evidence。Qwen3 server full973 这一层，则是 small-model trainable verification layer：在这一层里，`SRPD-v2` 已经成为当前最强的 trainable method，而且它不只是胜过 `SRPD-v1` 与 `SRPD-v3`，还已经明显高于同口径下的 Qwen3 direct ranking 与 Qwen3 structured risk。换句话说，我现在不需要再用“student 是否反超 DeepSeek teacher”来定义这条路线是否成立，因为那并不是当前论文真正需要证明的命题；更准确的命题应当是：在 small-model 赛道、统一 candidate schema、统一服务器口径下，`SRPD-v2` 是否已经成为当前最强的 uncertainty-aware trainable entrant。到这一步，答案已经是肯定的。

这也直接改变了我对后续推进顺序的判断。week7.7 在今天收口时，已经不再是“我还在试图让一个模糊的 SRPD family 站住”，而是“我已经确认 Beauty full973 下 small-model inner compare 的最终王者是 `SRPD-v2`，并且这一结论建立在完整的 `v1/v2/v3` 真实 train/eval/compare 之上”。因此，后续方法层不应该再无条件发散去追一个拍脑袋的 `v4`，也不应该继续把“是否超过 DeepSeek teacher”当作唯一标准。更合理的后续顺序，应当是：第一，用当前已经收敛出的 `SRPD-v2 full973` 作为 small-model inner winner，进入下一步跨域验证；第二，在跨域验证里优先补 `Books` 和 `Electronics` 这类风格差异较大的域，先验证 `SRPD-v2` 是否只是 Beauty 特例；第三，只有在跨域表现暴露出明确失败模式之后，才考虑是否需要设计一个更有假设支撑的 `v4`。也就是说，到今天这份第三天报告最终收口时，我已经不需要再把 week7.7 理解成“继续试方法”的阶段，而应当更准确地把它理解成：Beauty full973 上的 small-model trainable winner 已经确认，项目后续应进入跨域验证和 outer compare 候选筛选，而不是重新打开无边界的方法搜索。

最后，我也要把一个非常实际但容易被忽视的工程边界写清楚，避免后续 week7.7 之后的跨域实验污染当前已经落盘的 full973 结果。跨域本身不会自动影响已经完成的 Beauty 训练和实验，因为当前项目的实验组织是按 `exp_name / output_dir / adapter_output_dir / summary_path` 进行隔离的。只要我继续为 `Books`、`Movies`、`Electronics` 使用各自独立的配置、独立的 `outputs/...` 目录、独立的 `artifacts/adapters/...` 路径和独立的 summary 文件，例如 `qwen3_rank_books_srpd_v2_*`、`qwen3_rank_electronics_srpd_v2_*` 这样的命名，那么后续跨域运行只是在现有结果之上新增证据，不会反向覆盖 Beauty full973 已经完成的 train/eval/compare。真正会污染旧结果的，不是“开始做跨域”，而是错误地复用已有 Beauty 实验名、直接覆盖现有 `outputs/beauty_*` 或 `artifacts/adapters/qwen3_rank_beauty_*` 路径。因此，后续我在跨域阶段要特别坚持一个原则：所有新域实验都作为新增 exp 进入统一 compare，不复用 Beauty full973 的任何 output 路径，不重写当前已经确认的 inner winner 文件。把这一点写进报告很重要，因为它确保 week7.7 的 current winner 结论在工程上也是稳定可追踪的，而不是一轮新实验之后就被无意冲掉。

在完成 Beauty full973 small-model inner winner 的确认之后，我没有直接跳去 outer compare，而是先把跨域验证真正启动到了可以写进本轮报告的程度。具体来说，我已经在本地把 `Books small` 的 DeepSeek teacher 侧完整跑通，并且这一次不是早期那种 100-sample pilot，而是与当前 cross-domain small 口径一致的全量 small 版本：`pointwise full3000`、`candidate ranking full500`、`pairwise coverage full500`。在执行层上，我延续了与 Beauty 对齐的同一套“提速但不改方法”口径，也就是更短的 ranking 输出契约、`max_new_tokens = 96`、`resume_partial = true`、`checkpoint_every_batches = 1`、以及 pointwise/rank/pairwise 长任务的断点续跑机制。因此，Books 这一轮不是一个脱离主线的新分支，而是 Beauty 之后第一个严格在相同执行层原则下启动的跨域 teacher-strengthening 验证域。

从真实结果上看，Books small 的 teacher 侧已经给了我一组很重要的跨域信号。`books_deepseek_pointwise_full3000` 的 pointwise 诊断结果为 `accuracy = 0.8363333333333334`、`ECE = 0.044753333333333443`、`Brier = 0.13713419999999998`，说明在 Books 这个域里，teacher 侧 pointwise uncertainty 本身就比当前 Beauty full 的诊断状态更干净、更可校准。`books_deepseek_rank_full500` 的 direct ranking 结果则给出了 `NDCG@10 = 0.7527263536509895`、`MRR = 0.6735333333333333`，而在此基础上构造出的 `books_deepseek_rank_full500_structured_risk` 并没有像 Beauty 一样出现明显 rerank utility gain，其 structured-risk 行与 direct 行在主指标上几乎持平，同时 `changed_ranking_fraction = 0.004`。我对这件事的理解不是“structured risk 在 Books 上失效”，而是更精确地说：Books 域的 teacher direct ranking 本身已经很强，因此 uncertainty-aware rerank 在这一域更像一个保守的风险控制层，而不是一个强力重排器。这个结论很有价值，因为它说明 uncertainty-aware decision 的作用空间是域相关的，而这种域相关性正是后续跨域比较必须面对、而不是应该回避的真实问题。

更重要的是，Books small 的 SRPD teacher-data 也已经完整准备好了。`qwen3_rank_books_srpd_v1_small500`、`qwen3_rank_books_srpd_v2_small500`、`qwen3_rank_books_srpd_v3_small500` 均已实现 `500 / 500 matched`，并构成 `400 train / 100 valid` 的标准 split。其中，`SRPD-v2` 的 summary 给出了 `avg_sample_weight = 1.105289`，这说明 uncertainty-weighted distillation 在 Books 域已经拥有可直接进入 small-model LoRA 的训练输入；换句话说，我现在不只是“想在 Books 上做 SRPD”，而是已经把 Books 域推进到了与 Beauty 相同的 teacher-data ready 状态。到这一步，跨域工作就不再是纸面规划，而是已经至少在一个域里落到了真正可训练、可对比、可继续向 server-side small-model 验证推进的阶段。

也正因为 Books small 已经给出了真实、可解释的 teacher 侧信号，我对后续跨域策略的判断也进一步收紧了。此前我一度考虑是否应该把其他域继续砍成 100 或 300，先做更轻量的 pilot；但现在结合 Books small 的结果与三个域的真实 small 规模，我已经更明确地确认：后续跨域没有必要再人为缩样。当前 `Books / Electronics / Movies` 的 `small` 本身就已经对应 `ranking 500 / pointwise 3000 / pairwise 500`，这并不是一个过小到只能做 smoke test 的量级，而是足够承担第一轮跨域筛选任务的稳定规模。相对于 Beauty `full973 / full5838 / full500`，它们大约是“一半量级”的真实实验，而不是另一个更大的数量级。因此，我现在不再把 small 理解成“不得不接受的缩水版”，而是更准确地把它理解成：跨域第一轮里最有信息增益、最不浪费时间、又足够能看出方法稳定性的筛选层。这里的逻辑已经清楚：Beauty full973 负责封王，其他域 small-full 负责判断 `SRPD-v2` 是否只是 Beauty 特例；只有在 small-full 跨域结果进一步表明某个域非常值得升级时，我才需要考虑把那个域升 full，而不是现在先把所有域都烧大。

因此，在今天这份第三天报告最终收口时，我要把跨域下一步正式固定下来：后续不是只做一个 `Books` 来看看，而是直接推进三域 `small` 全量，按 `Books / Electronics / Movies` 这一组独立命名、独立输出、独立 teacher-data 的路径并行推进。其中，Books small teacher 层已经完成并可以作为第一块真实跨域证据；Electronics 与 Movies 则将继续沿用与 Beauty 完全对齐的执行层提速口径，从 DeepSeek teacher 侧开始往下接。这样做的意义，不是为了把 week7.7 重新拖回“大而全”的低效状态，而是为了尽快完成一轮足够诚实的跨域筛查：如果 `SRPD-v2` 在至少两个额外域上仍然保持相对 direct/structured risk 的优势，我就能更有底气地把它作为 small-model outer entrant；如果它在其他域上明显不稳，我也能更快知道问题出在 domain shift，而不是继续在 Beauty 上反复加样本。换句话说，今天第三天报告在最终版本里，不仅要记录 Beauty full973 已经封出的 small-model inner winner，也要明确写下：我已经正式把项目推进到了跨域验证阶段，而且这一阶段不会覆盖已有结果、不会改研究主线、不会把方法学纪律打散，而是围绕 `SRPD-v2` 作为当前最强 trainable line，继续做一个更像顶会收口而不是更像无边界试错的 next step。
