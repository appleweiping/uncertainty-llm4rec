今天这一步，我把 week7.7 从“Beauty 300-sample 的 trainable family 排序确认”正式推进到了“Beauty full teacher strengthening 已经真实落地、full973 SRPD 训练输入已经真实备齐，而且 ranking 执行链已经开始进入可接受吞吐区间”的阶段。对当前项目来说，这一步的价值非常明确：在我已经知道 `SRPD-v2` 是 Beauty 300-sample server run 下 trainable winner、但整体 trainable line 仍然弱于 strongest hand-crafted baseline 的前提下，我不能直接硬开 outer compare，也不能直接把 week8 四域矩阵全烧起来；更合理的做法，是先把 Beauty 单域的 teacher 强度和训练样本规模抬到接近“封王级确认”的状态，再重新检查 `SRPD-v2` 这条 uncertainty-aware trainable 主线在更强 teacher、更多样本下到底能不能继续增强。因此，今天的核心任务不是再跑一轮旧的 `v1/v2/v3`，而是把 Beauty 上的 DeepSeek compact evidence 从旧的 100-sample 层推进到当前真实可用的 full 版本，把这一整套结果沉淀成可以直接喂给 `SRPD-v1/v2/v3 full973` 的 teacher-data，并进一步开始处理 full973 ranking 在服务器上的实际吞吐问题。

这一步里最先需要澄清的事实，是我口头上一直说的“1000”在 Beauty 上并不是一个字面可执行的数字。实际检查本地数据后，我确认 `data/processed/amazon_beauty/ranking_test.jsonl` 当前只有 973 条 ranking events，`data/processed/amazon_beauty/pairwise_coverage_test.jsonl` 当前只有 500 条 pairwise coverage rows，而 `data/processed/amazon_beauty/test.jsonl` 的 pointwise 全量则是 5838 条。因此，week7.7 这一轮真正诚实、可复现、可写进论文的方法口径，不应该再写成含糊的“Beauty 1000”，而应该写成：`pointwise full5838`、`candidate ranking full973`、`pairwise support full500`。这个口径直接决定后面所有 full973 配置、teacher-data summary、以及 server-side full973 SRPD train/eval/compare 的解释边界。今天的第一项实际贡献，就是把这个边界用配置和真实运行结果固定下来，而不是继续停留在口头设想。

围绕这个边界，我完成了 full 版本配置层的落地，包括 `beauty_deepseek_pointwise_full.yaml`、`beauty_deepseek_rank_full973.yaml`、`beauty_deepseek_pairwise_coverage_full500.yaml`，以及与之配套的 `qwen3 local full`、`structured risk full973`、`SRPD-v1/v2/v3 full973` teacher-data config、LoRA config 和 batch config。这一层的作用不是为了“多几个 yaml”，而是为了把当前项目的 full-stage 路线真正变成一条可以从 DeepSeek teacher 强化一路走到 Qwen3 trainable reconfirmation 的正式实验链：DeepSeek full pointwise 负责 uncertainty diagnosis 和 calibrated teacher 信号，DeepSeek full973 ranking 负责 direct teacher decision，DeepSeek structured risk full973 负责 strongest handcrafted teacher，DeepSeek pairwise full500 负责保留 v3 的 mechanism evidence，而 `qwen3_rank_beauty_srpd_v1/v2/v3_full973` 则把这一切对接成可以直接上服务器的训练输入。到这一步为止，week7.7 后半段已经不再是一个模糊计划，而是一套真实工程入口。

今天真正最值钱的工作不只是配好 full 配置，而是把本地 DeepSeek API 调用速度问题真正解决掉。此前 pointwise、pairwise 和 API 侧 ranking 的推理链本质上是串行单请求，而且 `main_infer.py` 与 `main_pairwise.py` 还会在全部结束后才一次性写盘，这就导致两个问题：第一，速度极慢，pointwise full5838 按旧链路几乎是不可接受的几十小时级；第二，任何一次网络抖动都会让长跑任务从 0 开始，非常不适合 full 阶段。我今天针对这条链做的所有修改都严格限定在执行层，不改变 prompt、parser、uncertainty 定义、structured risk 公式、SRPD 训练目标和评价指标。具体来说，我在 `src/llm/api_backend.py` 里给 API backend 加上了受控并发的 `batch_generate`，在 `src/llm/inference.py` 里让 pointwise 与 pairwise 也真正按 backend 的 batch 跑起来，在 `configs/model/deepseek.yaml` 里显式加入了 `runtime.batch_size: 8` 与 `runtime.max_concurrency: 8`，并进一步在 `main_rank.py` 与 `main_pairwise.py` 中加入了 `--resume_partial` 与 `--checkpoint_every_batches` 机制，使得 ranking 与 pairwise 长任务能够批次级断点续跑。同时，我还把 `APIConnectionError`、`APITimeoutError` 这一类连接抖动正式纳入自动重试，而不再把它们直接当成不可恢复失败。这里的研究边界非常清楚：我改的是“怎么把相同方法更稳、更快地跑出来”，不是“为了速度去换一个更容易赢的实验定义”。

这些执行层优化已经得到了真实运行结果的支持。首先，`pointwise full5838` 已经在本地用 DeepSeek API 完整跑通，并完成了后续 eval 与 calibrate。最终原始 pointwise 诊断表显示 `num_samples = 5838`，`accuracy = 0.6565604659129839`，`avg_confidence = 0.7494552929085304`，`Brier = 0.26432216512504286`，`ECE = 0.21877697841726612`，`AUROC = 0.6184730581581528`。更重要的是，整条 full5838 pointwise 推理耗时大约 42 分钟，而根据我在优化前看到的串行速度估算，它原本会是几乎不可接受的超长时间。这说明受控并发对 API teacher strengthening 是真实有效的，而且不会改实验本身。其次，`candidate ranking full973` 也已经用同样的并发链跑完，中间一度出现过一次 DNS/连接级错误，但在我把 API 连接异常也纳入重试、并给 ranking 加了 checkpoint/resume 后，这条链重新跑通并最终完成，耗时大约 15 分钟。最后，`pairwise full500` 也已经跑完，耗时大约 3 分 40 秒。也就是说，今天我不只是证明了“DeepSeek API 理论上能跑 full”，而是已经把本地 DeepSeek full teacher 路线改造成了一条可接受、可恢复、可继续向前推进的真实执行链。

在 full-stage DeepSeek teacher 结果上，最重要的 ranking 结果也已经落地。`beauty_deepseek_rank_full973` 的 direct ranking 结果显示，在 973 条 ranking events 上，`HR@10 = 0.998972250770812`，`NDCG@10 = 0.6351559835664434`，`MRR = 0.5187541891952276`，`parse_success_rate = 0.9948612538540597`。在此基础上，我又用 `beauty_deepseek_pointwise_full` 的 calibrated uncertainty 作为信号，完成了 `beauty_deepseek_rank_full973_structured_risk` 的 structured-risk rerank。其结果是：`HR@10 = 1.0`，`NDCG@10 = 0.6355933223424297`，`MRR = 0.5189962315861597`，`changed_ranking_fraction = 0.0041109969167523125`，`avg_uncertainty_coverage_rate = 0.5005138746145941`。这个结果说明在 Beauty full973 范围里，structured risk 仍然比 direct ranking 略强，也就是说，当前 uncertainty-aware handcrafted mainline 在 full-stage teacher setting 下依然站得住。这一点对于后面重新评估 `SRPD-v2 full973` 是否有机会追上 strongest handcrafted baseline 是一个非常重要的参照系。

在 pairwise mechanism 这一侧，我也没有停在 raw pairwise predictions，而是把 full500 的 weighted pairwise-to-rank 结果完整做了出来。`beauty_deepseek_pairwise_coverage_full500_to_rank` 成功将 500 条 pairwise 结果对齐回 973 条 ranking events，对齐覆盖率为 `500 / 973 = 51.39%`。在这个 overlap subset 上，direct ranking 行给出的 `NDCG@10 = 0.6419210432266884`、`MRR = 0.5279623188405796`，而 `pairwise_to_rank_weighted_win_count` 的结果是 `NDCG@10 = 0.7669470017331042`、`MRR = 0.6983333333333334`，`parse_success_rate = 1.0`，`pairwise_supported_event_fraction = 0.513874614594039`，`pairwise_supported_candidate_fraction = 0.33333333333333326`，`pairwise_avg_reliability_weight = 0.6580272138067402`。这个结果再次验证了我此前的判断：weighted pairwise-to-rank 的确属于 uncertainty-aware mechanism line，而且它在当前 overlap subset 上依旧非常强。但它的强，不等于 trainable final method 已经赢；它更准确的角色，仍然是 supporting mechanism evidence，用来解释为什么 `SRPD-v3` 在设计上值得被保留，而不是直接替代 `SRPD-v2` 成为当前唯一主线。

在 DeepSeek full teacher 真正跑完后，我继续把 `SRPD-v1/v2/v3 full973` 的 teacher-data 直接构建出来，这一步的意义在于把今天这轮本地 DeepSeek full 结果无缝转换成服务器训练的直接输入，而不是等到服务器上再手工拼接。结果非常干净：`qwen3_rank_beauty_srpd_v1_full973`、`qwen3_rank_beauty_srpd_v2_full973`、`qwen3_rank_beauty_srpd_v3_full973` 全部实现了 `973 / 973 matched`，状态均为 `srpd_teacher_data_ready`。其中，`v2` summary 显示 `778 train / 195 valid`，`avg_sample_weight = 1.198151`；`v3` summary 则显示 `778 train / 195 valid`，`avg_sample_weight = 1.265779`，`avg_pairwise_preferences = 0.513875`。这说明 full973 阶段的 trainable line 已经不再缺输入，不再缺 teacher，也不再缺 pairwise support rows。对当前项目状态而言，这一步非常重要，因为它把“本地 DeepSeek full teacher 强化”和“服务器 Qwen3 trainable reconfirmation”正式连成了一条完整链：今天本地已经把该做的 DeepSeek full teacher 全做完了，接下来服务器只需要聚焦在 `Qwen3 SRPD-v1/v2/v3 full973 train/eval/compare` 本身，而不需要再为 teacher 侧补洞。

在今天这轮 full teacher 工作做完之后，我还专门把注意力转回到了服务器端 full973 ranking 的吞吐问题上，因为这一层会直接决定方案 A 的推进成本。实际在服务器上启动 `beauty_qwen3_local_rank_full973` 之后，我观察到当前 `Qwen3-8B local ranking full973` 在 `batch_size = 4` 下大约落在每个 ranking event 4 到 5 秒左右，这意味着整条 full973 direct ranking 仍然是小时级任务。从工程判断上说，这个速度并不离谱，因为这里跑的是本地小模型自回归 listwise generation，不是云端 DeepSeek 服务；而且当前 ranking prompt 仍然要求返回完整 `ranked_item_ids + topk_item_ids + confidence + reason`，同时 `max_new_tokens` 也还处在相对宽松的长生成区间，因此它的吞吐自然会慢于本地 DeepSeek API 那条并发强化链。问题在于，即便这种慢是“合理慢”，它依然会显著拖累 full973 的 compare 节奏，所以我没有把它当作只能接受的自然上限，而是继续只在执行层做了第二轮收紧优化。

这轮服务器端 ranking 提速的核心不是换方法，而是继续压缩输出契约与生成长度。我把 `prompts/candidate_ranking.txt` 改成更短 JSON，只保留必须的 `ranked_item_ids`，不再默认要求 `topk_item_ids`、`confidence` 和 `reason`；同时在 `main_rank.py` 中正式加入了 `max_new_tokens` 配置入口，并把 `beauty_qwen3_local_rank_full973.yaml` 与 `beauty_qwen3_local_rank_medium.yaml` 默认都改成了 `max_new_tokens: 96`、`resume_partial: true`、`checkpoint_every_batches: 1`。与之对应，我也把 `qwen3_rank_beauty_srpd_v1/v2/v3_full973.yaml` 的 eval generation 长度统一压到了 `96`，并在 `main_eval_lora_rank.py` 中加入了与 ranking 主入口一致的 `resume_partial + checkpoint_every_batches` 机制。这里的设计边界也非常明确：我没有改 parser 语义，没有改 ranking 指标逻辑，没有改 uncertainty 主线，只是把“模型必须多说的话”缩减到了“评估真正需要它说的话”。尤其重要的是，我在改完 prompt 之后还重新跑了 `main_validate_multitask_prompt_parser.py` 来确认 parser 仍然能稳定吃下新的更短输出口径，这样后面的速度改进就不会以损失解析稳定性为代价。

关于是否要在这一轮就直接切换到 vLLM 或常驻 serving，我的判断依然是谨慎保留，而不是立刻引入。原因并不是 vLLM 没价值，相反，它很可能会在更大规模阶段进一步改善本地 Qwen3 的吞吐；但对当前 week7.7 来说，最重要的问题仍然是：在同一条方法主线、同一套 rank prompt 语义、同一条 uncertainty-aware compare 口径下，`Qwen3 direct rank full973`、`Qwen3 structured risk full973` 与 `SRPD-v2 full973` 的相对位置到底是什么。如果我现在同时改变输出契约、生成长度、backend execution mode，变量就会过多，后面很难把任何 gain 或 regression 清楚归因。因此，今天的选择是：先用更短 JSON、96 tokens、resume/checkpoint 把 current local-HF backend 这条链压到更合理的执行状态，再在这个新口径下完成方案 A；如果在这一口径下 full973 ranking 仍然慢到不可接受，那么 vLLM 或 resident serving 才应该作为 week7.7 之后的下一层执行模式升级，而不是现在就混进方法确认阶段。

为了避免后面执行时再回头整理，我把方案 A 在此固定成一套可直接复制到服务器上的最短命令链。第一条是 `python main_rank.py --config configs/exp/beauty_qwen3_local_rank_full973.yaml --overwrite`，它负责在新的短契约、96 tokens 和断点续跑口径下重做 Beauty direct ranking full973；第二条是 `python main_rank_rerank.py --config configs/exp/beauty_qwen3_local_structured_risk_full973.yaml`，它负责在同一新口径下重做当前 strongest handcrafted baseline；第三条是 `python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml --startup_check`，它负责确认 train/valid/eval 输入、Qwen3 本地模型路径、adapter 输出目录和 framework 输出目录都已经对齐；第四条是 `python main_lora_train_rank.py --config configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml`，它负责正式训练当前最值得优先确认的 uncertainty-aware trainable winner；第五条是 `python main_eval_lora_rank.py --config configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml --overwrite --resume_partial`，它负责在同一套短契约和恢复逻辑下完成 full973 ranking eval；第六条是 `python main_compare_srpd_frameworks.py --configs configs/lora/qwen3_rank_beauty_srpd_v1_full973.yaml configs/lora/qwen3_rank_beauty_srpd_v2_full973.yaml configs/lora/qwen3_rank_beauty_srpd_v3_full973.yaml --output_path outputs/summary/week7_7_srpd_framework_compare_full973.csv --markdown_path outputs/summary/week7_7_srpd_framework_compare_full973.md`，它负责把 full973 的 trainable framework 结果正式收回统一 compare 表中。到这一步为止，week7.7 第三天不只是记录了一批“已经发生的工作”，也把“此刻服务器端最应该如何继续推进”确定成了一条足够短、足够稳、足够诚实的执行路径。

在后续真实服务器运行完成之后，我已经拿到了这轮 full973 reconfirmation 最关键的结果，因此这里必须把第三天的阶段结论进一步从“执行方案”升级成“真实 full973 结论”。首先，`beauty_qwen3_local_rank_full973` 已经真实完成，得到 `NDCG@10 = 0.6140307799644382`、`MRR = 0.4899990211912103`、`parse_success_rate = 0.9948612538540597`。其次，`beauty_qwen3_local_pointwise_full` 与 `beauty_qwen3_local_structured_risk_full973` 也已经真实跑通，其中 pointwise full 诊断结果为 `accuracy = 0.7262761219595752`、`ECE = 0.17295135320315183`、`Brier = 0.21858090099349092`，而 structured risk full973 的 rerank 结果为 `NDCG@10 = 0.6140777971356092`、`MRR = 0.4900479616306954`。这说明在当前小模型 full973 口径下，structured risk 相对 direct ranking 的提升仍然存在，但幅度已经很小，因而真正决定 trainable line 是否值得继续推进的，不再是 hand-crafted family 内部的微小差异，而是 SRPD 在同一口径下能否把 teacher 规律稳定蒸馏进 Qwen3-8B small-model。

更重要的是，`SRPD-v1/v2/v3 full973` 这条 trainable line 现在已经完整闭环并给出了真实 family 排序。`SRPD-v1_structured_risk_teacher_sft_full973` 的结果是 `NDCG@10 = 0.6261187576622534`、`MRR = 0.5064576909900651`、`parse_success_rate = 1.0`；`SRPD-v2_uncertainty_weighted_sft_full973` 的结果是 `NDCG@10 = 0.6353658182685563`、`MRR = 0.518448098663926`、`parse_success_rate = 0.998972250770812`；`SRPD-v3_pairwise_preference_enhanced_full973` 的结果则是 `NDCG@10 = 0.623807033106589`、`MRR = 0.503357314148681`、`parse_success_rate = 0.998972250770812`。这组 full973 结果使 trainable family 的内部关系第一次在近千样本、小模型、统一服务器口径下被完全坐实：`v2 > v1 > v3 > direct ranking`。其中，`v1` 已经证明“structured-risk teacher -> Qwen3 LoRA student”本身就是有效的；`v2` 进一步证明 uncertainty-weighted distillation 不是形式升级，而是能带来真实增益；`v3` 则说明当前 pairwise preference enhancement 的注入方式尚未把 mechanism signal 转化为额外收益，因此它仍然保留研究价值，但在当前版本里不能被写成 final small-model winner。

基于这一轮 full973 的真实结果，我对 inner compare、teacher compare 和后续 outer compare 的关系也有了更精确的理解。DeepSeek full973 那一层，仍然应该被写作 teacher / evidence layer：在那里，structured risk 是 strongest handcrafted teacher family，而 weighted pairwise-to-rank 是非常强的 mechanism evidence。Qwen3 server full973 这一层，则是 small-model trainable verification layer：在这一层里，`SRPD-v2` 已经成为当前最强的 trainable method，而且它不只是胜过 `SRPD-v1` 与 `SRPD-v3`，还已经明显高于同口径下的 Qwen3 direct ranking 与 Qwen3 structured risk。换句话说，我现在不需要再用“student 是否反超 DeepSeek teacher”来定义这条路线是否成立，因为那并不是当前论文真正需要证明的命题；更准确的命题应当是：在 small-model 赛道、统一 candidate schema、统一服务器口径下，`SRPD-v2` 是否已经成为当前最强的 uncertainty-aware trainable entrant。到这一步，答案已经是肯定的。

这也直接改变了我对后续推进顺序的判断。week7.7 在今天收口时，已经不再是“我还在试图让一个模糊的 SRPD family 站住”，而是“我已经确认 Beauty full973 下 small-model inner compare 的最终王者是 `SRPD-v2`，并且这一结论建立在完整的 `v1/v2/v3` 真实 train/eval/compare 之上”。因此，后续方法层不应该再无条件发散去追一个拍脑袋的 `v4`，也不应该继续把“是否超过 DeepSeek teacher”当作唯一标准。更合理的后续顺序，应当是：第一，用当前已经收敛出的 `SRPD-v2 full973` 作为 small-model inner winner，进入下一步跨域验证；第二，在跨域验证里优先补 `Books` 和 `Electronics` 这类风格差异较大的域，先验证 `SRPD-v2` 是否只是 Beauty 特例；第三，只有在跨域表现暴露出明确失败模式之后，才考虑是否需要设计一个更有假设支撑的 `v4`。也就是说，到今天这份第三天报告最终收口时，我已经不需要再把 week7.7 理解成“继续试方法”的阶段，而应当更准确地把它理解成：Beauty full973 上的 small-model trainable winner 已经确认，项目后续应进入跨域验证和 outer compare 候选筛选，而不是重新打开无边界的方法搜索。

最后，我也要把一个非常实际但容易被忽视的工程边界写清楚，避免后续 week7.7 之后的跨域实验污染当前已经落盘的 full973 结果。跨域本身不会自动影响已经完成的 Beauty 训练和实验，因为当前项目的实验组织是按 `exp_name / output_dir / adapter_output_dir / summary_path` 进行隔离的。只要我继续为 `Books`、`Movies`、`Electronics` 使用各自独立的配置、独立的 `outputs/...` 目录、独立的 `artifacts/adapters/...` 路径和独立的 summary 文件，例如 `qwen3_rank_books_srpd_v2_*`、`qwen3_rank_electronics_srpd_v2_*` 这样的命名，那么后续跨域运行只是在现有结果之上新增证据，不会反向覆盖 Beauty full973 已经完成的 train/eval/compare。真正会污染旧结果的，不是“开始做跨域”，而是错误地复用已有 Beauty 实验名、直接覆盖现有 `outputs/beauty_*` 或 `artifacts/adapters/qwen3_rank_beauty_*` 路径。因此，后续我在跨域阶段要特别坚持一个原则：所有新域实验都作为新增 exp 进入统一 compare，不复用 Beauty full973 的任何 output 路径，不重写当前已经确认的 inner winner 文件。把这一点写进报告很重要，因为它确保 week7.7 的 current winner 结论在工程上也是稳定可追踪的，而不是一轮新实验之后就被无意冲掉。

在完成 Beauty full973 small-model inner winner 的确认之后，我没有直接跳去 outer compare，而是先把跨域验证真正启动到了可以写进本轮报告的程度。具体来说，我已经在本地把 `Books small` 的 DeepSeek teacher 侧完整跑通，并且这一次不是早期那种 100-sample pilot，而是与当前 cross-domain small 口径一致的全量 small 版本：`pointwise full3000`、`candidate ranking full500`、`pairwise coverage full500`。在执行层上，我延续了与 Beauty 对齐的同一套“提速但不改方法”口径，也就是更短的 ranking 输出契约、`max_new_tokens = 96`、`resume_partial = true`、`checkpoint_every_batches = 1`、以及 pointwise/rank/pairwise 长任务的断点续跑机制。因此，Books 这一轮不是一个脱离主线的新分支，而是 Beauty 之后第一个严格在相同执行层原则下启动的跨域 teacher-strengthening 验证域。

从真实结果上看，Books small 的 teacher 侧已经给了我一组很重要的跨域信号。`books_deepseek_pointwise_full3000` 的 pointwise 诊断结果为 `accuracy = 0.8363333333333334`、`ECE = 0.044753333333333443`、`Brier = 0.13713419999999998`，说明在 Books 这个域里，teacher 侧 pointwise uncertainty 本身就比当前 Beauty full 的诊断状态更干净、更可校准。`books_deepseek_rank_full500` 的 direct ranking 结果则给出了 `NDCG@10 = 0.7527263536509895`、`MRR = 0.6735333333333333`，而在此基础上构造出的 `books_deepseek_rank_full500_structured_risk` 并没有像 Beauty 一样出现明显 rerank utility gain，其 structured-risk 行与 direct 行在主指标上几乎持平，同时 `changed_ranking_fraction = 0.004`。我对这件事的理解不是“structured risk 在 Books 上失效”，而是更精确地说：Books 域的 teacher direct ranking 本身已经很强，因此 uncertainty-aware rerank 在这一域更像一个保守的风险控制层，而不是一个强力重排器。这个结论很有价值，因为它说明 uncertainty-aware decision 的作用空间是域相关的，而这种域相关性正是后续跨域比较必须面对、而不是应该回避的真实问题。

更重要的是，Books small 的 SRPD teacher-data 也已经完整准备好了。`qwen3_rank_books_srpd_v1_small500`、`qwen3_rank_books_srpd_v2_small500`、`qwen3_rank_books_srpd_v3_small500` 均已实现 `500 / 500 matched`，并构成 `400 train / 100 valid` 的标准 split。其中，`SRPD-v2` 的 summary 给出了 `avg_sample_weight = 1.105289`，这说明 uncertainty-weighted distillation 在 Books 域已经拥有可直接进入 small-model LoRA 的训练输入；换句话说，我现在不只是“想在 Books 上做 SRPD”，而是已经把 Books 域推进到了与 Beauty 相同的 teacher-data ready 状态。到这一步，跨域工作就不再是纸面规划，而是已经至少在一个域里落到了真正可训练、可对比、可继续向 server-side small-model 验证推进的阶段。

也正因为 Books small 已经给出了真实、可解释的 teacher 侧信号，我对后续跨域策略的判断也进一步收紧了。此前我一度考虑是否应该把其他域继续砍成 100 或 300，先做更轻量的 pilot；但现在结合 Books small 的结果与三个域的真实 small 规模，我已经更明确地确认：后续跨域没有必要再人为缩样。当前 `Books / Electronics / Movies` 的 `small` 本身就已经对应 `ranking 500 / pointwise 3000 / pairwise 500`，这并不是一个过小到只能做 smoke test 的量级，而是足够承担第一轮跨域筛选任务的稳定规模。相对于 Beauty `full973 / full5838 / full500`，它们大约是“一半量级”的真实实验，而不是另一个更大的数量级。因此，我现在不再把 small 理解成“不得不接受的缩水版”，而是更准确地把它理解成：跨域第一轮里最有信息增益、最不浪费时间、又足够能看出方法稳定性的筛选层。这里的逻辑已经清楚：Beauty full973 负责封王，其他域 small-full 负责判断 `SRPD-v2` 是否只是 Beauty 特例；只有在 small-full 跨域结果进一步表明某个域非常值得升级时，我才需要考虑把那个域升 full，而不是现在先把所有域都烧大。

因此，在今天这份第三天报告最终收口时，我要把跨域下一步正式固定下来：后续不是只做一个 `Books` 来看看，而是直接推进三域 `small` 全量，按 `Books / Electronics / Movies` 这一组独立命名、独立输出、独立 teacher-data 的路径并行推进。其中，Books small teacher 层已经完成并可以作为第一块真实跨域证据；Electronics 与 Movies 则将继续沿用与 Beauty 完全对齐的执行层提速口径，从 DeepSeek teacher 侧开始往下接。这样做的意义，不是为了把 week7.7 重新拖回“大而全”的低效状态，而是为了尽快完成一轮足够诚实的跨域筛查：如果 `SRPD-v2` 在至少两个额外域上仍然保持相对 direct/structured risk 的优势，我就能更有底气地把它作为 small-model outer entrant；如果它在其他域上明显不稳，我也能更快知道问题出在 domain shift，而不是继续在 Beauty 上反复加样本。换句话说，今天第三天报告在最终版本里，不仅要记录 Beauty full973 已经封出的 small-model inner winner，也要明确写下：我已经正式把项目推进到了跨域验证阶段，而且这一阶段不会覆盖已有结果、不会改研究主线、不会把方法学纪律打散，而是围绕 `SRPD-v2` 作为当前最强 trainable line，继续做一个更像顶会收口而不是更像无边界试错的 next step。

在这一跨域决定写下之后，我实际上已经继续把另外两个域推进到了与 Books 相同的 teacher-ready 阶段，因此这里必须把第三天的跨域状态再进一步从“计划”修正为“已完成的事实”。`Electronics small` 现在已经完整跑通了 `pointwise full3000 -> eval -> calibrate -> rank full500 -> structured risk full500 -> pairwise full500 -> pairwise_to_rank full500 -> SRPD-v1/v2/v3 teacher-data` 整条链。其中，pointwise 诊断结果为 `accuracy = 0.7623333333333333`、`ECE = 0.18766`、`Brier = 0.23954340000000002`，direct ranking 结果为 `NDCG@10 = 0.6805537995985784`、`MRR = 0.5772999999999999`，structured risk 在这一域上与 direct 同样基本持平，而 `SRPD-v2` 的 teacher-data summary 则已经给出 `500 / 500 matched`、`400 train / 100 valid`、`avg_sample_weight = 1.146552`。这说明在 Electronics 这种更偏功能性、文本相关性更弱的域上，teacher 侧 uncertainty-aware rerank 依然没有表现成“大幅改排序”的强改写器，但 uncertainty-weighted SRPD 训练输入已经正式准备完毕，因此这个域已经具备进入后续 small-model server compare 的资格。

`Movies small` 这一侧也已经在同一轮工作里被完整推进到了 teacher-data ready。我已经跑通了 `movies_deepseek_pointwise_full3000`、后续 eval 与 calibrate、`movies_deepseek_rank_full500`、`movies_deepseek_rank_full500_structured_risk`、`movies_deepseek_pairwise_coverage_full500`、`movies_deepseek_pairwise_coverage_full500_to_rank`，并且在发现 `movies srpd` 配置里误把 structured-risk teacher 文件名写成 `rank_reranked.jsonl` 后，立即完成了路径修正并重新构建 `SRPD-v1/v2/v3 small500` teacher-data。Movies 的 pointwise 诊断结果为 `accuracy = 0.7383333333333333`、`ECE = 0.2618533333333333`、`Brier = 0.2978792`，direct ranking 结果为 `NDCG@10 = 0.5769810724546242`、`MRR = 0.44423333333333337`，structured risk 在这一域上同样与 direct 持平，而 `SRPD-v2` 的 teacher-data summary 已经给出 `500 / 500 matched`、`400 train / 100 valid`、`avg_sample_weight = 1.167195`。到这一步为止，Books、Electronics、Movies 三个域的 small-full DeepSeek teacher 层已经全部对齐，且都成功产出了 `SRPD-v1/v2/v3` 的可训练输入。

因此，我现在可以把跨域阶段的阶段性结论写得更明确。第一，三域 `small` 并不是一个流于形式的“再试试看”层，而是已经被我真正执行成了一个和 Beauty 相同口径的跨域筛查层：统一走 pointwise calibrated uncertainty、统一走 500-row ranking split、统一保留 pairwise support line、统一支持断点续跑和批次级 checkpoint。第二，当前三域 teacher 侧都在告诉我同一个事实：在 `Books / Electronics / Movies` 这些域上，DeepSeek direct ranking 本身已经很强，因此 structured risk 往往不会像 Beauty 那样明显改写排序；这不是坏消息，而是一个很重要的研究发现，因为它说明 uncertainty-aware rerank 的 utility gain 不是普遍同质的，它与域本身的 direct-ranking stability 强相关。第三，尽管 structured risk 在这些 small 域上不再表现出显著 rerank gain，`SRPD-v2` 所需的 uncertainty-weighted teacher-data 却已经在三域全部完成，这恰恰使后续 small-model server compare 变得更有价值：下一步真正需要回答的问题，不再是 teacher 会不会改排序，而是 `SRPD-v2` 在这些不同域的 small-model 赛道里，是否仍然能像 Beauty full973 那样把 uncertainty-aware teacher 模式稳定蒸馏进 Qwen3-8B。换句话说，今天第三天报告到这里已经不只是在记录 Beauty 的胜者，也是在记录一个新的跨域实验事实：三域 small teacher 层已全部就绪，而项目后续的真正焦点，应当转向 `SRPD-v2` 的跨域 small-model验证，而不是回到 teacher 侧继续反复确认已经成立的执行链。

在把三域 teacher 层全部备齐之后，我继续把跨域工作真正推进到了 small-model server compare 层，并且这一层现在也已经给出了不能回避的真实结果。`Books small500` 的 small-model 路线已经完整闭环：`books_qwen3_local_pointwise_small3000`、`books_qwen3_local_rank_small500`、`books_qwen3_local_structured_risk_small500`、`qwen3_rank_books_srpd_v2_small500` 均已真实跑通。最终，Books 域的 Qwen3 direct ranking 给出 `NDCG@10 = 0.6398271002354915`、`MRR = 0.5239000000000001`，structured risk 给出 `NDCG@10 = 0.6395142413461731`、`MRR = 0.5235`，而 `SRPD-v2` 则达到了 `NDCG@10 = 0.7057074243106245`、`MRR = 0.6117666666666667`。这意味着 Books 域给出了一个非常强的正例：在 teacher 侧 structured risk 几乎不改 direct 排序的前提下，uncertainty-weighted distillation 仍然能在 small-model 赛道里明显超过 current Qwen3 direct 与 Qwen3 structured risk。换句话说，Books 这一域证明了 `SRPD-v2` 的增益并不依赖于 teacher 侧必须先出现巨大的 rerank gain，它仍然有可能通过 uncertainty-aware weighting 学到比 small-model base ranking 更强的 decision pattern。

但是，跨域结果并没有继续沿着“全域稳定封王”的方向发展。`Electronics small500` 这一域的 small-model结果已经完整闭环，其中 `electronics_qwen3_local_rank_small500` 给出的 direct ranking 为 `NDCG@10 = 0.6583007651522705`、`MRR = 0.5471666666666666`，`electronics_qwen3_local_structured_risk_small500` 与 direct 几乎完全持平，而 `qwen3_rank_electronics_srpd_v2_small500` 的最终结果却只有 `NDCG@10 = 0.6429579068580471`、`MRR = 0.5278333333333333`。这说明在 Electronics 上，`SRPD-v2` 没有延续 Beauty 与 Books 的优势，反而落到了 current small-model direct/structured risk 之下。`Movies small500` 的结果则更进一步强化了这种不稳定性：direct ranking 为 `NDCG@10 = 0.5730601060765828`、`MRR = 0.43896666666666667`，structured risk 为 `NDCG@10 = 0.5731826885570888`、`MRR = 0.43910000000000005`，而 `qwen3_rank_movies_srpd_v2_small500` 则降到了 `NDCG@10 = 0.5373477157607484`、`MRR = 0.39223333333333327`。这两域一起说明，我不能把当前跨域结果写成“`SRPD-v2` 在所有域上都稳定优于 baseline”；更诚实的写法必须是：`SRPD-v2` 当前具有明显的 domain sensitivity，它在 Beauty 和 Books 上强，但在 Electronics 和 Movies 上没有保持优势。

对这一下降的原因，我现在有几条相对清楚、而且和已有结果一致的解释。第一，teacher 侧的 structured-risk intervention 在 Electronics 与 Movies 上几乎没有真正改写 ranking，这一点从 teacher compare 已经非常清楚：这些域里 `changed_ranking_fraction` 极低，structured risk 与 direct ranking 的主指标几乎持平。这意味着当我把 structured-risk teacher 蒸馏到 small-model 时，student 实际上没有拿到一个像 Beauty 那样“可被明显学习的重排行为差异”，而更像是在学习一个和 direct ranking 高度接近的 teacher 目标；在这种情况下，uncertainty-weighted training 更容易退化成对 small-model 原有 ranking 规律的噪声重加权，而不一定形成真实可迁移的 gain。第二，域本身的可蒸馏性可能不同。Books 更偏语义与主题匹配，Beauty 也更容易让用户历史与 candidate item 之间形成可读的关联模式，而 Electronics 与 Movies 这两类域里，小模型很可能更依赖于局部 lexical cues、短文本 surface overlap 或类别先验；这会导致 uncertainty-aware teacher pattern 本身更难被单 epoch、400/100 split 的 LoRA student 稳定吸收。第三，当前 cross-domain small500 的 trainable scale 仍然偏克制。每个域只有 `400 train / 100 valid`，训练只有 1 个 epoch，虽然这足以回答“能否跑通”和“是否存在明显方向性 gain”，但它也意味着一旦 teacher 与 direct 的差异本就不大，student 很容易在有限样本和单轮蒸馏下学不到足够强的结构信息。第四，training log 本身也给了一个不能忽视的信号：Books 的 `SRPD-v2` 训练虽然日志里有不稳定 loss，但最终 ranking 指标很强；Electronics 的训练日志出现了极大幅度的 loss 波动与 `grad_norm = 0`，Movies 的训练日志则相对平稳而且数值更“像正常训练”。这说明当前跨域掉点不太像是单一代码 bug，而更像是“域本身可蒸馏性 + 当前训练配方稳定性 + teacher/direct gap 大小”三者共同作用的结果。

基于这一组跨域 small-model 真结果，我现在必须把项目叙事从“继续扩大样本看看会不会都变好”调整成更有边界的分析式叙事。当前最准确的结论，不是 `SRPD-v2` 失败了，也不是它已经跨域封王，而是：它在 Beauty 和 Books 上已经证明 uncertainty-weighted distillation 这条 trainable 路线是能打的；但在 Electronics 和 Movies 上，它没有稳定超过 same-setting small-model baselines，因此当前版本的 `SRPD-v2` 不能被写成 domain-invariant final method。这个结论对顶会写作反而是有价值的，因为它把问题收得更清楚了：当前工作已经回答了“uncertainty-aware hand-crafted teacher 能否被蒸馏成强 trainable small-model method”，答案是“在部分域上可以，而且效果很强”；但它同时也暴露了第二个更深入的问题，即“这种蒸馏增益在何种域上成立、在何种域上衰减，以及 teacher-direct gap 与 student gain 之间到底是什么关系”。因此，如果我后面要继续扩展这条线，最合理的方向已经不是盲目再跑更多域，而是围绕这次掉点现象去做有假设的修正：例如重新分析 Electronics 与 Movies 中 teacher/direct gap 过小是否使 `v2` 的 uncertainty weighting 变成了低效监督；或者检查 current small-model prompt/contract 是否对 Books/Beauty 更友好，而对 Electronics/Movies 的 surface-form 信号保留不足。换句话说，到第三天最终收口时，我现在拥有的不只是更多实验，而是一组足够清楚的 mixed evidence：`SRPD-v2` 已经不再是纯粹的 Beauty 特例，但它也还远没有达到“无需附加条件即可跨域稳定”的程度；这正是当前项目最值得继续向下追问、也最应该如实写进报告和论文的地方。

基于这组 mixed evidence，我把后续修正方向正式压成了一个可以直接执行的 `SRPD-v4` 方案，而不是继续在 `v1/v2/v3` 之间做无边界补跑。`SRPD-v4` 的定义是：**gap-gated uncertainty-weighted distillation**。它仍然保留 `v2` 的核心思想，即把 uncertainty-aware structured-risk teacher 蒸馏到 small-model student；但它不再像当前 `v2` 那样对所有样本都统一施加 uncertainty 与 risk 权重，而是只在 teacher 真正提供了比 direct ranking 更多信息的样本上激活强化权重。更直白地说，`v4` 的核心问题意识就是：当 Electronics 与 Movies 这类域里 teacher 和 direct 几乎不分家时，如果我还强行把所有样本都按 uncertainty/risk 放大，实际上可能是在把 low-gap 样本的噪声监督进一步放大；而当 Books/Beauty 这类域里 teacher 的确提供了更有信息量的 decision shift 时，我又不应该放弃 `v2` 已经证明有效的 uncertainty-aware weighting。因此，`v4` 不是一个“再多加一点模块”的版本，而是一个更保守、更有条件地使用 weighting 的版本：**没有 teacher gap 的样本，退回近似 `v1` 的 base weight；有 teacher gap 的样本，才进入 uncertainty-aware boosted weighting。**

从实现上看，我已经把这个定义开始真正落到代码里，而不是只停留在报告里。核心修改文件是 `src/training/srpd_dataset.py`。在这一版实现中，我保留了 `v1 / v2 / v3` 的既有行为，同时新增了 `srpd_stage = v4` 的合法路径，并在 `_sample_weight(...)` 中加入了一个显式的 gate 机制。当前默认实现采取 `gate.mode = teacher_gap`：也就是只有当 `teacher_ranked_item_ids != direct_ranked_item_ids` 时，样本才激活 `uncertainty_scale` 与 `risk_scale` 所带来的额外加权；如果 teacher 与 direct 本身一致，则该样本直接回落到 `fallback_weight = 1.0`，也就是接近 `v1` 的普通 teacher-SFT 权重。这一做法正对应了我对掉点原因的第一条分析：在失败域里，teacher-direct gap 太小，因此当前 `v2` 的全量 uncertainty weighting 很可能不是在强调有价值的纠偏信息，而是在对本来就没什么 teacher 增益的样本施加额外扰动。用 gap gate 收住这一点，至少能先测试“失败域问题是不是主要来自 low-gap 样本被过度加权”。

为了让 `v4` 不只是一个数据构造层的小改动，而是真正能在失败域里验证，我还同步把训练层做了克制但有针对性的调整。我没有把这一步做成大换血，而只是把 `Electronics` 与 `Movies` 的 `v4` LoRA config 改成了更稳的失败域修正版：`num_train_epochs` 从 `1.0` 提到 `2.0`，`learning_rate` 从 `2e-4` 下调到 `1e-4`，`warmup_ratio` 从 `0.03` 调到 `0.05`，其余 LoRA 结构、prompt、evaluation contract、`max_new_tokens = 96` 和 output layout 全都保持一致。这里的逻辑也很明确：我并不想把 `v4` 变成一个“什么都改了”的混合版本，那样后面任何 gain 都没法归因；但既然已有结果和 training log 都说明 failed domains 可能同时存在“teacher gap 过小”和“训练稳定性偏弱”的问题，我就允许 `v4` 在失败域里带一层最轻量的稳定化训练修正，而不是继续完全沿用 `v2` 的训练配方。换句话说，`v4` 的本质不是“更复杂”，而是“更有边界地只改最可能的失败点”。

在文件层面，我已经把这套最小可执行的 `v4` 骨架补出来了。数据层新增的是 `configs/srpd/qwen3_rank_electronics_srpd_v4_small500.yaml` 与 `configs/srpd/qwen3_rank_movies_srpd_v4_small500.yaml`，它们都把 `srpd_stage` 设为 `v4`，并在 `weighting.gate` 下明确指定了 `mode: teacher_gap`、`fallback_weight: 1.0` 与轻量的 `gate_boost`。训练层新增的是 `configs/lora/qwen3_rank_electronics_srpd_v4_small500.yaml` 与 `configs/lora/qwen3_rank_movies_srpd_v4_small500.yaml`，它们将 `framework_output_dir`、`adapter_output_dir`、`training_summary_path`、`startup_check_path`、`framework_metrics_path` 等路径全部独立命名，不会覆盖当前已经跑完的 `v2` 结果。这一点很重要，因为我接下来需要用 `Electronics` 与 `Movies` 这两个失败域先做 `v4` 的 first validation，而不是把 Beauty/Books 那些已经给出强正例的域一起拉进来增加变量。因此，到这一步为止，`v4` 已经不是一句“可能可以试试看”的口头判断，而是一套真实代码 + 真实配置 + 真实验证顺序都已经明确的下一步方案。

关于先在哪个域验证，我现在的优先级也已经完全收紧了：**先做 Electronics，再做 Movies。** 原因不是 Movies 不重要，而是 Electronics 的失败形态更像“本来 direct 就很强、teacher gap 很小、`v2` 小幅掉点但还没有完全崩”，它更适合用来判断 gap-gated weighting 是否能把 failed domain 从“略低于 direct”拉回到“至少不输”。如果 `Electronics v4` 比 `Electronics v2` 明显更稳，说明我的核心假设——即失败域的主要问题在于 low-gap 样本被过度加权——是有支撑的；这时再去做 Movies，会更像对一条已经初步成立的修正路线做第二域验证。反过来，如果 `Electronics v4` 仍然完全拉不回来，那么问题就更可能不只在 weighting gate，而更深地落在域可蒸馏性、prompt contract、teacher-direct gap 本身过小这些层面；这样我就不需要急着把 `v4` 扩到更多域，而应该回头重新看失败域的监督结构本身。因此，我现在的执行顺序已经不再是“再继续补旧版本”，而是：先在失败域里用最小代价测试最关键假设，再决定 `v4` 值不值得进一步推广。

把这一点写进第三天报告其实非常重要，因为它意味着当前项目已经从“扩大实验矩阵”真正进入了“用结果反推方法修正”的阶段。第三天到这里为止，我已经做完了四件彼此相连的事情：第一，确认了 Beauty full973 small-model inner winner 是 `SRPD-v2`；第二，确认了三域 small teacher 层已经全部就绪；第三，确认了三域 server-side small-model result 并不是单向利好，而是 `Books` 强、`Electronics` 与 `Movies` 掉点；第四，我没有把掉点当成简单坏消息，而是把它收束成了一个可以被代码化、被配置化、被失败域优先验证的 `SRPD-v4` 方案。也就是说，当前阶段最合理的后续工作，不是先把 `v1`、`v3` 在所有域都补齐，也不是立刻把 `v4` 推到全域，而是用 `Electronics -> Movies` 这条最小验证链，判断 gap-gated uncertainty weighting 是否真能针对 failed domains 提供修复。如果这一条成立，那么 `v4` 才值得再回到 Beauty/Books 去做统一确认；如果这一条不成立，那么我至少也能更清楚地知道问题不是简单的“全量 uncertainty weighting 太激进”，而需要回头重新审视 failed domains 的 teacher-design 与 small-model trainability。
## SRPD-v4结果补充与SRPD-v5定义

在完成 `SRPD-v4` 的代码与配置落地之后，我已经把它真正推到两个失败域上做了第一轮验证，因此它现在不再只是“一个解释掉点现象的想法”，而是已经有真实训练、真实评估和真实对比结果支撑的修复版方法。`Electronics small500` 上，当前 small-model baseline 是 `direct ranking NDCG@10 = 0.6583007651522705, MRR = 0.5471666666666666`，而 `SRPD-v2` 会掉到 `NDCG@10 = 0.6429579068580471, MRR = 0.5278333333333333`。在完全相同的数据、模型和评估口径下，`SRPD-v4` 回升到了 `NDCG@10 = 0.6594050381443246, MRR = 0.5492333333333334`，不仅明显优于 `v2`，而且略微超过了 direct baseline。这个结果直接说明：此前对失败域的核心诊断是成立的，即在 teacher-direct gap 很小的域里，`v2` 的全量 uncertainty weighting 很可能把低信息样本的监督噪声一起放大了，而 `gap-gated weighting` 能够把这个问题收住。与此同时，`Electronics v4` 的 `avg_sample_weight = 1.000657` 也很关键，因为它说明 `v4` 不是靠更激进的加权重新站起来，而是靠更克制的加权重新站起来，这与方法设计目标完全一致。

`Movies small500` 上，`SRPD-v4` 同样给出了方向正确但不彻底的修复证据。该域的 `direct ranking` 为 `NDCG@10 = 0.5730601060765828, MRR = 0.43896666666666667`，`SRPD-v2` 会下降到 `NDCG@10 = 0.5373477157607484, MRR = 0.39223333333333327`，而 `SRPD-v4` 则回升到 `NDCG@10 = 0.546388555376638, MRR = 0.40436666666666665`。这说明 `v4` 在 Movies 上也明显优于 `v2`，因此修正方向并不是 Electronics 偶然有效；但它又没有像在 Electronics 上那样完全追回 direct baseline，因此当前最诚实的结论只能是：`SRPD-v4` 已经证明自己是一个 **effective repair**，而不是一个已经完成跨域封王的 **universal winner**。也就是说，`v4` 证实了“失败域掉点与 low-gap sample 过度加权有关”这个判断，但它同时也暴露出第二层更深的问题：在 `Movies` 这类域里，student 不仅会受错误的 sample weighting 影响，还可能会受 teacher target 本身不够稳定、或者 direct baseline 本身更值得保留的影响。

基于这个观察，我把下一步最后一版方法正式定义成 `SRPD-v5`，并且它不再是继续堆 uncertainty 模块的版本，而是一个 **gap-gated + direct-anchor-preserving distillation**。在这个定义里，我继续保留 `v4` 的核心门控思想：只有 teacher 与 direct 真正存在有意义分歧时，uncertainty-aware boosted weighting 才被激活；但与 `v4` 不同的是，`v5` 不再默认 student 的监督目标总是 structured-risk teacher ranking。相反，我在 `src/training/srpd_dataset.py` 里加入了一个更强的 direct-anchor 规则：当 teacher 并没有把正样本排得比 direct 更靠前时，训练目标不再强制采用 teacher ranking，而是直接退回到 direct ranking；只有当 teacher 真的改善了正样本的位置时，student 才去学习 teacher target。换句话说，`v5` 真正要解决的是 `Movies v4` 留下来的那一层问题，即“如何避免 student 在低收益 teacher 差异上偏离本来就不弱的 direct baseline”。这使得 `v5` 不只是更保守的 weighting 版本，而是更有边界地把 uncertainty-aware teacher supervision 限定在“teacher 确实对主任务更有利”的样本上。

到实现层面，我已经完成 `SRPD-v5` 的第一轮实作，而不是只停留在口头计划。核心改动仍然集中在 `src/training/srpd_dataset.py`，其中我新增了 `srpd_stage = v5`，并让它在数据构造阶段同时记录 `teacher_ranked_item_ids`、`direct_ranked_item_ids`、`target_ranked_item_ids`、正样本在 teacher/direct/target 中的位置，以及 `teacher 是否真正改善正样本位置` 这一判断信号。这个设计的目的是把“是否值得模仿 teacher”前移到 teacher-data 构造阶段，而不是等 student 训练完成后再去解释失败。与之对应，我新增了四个配置文件：`configs/srpd/qwen3_rank_electronics_srpd_v5_small500.yaml`、`configs/lora/qwen3_rank_electronics_srpd_v5_small500.yaml`、`configs/srpd/qwen3_rank_movies_srpd_v5_small500.yaml`、`configs/lora/qwen3_rank_movies_srpd_v5_small500.yaml`。在训练配方上，`v5` 继续保持失败域的轻量稳定化设置，并把 `learning_rate` 进一步收敛到 `8e-5`，目的是在不改变研究主线的前提下，尽可能把“错误 teacher 目标 + 过强 student 偏移”这两类风险一起压下去。

从研究叙事上讲，这一步也进一步说明当前项目并没有偏离 uncertainty 主线。相反，当前的每一次修正都比此前更明确地围绕 uncertainty 展开：`v2` 回答的是“uncertainty-aware distillation 能否在部分域里形成强 trainable gain”；`v4` 回答的是“当失败域中 teacher-direct gap 太小时，如何避免 uncertainty weighting 放大低信息样本”；而 `v5` 则继续追问“当 uncertainty-aware teacher 并不总是优于 direct baseline 时，如何让 student 只学习真正有价值的 uncertainty-aware correction，而不是盲目偏离 strong direct baseline”。因此，项目当前并没有滑向普通的 LoRA 调参工作，而是在进入 uncertainty-aware trainable recommendation 里更难也更有价值的一层：我们不再只证明 uncertainty 有用，而是在研究 uncertainty 何时有用、为什么在某些域里失效、以及如何把这种条件性有用结构化地编码进 teacher-data 和 distillation target 里。

在 `SRPD-v5` 真正跑到失败域之后，我已经可以把这一轮方法修正的最终结论写得更清楚了。`Movies small500` 上，`direct ranking` 仍然是 `NDCG@10 = 0.5730601060765828, MRR = 0.43896666666666667`，`SRPD-v2` 为 `NDCG@10 = 0.5373477157607484, MRR = 0.39223333333333327`，`SRPD-v4` 回升到 `NDCG@10 = 0.546388555376638, MRR = 0.40436666666666665`，而 `SRPD-v5` 的真实结果是 `NDCG@10 = 0.5433505118993123, MRR = 0.40020000000000006`。这说明 `v5` 并没有进一步超过 `v4`，反而较 `v4` 略有回落。结合 `Movies v5` teacher-data summary 中 `avg_sample_weight = 1.0`，以及 train/valid split 中 `srpd_target_source` 基本全部回落为 `direct_anchor_preserved` 这一事实，我现在可以更准确地解释这一现象：`Movies` 失败域的问题并不是“teacher 完全没用”，而是“teacher 有用但不能被全量、无条件地学”；`v2` 学得太猛，`v5` 又退得太狠，而 `v4` 反而停在了当前最合理的折中点上。

因此，到 week7.7 第三天最终收口时，当前 trainable uncertainty-aware family 的结构已经可以明确写成：`v2` 是主域与部分语义域中的强版本，`v4` 是当前针对失败域最有效的修复版，而 `v5` 则起到了一个很重要但并非封王的作用，即它证明了“完全退回 direct-anchor preserving”并不是当前失败域的最终答案。也就是说，当前如果必须在 `v2 / v4 / v5` 之间给出一个对失败域最有解释力、也最值得继续保留的版本，那么 **`SRPD-v4` 应当被正式视为当前 failed-domain repair line 的王者**。它不是全域 universal winner，但它已经在 `Electronics` 上完成修复并略超 direct，在 `Movies` 上也明显优于 `v2`，而 `v5` 的结果反过来说明 `v4` 所处的位置确实是当前这组修正里更合理的平衡点。

基于这一点，当前项目后续最合理的推进方式已经不再是继续发散到 `v6`、`v7`，而是把 `v4` 作为当前 uncertainty-aware trainable repair line 的正式代表收住，然后决定哪些实验值得“全通”。我现在的判断是：后续不应该再继续扩版本，而应该先做 **统一收口与体系打通**。更具体地说，下一步最值得做的不是再发明新方法，而是把当前已经站住的这些层次系统性整理出来：Beauty full973 上的 `v1/v2/v3` 主域结果，Books / Electronics / Movies 上的 `direct / structured risk / v2 / v4 / v5` 跨域 mixed evidence，以及 DeepSeek teacher 层的 same-setting references。换句话说，后面如果说“先全通”，最合理的含义不是“把更多新方法再跑一遍”，而是“把当前已成立的方法族、失败域修复线和跨域证据系统性补齐成可直接写进论文和对外 compare 的完整矩阵”。这一步做完之后，项目就会从“方法探索中”正式进入“论文收口与 outer compare 准备”阶段。

## SRPD-v6 DPO-style preference variant 补充设计

在继续讨论“还能不能把小模型 LoRA 做得更强”时，我重新检查了当前 `SRPD-v4 / v5` 的方法边界，并明确做了一个不覆盖历史结果的决定：已有 `SRPD-v5` 已经被定义为 `gap-gated direct-anchor SFT`，并且已经在 Beauty、Books、Electronics、Movies 上产生真实结果，因此不能再把新的 DPO-style preference variant 也命名成 v5，否则会污染现有四域矩阵和论文叙事。为了保持实验可追踪、结果不覆盖、GitHub/服务器同步不混乱，我把这条新路线正式命名为 `SRPD-v6_gap_gated_dpo_preference_sft`。它的思想是承接 `SRPD-v4`，而不是替代 `SRPD-v5`：`v4` 负责控制什么时候激活 stronger uncertainty weighting，避免 low-gap teacher 被无条件放大；`v6` 在此基础上额外把 structured-risk teacher order 和 uncertainty-gated teacher-gap event 转化为 chosen/rejected preference pairs，让 Qwen3 LoRA 不只学习整条 listwise ranking JSON，也直接看到“哪个候选应该排在另一个候选前面”的 pairwise supervision。

这条路线的动机非常清楚。普通蒸馏只是让 student 模仿 teacher 排序，容易被审稿人认为是常规 teacher SFT；而真正更像顶会问题的地方，是我如何把 uncertainty-aware teacher correction 编码成更细粒度的 preference learning signal。`SRPD-v6` 因此不是简单多加训练样本，而是把 structured risk 的排序信息拆成 DPO-style chosen/rejected pairs：当 teacher order 给出明确的前后关系时，前项作为 chosen、后项作为 rejected；当 teacher 相比 direct ranking 明确改善 positive item 位置时，positive item 会形成更强的 teacher-positive-gap preference；这些 pair 同时继承 v4 的 gate，不再把所有低信息 correction 都当成强监督。这样一来，student 的训练目标从“请复刻这条 teacher list”扩展为“请在 listwise ranking 之外，也学会这些局部偏好不等式”。这比纯 SFT 更贴近 ranking optimization，也更容易和 NH baseline 里的 DPO、self-play、preference debias、pairwise ranking 方法放到同一个机制层讨论。

工程上，我已经把这条线做成不破坏现有链路的增量实现。`src/training/srpd_dataset.py` 现在支持新的 `srpd_stage: v6`，并在构建 teacher-data 时生成 `srpd_dpo_style_preferences` 和 `srpd_dpo_style_preference_count` 字段，同时在 summary 里新增 `avg_dpo_style_preferences`，用于记录每个 event 生成了多少 chosen/rejected pair。`src/training/rank_dataset.py` 新增了 `build_rank_preference_examples`，它会读取这些 pair，使用 `prompts/pairwise_preference.txt` 构造 pairwise preference prompt，并输出 `{"preferred_item": "...", "confidence": ..., "reason": ...}` 形式的监督目标。`src/training/lora_rank_trainer.py` 则新增 `training.preference_sft` 配置入口：当 `enabled: true` 时，训练集会保留原来的 candidate-ranking SFT example，同时追加 DPO-style pairwise preference SFT example。这样 v6 仍然用同一套 `main_lora_train_rank.py -> main_eval_lora_rank.py` 服务器执行链，评估时仍然回到 candidate ranking schema，不需要立刻引入新的 TRL/DPO 依赖，也不会破坏当前 main evaluator。

为了让这条线直接可跑，我已经补齐四域配置：`configs/srpd/qwen3_rank_beauty_srpd_v6_full973.yaml`、`configs/srpd/qwen3_rank_books_srpd_v6_small500.yaml`、`configs/srpd/qwen3_rank_electronics_srpd_v6_small500.yaml`、`configs/srpd/qwen3_rank_movies_srpd_v6_small500.yaml` 负责构建 v6 teacher-data；`configs/lora/qwen3_rank_beauty_srpd_v6_full973.yaml`、`configs/lora/qwen3_rank_books_srpd_v6_small500.yaml`、`configs/lora/qwen3_rank_electronics_srpd_v6_small500.yaml`、`configs/lora/qwen3_rank_movies_srpd_v6_small500.yaml` 负责服务器 LoRA 训练和评估。四域 teacher-data 构建已经在本地真实跑通：Beauty full973 matched `973 / 973`，Books/Electronics/Movies small500 均 matched `500 / 500`；startup check 也已经全部通过。更关键的是，v6 的混合训练样本数已经被验证：Beauty 由 `778 train rows` 扩展成 `3890 train examples`，每条 ranking event 对应 `1` 个 listwise ranking SFT example 和 `4` 个 DPO-style preference examples；Books、Electronics、Movies 则分别由 `400 train rows` 扩展成 `2000 train examples`。这说明 v6 已经不是纸面想法，而是一条能够直接上服务器训练的完整工程链。

从论文贡献上看，我不会把 `SRPD-v6` 预设成必胜版本，而会把它写成 preference-optimization strengthened candidate。它要回答的问题是：当 `v2` 的 uncertainty-weighted SFT 在 positive domains 很强、`v4/v5` 在 failed-domain repair 中部分有效时，显式 pairwise preference supervision 能不能进一步提升 small model 对局部排序关系的吸收能力。如果 v6 在 Beauty/Books 上继续接近或超过 v2，它就说明 DPO-style local preference constraints 能增强 positive-domain 蒸馏；如果 v6 在 Electronics/Movies 上超过 v4/v5，它就说明 failed-domain 修复需要的不只是 gate 或 direct anchor，还需要更直接的 pairwise ranking signal；如果 v6 没有提升，结论也依然有价值，因为它会证明当前问题的瓶颈不是“缺少 pairwise supervision”，而可能是 teacher gap、domain property 或 small-model capacity。本质上，v6 把我们从普通蒸馏推进到 preference-aware uncertainty distillation，也为后面和 NH baseline 中 DPO/self-play/preference debias 类方法比较提供了更强的同协议接口。
