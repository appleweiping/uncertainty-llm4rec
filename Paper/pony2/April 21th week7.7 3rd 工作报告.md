# April 21th week7.7 3rd 工作报告

今天这一步，我把 week7.7 从“Beauty 300-sample 的 trainable family 排序确认”正式推进到了“Beauty full teacher strengthening 已经真实落地、full973 SRPD 训练输入已经真实备齐”的阶段，而且这次不是只补计划，不是只补 batch 入口，也不是只补文档，而是把 DeepSeek 本地 API 这一侧真正跑起来，把 full teacher 链完整做完了。对当前项目来说，这一步的价值非常明确：在我已经知道 `SRPD-v2` 是 Beauty 300-sample server run 下 trainable winner、但整体 trainable line 仍然弱于 strongest hand-crafted baseline 的前提下，我不能直接硬开 outer compare，也不能直接把 week8 四域矩阵全烧起来；更合理的做法，是先把 Beauty 单域的 teacher 强度和训练样本规模抬到接近“封王级确认”的状态，再重新检查 `SRPD-v2` 这条 uncertainty-aware trainable 主线在更强 teacher、更多样本下到底能不能继续增强。因此，今天的核心任务不是再跑一轮旧的 `v1/v2/v3`，而是把 Beauty 上的 DeepSeek compact evidence 从旧的 100-sample 层推进到当前真实可用的 full 版本，并把这一整套结果沉淀成可以直接喂给 `SRPD-v1/v2/v3 full973` 的 teacher-data。

这一步里最先需要澄清的事实，是我口头上一直说的“1000”在 Beauty 上并不是一个字面可执行的数字。实际检查本地数据后，我确认 `data/processed/amazon_beauty/ranking_test.jsonl` 当前只有 973 条 ranking events，`data/processed/amazon_beauty/pairwise_coverage_test.jsonl` 当前只有 500 条 pairwise coverage rows，而 `data/processed/amazon_beauty/test.jsonl` 的 pointwise 全量则是 5838 条。因此，week7.7 这一轮真正诚实、可复现、可写进论文的方法口径，不应该再写成含糊的“Beauty 1000”，而应该写成：`pointwise full5838`、`candidate ranking full973`、`pairwise support full500`。这个口径非常重要，因为它直接决定后面所有 full973 配置、teacher-data summary、以及 server-side full973 SRPD train/eval/compare 的解释边界。如果这一点不写清楚，后面很容易把“full973 的 ranking confirmation”和“500-row pairwise support ceiling”混成一个虚假的 full1000 叙事。今天的第一项实际贡献，就是把这个边界用配置和真实运行结果固定下来，而不是继续停留在口头设想。

围绕这个边界，我先完成了 full 版本配置层的落地，包括 `beauty_deepseek_pointwise_full.yaml`、`beauty_deepseek_rank_full973.yaml`、`beauty_deepseek_pairwise_coverage_full500.yaml`，以及与之配套的 `qwen3 local full`、`structured risk full973`、`SRPD-v1/v2/v3 full973` teacher-data config、LoRA config 和 batch config。这一层的作用不是为了“多几个 yaml”，而是为了把当前项目的 full-stage 路线真正变成一条可以从 DeepSeek teacher 强化一路走到 Qwen3 trainable reconfirmation 的正式实验链：DeepSeek full pointwise 负责 uncertainty diagnosis 和 calibrated teacher 信号，DeepSeek full973 ranking 负责 direct teacher decision，DeepSeek structured risk full973 负责 strongest handcrafted teacher，DeepSeek pairwise full500 负责保留 v3 的 mechanism evidence，而 `qwen3_rank_beauty_srpd_v1/v2/v3_full973` 则把这一切对接成可以直接上服务器的训练输入。到这一步为止，week7.7 后半段已经不再是一个模糊计划，而是一套真实工程入口。

不过，今天真正最值钱的工作不只是配好 full 配置，而是把本地 DeepSeek API 调用速度问题真正解决掉。此前 pointwise、pairwise 和 API 侧 ranking 的推理链本质上是串行单请求，而且 `main_infer.py` 和 `main_pairwise.py` 还会在全部结束后才一次性写盘，这就导致两个问题：第一，速度极慢，pointwise full5838 按旧链路几乎是不可接受的几十小时级；第二，任何一次网络抖动都会让长跑任务从 0 开始，非常不适合 full 阶段。我今天针对这条链做的所有修改都严格限定在执行层，不改变 prompt、parser、uncertainty 定义、structured risk 公式、SRPD 训练目标和评价指标。具体来说，我在 `src/llm/api_backend.py` 里给 API backend 加上了受控并发的 `batch_generate`，在 `src/llm/inference.py` 里让 pointwise 和 pairwise 也真正按 backend 的 batch 跑起来，在 `configs/model/deepseek.yaml` 里显式加入了 `runtime.batch_size: 8` 和 `runtime.max_concurrency: 8`，并进一步在 `main_rank.py` 与 `main_pairwise.py` 中加入了 `--resume_partial` 与 `--checkpoint_every_batches` 机制，使得 ranking 和 pairwise 长任务能够批次级断点续跑。同时，我还把 `APIConnectionError`、`APITimeoutError` 这一类连接抖动正式纳入自动重试，而不再把它们直接当成不可恢复失败。这里的研究边界非常清楚：我改的是“怎么把相同方法更稳、更快地跑出来”，不是“为了速度去换一个更容易赢的实验定义”。

这些执行层优化不是停留在代码层面的“我觉得应该会更快”，而是已经得到了真实运行结果的支持。首先，`pointwise full5838` 已经在本地用 DeepSeek API 完整跑通，并完成了后续 eval 与 calibrate。最终原始 pointwise 诊断表显示 `num_samples = 5838`，`accuracy = 0.6565604659129839`，`avg_confidence = 0.7494552929085304`，`Brier = 0.26432216512504286`，`ECE = 0.21877697841726612`，`AUROC = 0.6184730581581528`。更重要的是，整条 full5838 pointwise 推理耗时大约 42 分钟，而根据我在优化前看到的串行速度估算，它原本会是几乎不可接受的超长时间。这说明受控并发对 API teacher strengthening 是真实有效的，而且不会改实验本身。其次，`candidate ranking full973` 也已经用同样的并发链跑完，中间一度出现过一次 DNS/连接级错误，但在我把 API 连接异常也纳入重试、并给 ranking 加了 checkpoint/resume 后，这条链重新跑通并最终完成，耗时大约 15 分钟。对比之前的串行预期，这同样是实打实的执行层提速。最后，`pairwise full500` 也已经跑完，耗时大约 3 分 40 秒。也就是说，今天我不只是证明了“DeepSeek API 理论上能跑 full”，而是已经把本地 DeepSeek full teacher 路线改造成了一条可接受、可恢复、可继续向前推进的真实执行链。

在 full-stage DeepSeek teacher 结果上，最重要的 ranking 结果也已经落地。`beauty_deepseek_rank_full973` 的 direct ranking 结果显示，在 973 条 ranking events 上，`HR@10 = 0.998972250770812`，`NDCG@10 = 0.6351559835664434`，`MRR = 0.5187541891952276`，`parse_success_rate = 0.9948612538540597`。在此基础上，我又用 `beauty_deepseek_pointwise_full` 的 calibrated uncertainty 作为信号，完成了 `beauty_deepseek_rank_full973_structured_risk` 的 structured-risk rerank。其结果是：`HR@10 = 1.0`，`NDCG@10 = 0.6355933223424297`，`MRR = 0.5189962315861597`，`changed_ranking_fraction = 0.0041109969167523125`，`avg_uncertainty_coverage_rate = 0.5005138746145941`。这个结果非常关键，因为它说明在 Beauty full973 范围里，structured risk 仍然比 direct ranking 略强，换句话说，当前 uncertainty-aware handcrafted mainline 在 full-stage teacher setting 下仍然站得住。这一点对于后面重新评估 `SRPD-v2 full973` 是否有机会追上 strongest handcrafted baseline 是一个非常重要的参照系：如果 full973 下 structured risk 依旧强，那 trainable line 是否能真正逼近它，就成为下一步 server-side full973 reconfirmation 的核心问题。

在 pairwise mechanism 这一侧，今天我也没有停在 raw pairwise predictions，而是把 full500 的 weighted pairwise-to-rank 结果完整做了出来。`beauty_deepseek_pairwise_coverage_full500_to_rank` 成功将 500 条 pairwise 结果对齐回 973 条 ranking events，对齐覆盖率为 `500 / 973 = 51.39%`。在这个 overlap subset 上，direct ranking 行给出的 `NDCG@10 = 0.6419210432266884`、`MRR = 0.5279623188405796`，而 `pairwise_to_rank_weighted_win_count` 的结果是 `NDCG@10 = 0.7669470017331042`、`MRR = 0.6983333333333334`，`parse_success_rate = 1.0`，`pairwise_supported_event_fraction = 0.513874614594039`，`pairwise_supported_candidate_fraction = 0.33333333333333326`，`pairwise_avg_reliability_weight = 0.6580272138067402`。这个结果再次验证了我此前的判断：weighted pairwise-to-rank 的确属于 uncertainty-aware mechanism line，而且它在当前 overlap subset 上依旧非常强。但它的强，不等于 trainable final method 已经赢；它更准确的角色，仍然是 supporting mechanism evidence，用来解释为什么 `SRPD-v3` 在设计上值得被保留，而不是直接替代 `SRPD-v2` 成为当前唯一主线。因此，今天这条结果并没有改变我对当前 inner hierarchy 的判断：它强化了 pairwise mechanism 的研究价值，但并没有自动让 `v3` 重新变成当前最强 trainable winner。

在 DeepSeek full teacher 真正跑完后，我继续把 `SRPD-v1/v2/v3 full973` 的 teacher-data 直接构建出来，这一步的意义在于把今天这轮本地 DeepSeek full 结果无缝转换成服务器训练的直接输入，而不是等到服务器上再手工拼接。结果非常干净：`qwen3_rank_beauty_srpd_v1_full973`、`qwen3_rank_beauty_srpd_v2_full973`、`qwen3_rank_beauty_srpd_v3_full973` 全部实现了 `973 / 973 matched`，状态均为 `srpd_teacher_data_ready`。其中，`v2` summary 显示 `778 train / 195 valid`，`avg_sample_weight = 1.198151`；`v3` summary 则显示 `778 train / 195 valid`，`avg_sample_weight = 1.265779`，`avg_pairwise_preferences = 0.513875`。这说明 full973 阶段的 trainable line 已经不再缺输入，不再缺 teacher，也不再缺 pairwise support rows。对当前项目状态而言，这一步非常重要，因为它把“本地 DeepSeek full teacher 强化”和“服务器 Qwen3 trainable reconfirmation”正式连成了一条完整链：今天本地已经把该做的 DeepSeek full teacher 全做完了，接下来服务器只需要聚焦在 `Qwen3 SRPD-v1/v2/v3 full973 train/eval/compare` 本身，而不需要再为 teacher 侧补洞。

基于今天完成的全部工作，我对项目当前阶段的判断比昨天更清楚了。第一，`SRPD-v2` 依然是当前最应该优先继续确认的 trainable 主线。理由不是偏好，而是已有真实结果和 full-stage teacher strengthening 一起指向它：在 Beauty 300-sample server run 中，它已经是 `v1/v2/v3` 中表现最好的 trainable 版本；而今天 full973 teacher-data 构建完成后，它又成为最自然、信息增益最高的 first full973 candidate。第二，当前项目并没有偏离 uncertainty 主线。相反，今天 full-stage 的所有关键推进都进一步说明，当前最有研究价值的方向仍然是 uncertainty-aware ranking，而不是随意换一条更好讲故事的路线：DeepSeek structured risk full973 仍然略胜 direct ranking；`SRPD-v2 full973` 的 sample weights 已经清楚地把 uncertainty 信号写进训练输入；而 `v3` 目前依旧只是保留为 mechanism-enhanced variant，而不是被硬写成 final winner。第三，今天并不意味着项目已经具备立刻打开 outer compare 的资格。虽然 DeepSeek full teacher 强化已经成功，但真正决定 trainable line 是否有资格对外比较的，不是 teacher 侧更强了，而是 full973 下的 `Qwen3 SRPD-v2` 能不能在服务器上逼近甚至超过 strongest handcrafted baseline。因此，最合理的下一步不是全面烧服务器，不是同时开 `v1/v2/v3 full973` 和 outer compare，而是先走方案 A：优先跑 `Qwen3 direct rank full973`、`Qwen3 structured risk full973`、`SRPD-v2 full973 startup/train/eval/compare`，用最小但最有信息增益的一组 full973 server experiments，确认当前 trainable winner 在更强 teacher、更大样本下到底是否继续成立。如果这一轮 `SRPD-v2 full973` 变强并逼近或超过 strongest handcrafted baseline，那么后续再补完整 `v1/v3 full973` 和 outer compare 才有意义；如果它依旧明显落后，那么项目下一步应优先继续优化 trainable uncertainty line，而不是急着对外宣战。换句话说，今天这份 3rd day 报告的真正结论，不是“我们已经赢了”，而是“我已经把 Beauty full teacher strengthening 真正做完了，已经把下一步 full973 server reconfirmation 的路彻底铺平，而且我现在知道该优先跑哪一条、为什么优先跑它，以及为什么这一步仍然没有偏离 uncertainty 主线”。
