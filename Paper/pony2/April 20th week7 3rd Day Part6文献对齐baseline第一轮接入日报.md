**April 20th week7 3rd Day Part6文献对齐baseline第一轮接入日报**

今天进入 week7 第三个关键任务，也就是 Part6 的 literature-aligned baseline 第一轮接入。和前两天一样，今天我没有回头新增 ranking family，也没有把 fully fused 或 local swap 重新抬成默认主线，因为 week6 已经完成方法空间的收敛，当前默认主线仍然是 structured risk current best family。今天真正要解决的是另一个顶会级风险：如果我的实验一直只在自己定义的 uncertainty-aware 框架里和自己比较，reviewer 很容易质疑它是封闭式内部 compare。因此今天的目标不是堆 baseline 名字，而是先建立一组最小但严格的文献范式对齐 baseline，让 candidate ranking 和 pairwise preference 都能在同一候选集、同一 split、同一指标下拥有外部可解释的对照对象。这一步的价值不在于指标一定多高，而在于项目开始具备从“方法内部自洽”走向“可与文献组织方式正面对照”的防守能力。

代码层面，今天新增了 `src/baselines/literature_rank_baseline.py`、`src/baselines/literature_pairwise_baseline.py` 和统一入口 `main_run_literature_baselines.py`。ranking 侧没有另起一套不可控逻辑，而是在已有 candidate-order、popularity-prior、longtail-prior 的基础上补了 `history_overlap_rank`，用用户历史文本和候选文本的轻量 token overlap 作为不使用 uncertainty 的内容相似度 baseline。这个 baseline 很朴素，但它的好处是和当前 candidate ranking task 完全共享候选集和评价口径，不需要引入新的数据划分，也不会把文献对齐变成另一个独立任务。pairwise 侧新增了 `history_overlap_pairwise` 和 `popularity_prior_pairwise`，它们直接读取当前 pairwise 样本，输出和 LLM pairwise 一致的 `preferred_item_pred` schema，因此可以复用 `pairwise_accuracy`、parse success、confidence bins 等同一套评价逻辑。为了兼容旧入口，我保留并重写了 `src/baselines/literature_rank_baselines.py` 作为兼容导出，不让此前的 `main_literature_baseline.py` 断掉。

配置和接口层面，今天新增 `configs/baseline/week7_literature_baselines.yaml`，把 Beauty 域、Llama 3.1 8B Instruct base-only 身份、ranking input、pairwise input、baseline 列表、summary path 和 `max_samples=100` 固定下来。这个配置里显式保留了 `model_family=local_hf_base_only` 和空的 `adapter_path`，这不是装饰字段，而是为了把“base-only 本地 7B/8B -> 可选 LoRA adapter -> 后续可选 vLLM serving”的路线提前接进 baseline 体系。今天不训练 LoRA，也不加载 adapter，但表头已经能表达未来 adapter 身份；这样后续如果 reviewer-gap 需要 base+adapter 对照，不需要重写 baseline summary schema。`requirements.txt` 也补入 `PyYAML`，因为项目配置系统长期依赖 YAML，服务器环境需要明确安装这个依赖，而不能靠隐式环境碰运气。

结果层面，今天已经用 `py -3.12 main_run_literature_baselines.py --config configs/baseline/week7_literature_baselines.yaml` 跑通第一轮 100-sample Beauty baseline，并生成 `outputs/summary/week7_day3_literature_baseline_summary.csv`。ranking 侧四个 baseline 都产出了 task-native `rank_predictions.jsonl` 和 `ranking_metrics.csv`；pairwise 侧两个 baseline 也产出了 `pairwise_predictions.jsonl` 和 `pairwise_metrics.csv`。从当前 compact 结果看，ranking 侧因为候选集只有 6 个而评价是 @10，所以 HR@10 天然饱和，这个指标在当前 setting 下不能作为强结论；更应该关注 NDCG@10 和 MRR，以及 baseline 是否在同一候选集、同一评价口径下可运行。当前 `history_overlap_rank` 在 100-sample compact 设置下取得了较高的 NDCG/MRR，说明即使是不使用 uncertainty 的轻量内容相似度 baseline，也能提供一个有意义的外部参照；pairwise 侧 `history_overlap_pairwise` 和 `popularity_prior_pairwise` 分别给出了可解释的 preference 对照，后者在当前样本上 pairwise accuracy 更高，但这仍然只是第一轮 compact baseline，不应被写成最终结论。

文档层面，今天新增 `docs/week7_day3_literature_baseline_notes.md`，明确说明这些 baseline 的文献对齐含义：它们不是宣称完整复现 OpenP5、EvaLLMasRecommender 或 Exp3rt，而是把这些工作强调的统一 task schema、同 setting ranking/reranking 评测、pairwise preference 机制任务等组织原则，先落到当前项目可以真实运行的最小 baseline 组里。README 也同步补入 Part6 第一轮 literature-aligned baseline 的入口和产物，但保持项目级写法，没有写成机械 day log，也没有暴露过细实验细节。`Paper/pony2/version pony week7 工作计划.md` 同步补充 day3 当前执行状态，说明本轮已经落成 ranking 与 pairwise 两类 baseline、统一入口和 base-only / adapter-ready schema。

今天的阶段性结论是，version pony 已经从 Part5 的内部方法闭环，进一步走向 Part6 的外部防守结构。当前 baseline 还不是最终 literature benchmark，也不是大规模强基线全集，但它已经完成了最重要的第一步：ranking 和 pairwise 都有了同任务、同候选集、同指标、可运行、可汇总的 literature-aligned baseline 组；base-only 本地模型身份和未来 LoRA adapter 身份也已经进入同一张 summary schema。接下来 week7 第四天要做的，不是再继续乱加 baseline，而是把 uncertainty-source、decision-formulation 和今天的 literature-aligned baseline 放进统一 baseline matrix，明确哪些 baseline 回答 uncertainty 定义问题，哪些 baseline 回答 decision formulation 问题，哪些 baseline 回答外部文献防守问题。
