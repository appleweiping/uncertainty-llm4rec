Qwen3-8B 本地最小生成测试通过之后，week7 day1 的服务器本地主实验后端已经从“配置可用”推进到“真实加载、chat template 与 GPU 推理可用”。但这一步也暴露出一个必须在大实验前先处理的执行层问题：Qwen3 默认可能输出 `<think>...</think>` 推理内容，而 version pony 现有的 pointwise、candidate ranking、pairwise 三条入口原本假设模型直接返回最终 JSON 或可解析文本。如果不先修复这一层，后续 batch 即使能跑，也会把 parser 失败、JSON 截断、ranking 列表污染等问题带进结果表，最终影响的是实验可信度而不是方法本身。因此这次修复没有继续推进 medium-scale，也没有调整 structured risk、pairwise mechanism line、local swap 或 fully fused 的研究定位，而是专门把 Qwen3 输出层兼容性收紧到可控状态。

这次我先检查了本地后端、prompt 和 parser 的真实链路。`configs/model/qwen3_8b_local.yaml` 原本已经固定了 ModelScope 本地路径、`local_files_only` 和 base-only 路线，但还没有显式关闭 thinking；三类 prompt 原本要求 strict JSON，但没有直接禁止 hidden reasoning 或 `<think>` block；统一 parser 位于 `src/llm/parser.py`，pointwise、candidate ranking 和 pairwise 最终都会经过这里解析。基于这个结构，最稳妥的修法不是只在某个任务入口临时截字符串，而是做三层兼容：第一，在 Qwen3 model config 中加入 `enable_thinking: false`；第二，让 `src/llm/local_hf_backend.py` 在调用 chat template 时尽量把 `enable_thinking` 传给 tokenizer，如果当前 tokenizer 不支持该参数则自动回退；第三，在 `src/llm/parser.py` 增加统一的 `strip_thinking_blocks` 清洗函数，在三类 parser 进入 JSON 抽取、正则抽取和文本偏好抽取前先去掉 `<think>...</think>`。这样即使模型仍然输出 thinking 段，后续 parser 也只会看到最终答案。

具体文件上，本轮修改集中在执行兼容层。`src/llm/parser.py` 新增了 thinking block 清洗逻辑，并让 `parse_pointwise_response`、`parse_candidate_ranking_response`、`parse_pairwise_preference_response` 共享这一层；`src/llm/local_hf_backend.py` 新增 `enable_thinking` 参数并在 chat template 调用中传递，保持对 Qwen3 友好，同时兼容其他 tokenizer；`src/llm/__init__.py` 从 model config 的 generation 字段读取 `enable_thinking`；`configs/model/qwen3_8b_local.yaml` 和 `configs/model/local_llm.yaml` 显式设置 `enable_thinking: false`；`prompts/pointwise_yesno.txt`、`prompts/candidate_ranking.txt`、`prompts/pairwise_preference.txt` 现在都明确要求不输出 hidden reasoning、chain-of-thought、analysis 或 `<think>` block，只返回最终 JSON；`main_backend_check.py` 的三类最小 prompt 也同步加入 final-only 约束，避免 smoke check 自己成为污染源。README 和 week7 后端文档同步说明了这只是 Qwen3 执行兼容修复，不改变 week6 方法主线。

本地验证方面，我用 synthetic Qwen3-style 输出分别测试了 pointwise、candidate ranking 和 pairwise：输入中都包含 `<think>...</think>` 段，清洗后 pointwise 能正常解析 `recommend/confidence/reason`，candidate ranking 能正常解析 `ranked_item_ids/topk_item_ids/confidence/reason` 且 `parse_success=True`，pairwise 能正常解析 `preferred_item/confidence/reason` 且 `parse_success=True`。随后运行了 Qwen3 backend check 的 dry-run，确认 `configs/model/qwen3_8b_local.yaml` 仍然能进入三任务 backend check schema；也运行了 `configs/batch/week7_local_scale.yaml` 的 dry-run，确认 pointwise、candidate ranking、pairwise 三条 smoke config 仍然能进入 batch registry。由于当前本地机器并不持有服务器上的 `/home/ajifang/autodl-tmp/models/Qwen3-8B` GPU 模型目录，本地没有伪造非 dry-run GPU smoke 结果；真正的 parse_success 恢复需要在服务器上用同一套代码执行 `main_backend_check.py` 和 `RUN_SMOKE=1` 的三任务 smoke。

这次修复后的阶段判断很清楚：Qwen3-8B 可以继续作为 week7 之后的服务器主实验执行 backbone，但在进入 medium-scale 之前，必须先在服务器上重新跑一次 backend check 和三任务 smoke，确认 raw response 中即便出现 thinking，parser 也能稳定吃到最终 JSON。这个问题的性质是执行后端适配，不是方法改动；structured risk 仍然是 current best ranking family，pairwise 仍然是机制层证据线，local swap 和 fully fused 仍然保留为 retained exploratory family。做完这一层之后，后续 batch 才能更放心地扩大到 Beauty medium-scale，并继续承接 week8 的多域、多模型和 robustness 矩阵。
