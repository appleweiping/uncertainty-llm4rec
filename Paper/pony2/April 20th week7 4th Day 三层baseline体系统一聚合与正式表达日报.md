今天我没有继续新增 ranking family，也没有把 fully fused 或 local swap 拉回默认主实验主线，而是把 week7 第四天定位为 Part6 baseline 证据层的正式结构化收口。这个选择是必要的，因为 week5 和 week6 已经完成了 Part5 的方法闭环，week7 前三天也已经把服务器本地主后端、batch/registry 和第一轮 literature-aligned baseline 接入项目结构；如果这时继续只分散地保留 estimator compare、same-task baseline 和 literature baseline，那么后续 week8 一旦进入多域、多模型、多噪声矩阵，baseline 叙事就会重新变成手工解释。今天要解决的不是“再多一个对照方法”，而是让 reviewer 能看清楚三件事：uncertainty 如何定义，uncertainty 是否改善同任务决策，以及当前方法是否能和接近文献范式的 baseline 在同一 setting 下比较。

工程上，今天新增了 `src/analysis/aggregate_baseline_results.py` 和统一入口 `main_compare_baselines.py`。前者负责读取现有三类核心表：`outputs/summary/week6_day3_estimator_compare.csv`、`outputs/summary/week6_day4_decision_baseline_compare.csv` 和 `outputs/summary/week7_day3_literature_baseline_summary.csv`，并把它们规范到同一个 baseline matrix schema；后者是面向后续本地和服务器运行的稳定入口，默认输出 `outputs/summary/week7_day4_baseline_matrix.csv`。这套实现里我没有重算方法指标，也没有重新定义实验口径，而是保留每一行的来源文件、任务、模型、baseline layer、baseline family、baseline name、模型来源组、adapter 字段、当前角色和证据问题。这样后续服务器上补 Llama 3.1 8B Instruct 的真实运行、或者后续进入 LoRA-adapted model 时，不需要再推翻表结构，只需要让新的 summary 进入同一个聚合入口。

今天生成的核心结果是 `outputs/summary/week7_day4_baseline_matrix.csv`，当前共 45 行，其中 uncertainty-source 层 27 行，decision-formulation 层 12 行，literature-aligned 层 6 行。这个比例本身符合当前项目状态：uncertainty-source 层来自 week6 day3 的 estimator compare，覆盖 raw verbalized、calibrated、self-consistency、fused 等不同 uncertainty source；decision-formulation 层来自 week6 day4 的 same-task baseline 与 uncertainty-aware decision family，包括 direct candidate ranking、plain pairwise aggregation、structured risk current best family、local swap 和 fully fused retained family；literature-aligned 层来自 week7 day3，覆盖 candidate-order、popularity-prior、longtail-prior、history-overlap ranking，以及 pairwise 侧的 history-overlap 和 popularity-prior baseline。今天还新增了 `docs/week7_day4_baseline_system.md`，用来解释三层 baseline 各自回答什么问题，以及 API 组、本地 HF 主模型组、未来 LoRA 组在实验体系里的身份边界。README 也同步更新了 baseline matrix 入口和产物位置，但保持项目级表达，没有写成逐条命令教程。

从结果解释上，今天最重要的结论不是某一个 baseline 指标高低，而是 version pony 的 baseline 体系第一次具备了论文级结构。uncertainty-source baseline 现在被明确定位为回答“uncertainty 如何定义以及哪类 source 更稳定”；decision-formulation baseline 被定位为回答“同任务下不使用 uncertainty 的方法与 uncertainty-aware 决策之间有什么差异”；literature-aligned baseline 被定位为回答“当前方法是否已经能在同候选集、同划分、同指标下与接近文献范式的对照方法比较”。同时，模型身份也被写清楚：官方 API 组仍然是外部黑盒观察和小规模 cross-model 参考，本地 Hugging Face 小模型组才是后续主实验吞吐和系统比较的核心，LoRA-adapted model 现在只预留接口，不在 day4 强行展开。这避免了后面把 API 观察结果、本地 base-only 结果和未来 adapter 结果混成一个不清楚的模型集合。

今天的边界也需要明确记录。当前矩阵仍然复用了 Part5 compact evidence 的既有结果，因此它是 baseline 体系的结构化收口，不是服务器真实 scale-up 的最终结论。部分 uncertainty-source 与 decision-formulation 行来自 Beauty + Qwen 的 compact evidence，literature-aligned 行则已经带有 Llama 3.1 8B Instruct local-HF 的 base-only 身份字段。这个状态是诚实且合理的：今天先把 schema、入口和叙事边界定住，后续上服务器跑真实 local-HF 主实验时，同一张矩阵可以直接吸收新结果，而不是重新手工拼表。structured risk 仍然是 current best ranking family；local swap 和 fully fused 继续作为 retained exploratory family 保留；pairwise 继续作为机制层证据，而不是主决策替代线。完成这一步之后，week7 已经从单纯“有 baseline 文件”推进到“有 baseline 证据系统”，可以继续承接后续服务器运行和 week8 的五维验证矩阵。
