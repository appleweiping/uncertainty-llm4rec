# week6 magic 7th Day 四域DeepSeek最小跨域复制与承接骨架落地日报

在 Part5 方法闭环和 figure pack 已经完成之后，这一天我没有直接发散到多模型全矩阵，而是优先做四域统一 DeepSeek 的最小跨域复制。这样安排是因为当前论文最需要补的不是更多模型名字，而是回答一个更基础也更尖锐的问题：week6 的结论是不是 Beauty + Qwen 的局部现象。为了把这个问题压实，我把 Movies、Beauty、Books、Electronics 四个正式数据域放进同一套 compact DeepSeek 口径，每个域先控制在 100 个 sample 左右，优先推进 pointwise diagnosis 和 candidate ranking 两条主线。pairwise 这一天没有强行四域铺开，因为它当前最需要修的是 coverage，而不是在覆盖不足的状态下复制更多域；主线任务先形成四域最小证据矩阵，才是更稳的节奏。

工程上，我补齐了八份四域最小配置：`configs/exp/movies_deepseek_pointwise.yaml`、`configs/exp/movies_deepseek_rank.yaml`、`configs/exp/beauty_deepseek_pointwise.yaml`、`configs/exp/beauty_deepseek_rank.yaml`、`configs/exp/books_deepseek_pointwise.yaml`、`configs/exp/books_deepseek_rank.yaml`、`configs/exp/electronics_deepseek_pointwise.yaml` 和 `configs/exp/electronics_deepseek_rank.yaml`。这些配置显式写清了 input path、exp_name、output_dir、model config 和 max_samples，并统一指向 DeepSeek 通道。考虑到当前 Movies、Books 和 Electronics 的完整 processed split 已经通过 compact small-domain 文件稳定存在，这一轮使用这些 compact domain slices 承接四域最小矩阵；Beauty 则使用完整 Beauty processed split。代码层面，我新增了 `src/analysis/aggregate_cross_domain_minimal_results.py`，用于统一读取四域 pointwise/rank 结果并生成最小 compare 表；新增了 `src/baselines/literature_rank_baselines.py` 和 `main_literature_baseline.py`，把 literature-aligned baseline 从计划项推进成一个真实 runnable 的 schema bridge；同时新增了 `src/utils/exp_registry.py` 和 `main_batch_run.py`，让四域实验和 baseline 实验能被统一登记到状态表里。

真实执行状态上，这一天完成了四域 candidate ranking 的 DeepSeek 100-sample 推理和评估。Movies、Beauty、Books、Electronics 四个域都已经生成 `outputs/{domain}_deepseek_rank/predictions/rank_predictions.jsonl`，并通过 `main_eval_rank.py` 生成对应 `ranking_metrics.csv`。四域 pointwise 侧复用了已有 100-sample DeepSeek 诊断结果：Movies 对应 `movies_small_deepseek`，Beauty 对应 `beauty_deepseek`，Books 对应 `books_small_deepseek`，Electronics 对应 `electronics_small_deepseek`。最终 `outputs/summary/week6_magic7_4domain_deepseek_status.csv` 显示四个域在 pointwise 和 rank 两类任务下都有可用评估行，其中 rank 是本日新跑出的统一 DeepSeek listwise 结果；`outputs/summary/week6_magic7_4domain_deepseek_compare.csv` 则把 domain、task、model、samples、ECE、Brier、HR@10、NDCG@10、MRR 和 parse_success_rate 统一放进一张表里。除此之外，我还生成了 `outputs/summary/week6_magic7_literature_baseline_status.csv`、`outputs/summary/week6_magic7_batch_status.csv` 和 `outputs/summary/week6_magic7_cross_domain_notes.md`。

从四域最小矩阵得到的判断要分清“可以说什么”和“还不能说什么”。现在可以说的是，version pony 已经具备把 Part5 主线复制到四个 Amazon 域上的工程能力，且四域 candidate ranking 都能在统一 DeepSeek 后端下形成真实 listwise 预测和指标表。当前 100-sample 结果里，Books 的 candidate ranking 指标最高，Electronics 次之，Beauty 和 Movies 相对更低，这说明不同域的 listwise 难度和语义结构差异已经开始显现；pointwise 诊断层也显示不同域的 ECE/Brier 差异明显，不能把 Beauty 上的 calibration 结论简单外推到所有域。还不能过度外推的是，这个矩阵仍然只是 compact 100-sample evidence matrix，不是最终大规模主实验；它还没有接入 structured risk current best family 的四域 rerank，也没有接入完整 literature-aligned baseline 和 robustness setting。因此它支撑的是“跨域复制骨架已经成立”和“后续扩展方向清楚”，不是最终顶会主结果的强结论。

literature-aligned baseline 和 batch/registry 也从这一天开始正式进入项目结构。`main_literature_baseline.py` 当前先实现了一个 popularity prior ranking baseline，它不是最终强 baseline 集合，而是用于验证统一输入输出 schema、ranking metrics 和 baseline status 表能否闭合；这一步让 baseline 不再只是 week7/week8 计划里的文字，而是变成项目里真实可运行、可登记、可扩展的入口。`main_batch_run.py` 和 `src/utils/exp_registry.py` 当前也只是最小状态登记骨架，但已经能记录 exp_name、domain、task、model、status、output_dir 和 eval_ready，为后续中等规模 batch、四域 structured risk rerank、强 baseline 接入和 robustness 矩阵打底。到这一天结束，week6 的后续承接已经从“方法闭环完成”推进到“证据系统开始跨域复制”，下一步就应该沿着这个矩阵继续接 structured risk 主线、pairwise coverage 修复和 literature-aligned baseline，而不是回头继续扩新的 ranking family。
