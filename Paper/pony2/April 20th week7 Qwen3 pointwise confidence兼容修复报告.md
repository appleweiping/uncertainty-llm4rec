服务器端重新运行 Qwen3 backend check 后，当前问题已经从模型加载、设备放置和 candidate/pairwise 解析，收敛到 pointwise 输出格式兼容。`candidate_ranking` 与 `pairwise_preference` 已经能够 `parse_success=True`，说明 local HF backend、Qwen3 本地路径、chat template、GPU 推理和 `<think>` 清洗主链基本打通；剩下的失败来自 pointwise 输出中的 `confidence` 被 Qwen3 写成 `"low"`，而原 parser 期望它是可以直接转成 float 的 0 到 1 数值。这不是方法层问题，也不是 Part5/Part6 研究结论变化，而是 Qwen3 执行层输出格式还需要更稳的约束和兜底。

本轮修复保持最小范围。第一，`prompts/pointwise_yesno.txt` 中对 `confidence` 的要求进一步收紧，从“0 到 1 的 number”改成明确的 numeric value，并禁止使用 `"low"`、`"medium"`、`"high"` 这类等级词。第二，`main_backend_check.py` 中 pointwise smoke prompt 同步加入同样约束，避免 backend check 自己给模型留下输出等级词的空间。第三，`src/llm/parser.py` 的 `_normalize_confidence` 增加了保守等级映射：`low -> 0.2`、`medium/mid/moderate -> 0.5`、`high -> 0.8`，并额外支持 `very low -> 0.1`、`very high -> 0.9`。这个映射只作为执行兼容兜底，正常路径仍然优先解析数值型 confidence；如果模型按 prompt 返回 0 到 1 数字，parser 行为不会改变。

本地验证使用了 Qwen3 当前失败样式的 synthetic 输出，例如 `{"recommend":"no","confidence":"low","reason":"..."}`，修复后 pointwise parser 能返回 `recommend=no`、`confidence=0.2`，不会再抛出 `ValueError`。同时也验证了带 `<think>...</think>` 的 pointwise 输出、candidate ranking 中的 `"confidence":"high"` 和 pairwise 中的 `"confidence":"low"`，三类 parser 都能稳定归一化 confidence 并继续给出可解析结果。`main_backend_check.py` 和 `src/llm/parser.py` 的语法检查通过，本地 dry-run backend check 与 `week7_local_scale` batch dry-run 也通过。由于本地环境没有服务器 GPU 模型目录，真实 backend check 仍应以服务器重新运行结果为准。

修复后的服务器下一步很明确：在项目根目录重新执行 `python main_backend_check.py --model_config configs/model/qwen3_8b_local.yaml --status_path outputs/summary/week7_day1_backend_check.csv`，目标是 pointwise、candidate ranking、pairwise 三条 backend check 都不再因为输出格式失败；如果三条都通过，再执行 `RUN_SMOKE=1 bash scripts/week7_day1_server_backend_check.sh "$PWD"`，最后才进入 `python main_batch_run.py --batch_config configs/batch/week7_local_scale.yaml --run`。这条路线继续保持 week7 的边界：Qwen3-8B local backend 是执行 backbone，structured risk 仍然是 current best ranking family，pairwise 仍然是 mechanism line，local swap 和 fully fused 仍然只是 retained exploratory family。
