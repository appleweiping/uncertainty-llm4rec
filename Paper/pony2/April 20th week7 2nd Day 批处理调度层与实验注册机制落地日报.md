**April 20th week7 2nd Day 批处理调度层与实验注册机制落地日报**

今天进入 week7 第二个执行层任务，我没有继续扩新的 ranking family，也没有把 local swap 或 fully fused 重新拉回默认主实验主线，而是把注意力放在批处理调度层、实验注册机制和服务器运行流程上。这个选择和 week7 的整体定位是一致的：week5 和 week6 已经证明 Part5 的多任务方法闭环成立，week7 真正要解决的是这套方法能不能被稳定执行、失败后能不能恢复、结果能不能被统一追踪。换句话说，今天的核心问题不是“再多一个方法能不能涨点”，而是“当 pointwise、candidate ranking、pairwise 三条任务线同时进入服务器本地 HF 后端以后，我是否还在靠手工记命令维持实验”。顶会级实验不能建立在人工记忆上，尤其后面 week8 会继续扩到多域、多模型、多噪声和更强 baseline；如果没有 registry 和 batch 层，所有结果都会重新散落成不可追踪的输出目录。

工程实现上，今天首先重构了 `main_batch_run.py`，把它从早期的 status-only registry 小脚本推进成真正的 batch 入口。它现在支持 `--batch_config` 读取统一 batch 清单，也保留 `--configs` 作为临时兼容入口；默认模式是 dry-run 和注册检查，不会在本地误启动推理，只有显式加 `--run` 才会真正调起实验。这个安全默认很重要，因为当前真实工作流是本地写代码和推 GitHub，服务器再 pull 代码运行；我不能让本地框架验证误写成服务器真实推理。配套新增的 `src/utils/exp_launcher.py` 负责把每个 experiment spec 解析成标准命令，自动根据 task 选择 `main_infer.py`、`main_rank.py` 或 `main_pairwise.py`，并记录输入路径、输出路径、模型、method family、method variant、命令、日志路径、返回码和耗时。`src/utils/exp_registry.py` 则被扩展成更完整的 registry schema，能够记录 `batch_name`、`exp_name`、`domain`、`task`、`model`、`status`、`output_dir`、`retry_count`、`latency_sec`、`error_message`、`stdout_path`、`stderr_path`、`dry_run` 等字段，使失败恢复和后续聚合有了稳定依据。

配置与运行约定方面，今天新增 `configs/batch/week7_local_scale.yaml`，先把 Beauty 域的三条本地 HF smoke 任务纳入同一个小型 batch：pointwise、candidate ranking 和 pairwise preference。这个 batch 不是中等规模实验，也不是 week8 矩阵，只是为了验证三类任务是否能被同一套调度层识别、记录和启动。服务器侧新增 `scripts/run_sync_and_batch.sh`，把进入仓库、可选 `git pull --ff-only`、选择 batch config、设置 `PYTHON_BIN`、执行 `main_batch_run.py` 这些动作固定成一个清楚的脚本入口；文档侧新增 `docs/week7_day2_batch_workflow.md` 和 `docs/week7_server_execution.md`，分别说明 batch 清单写法、默认 dry-run 逻辑、真实 `--run` 逻辑、失败后 `--only_failed` 续跑、以及本地 push / 服务器 pull / 激活 conda / 检查 backend / 启动 batch / 查看 registry 的完整执行约定。README 也同步补了 Week7 batch registry 的项目级说明，但没有写入任何服务器密码、临时连接、端口或远程桌面信息。

今天的本地验证需要诚实表述。由于真实 Llama 3.1 8B Instruct 推理要在服务器 conda 环境中完成，我没有在本地伪造 GPU 加载结果，而是先完成了 batch 框架的源码级检查：`main_batch_run.py`、`src/utils/exp_launcher.py` 和 `src/utils/exp_registry.py` 已经通过 AST 解析检查，说明新增入口和工具层没有语法层面的阻塞。当前本地默认 Python 环境缺少 `PyYAML`，因此不能在这台工作站上直接跑出 `outputs/summary/week7_day2_batch_status.csv`，这个状态表需要在安装了项目依赖的服务器 conda 环境中生成。这个边界不是功能回避，而是执行环境边界：batch registry 的 schema、路径解析、命令构造、dry-run 安全边界和失败状态记录已经在代码中落地，真实 registry 文件要等服务器环境确认依赖后补跑。后续上服务器后，只需要在确认模型路径、依赖环境和 backend check 通过之后，先运行 `PYTHON_BIN=/path/to/env/bin/python python main_batch_run.py --batch_config configs/batch/week7_local_scale.yaml` 生成 dry-run 状态表，再运行带 `--run` 的真实版本；如果某些任务失败，再用 `--only_failed --run` 只重跑失败项，而不是整批靠手工命令重来。

今天的阶段性贡献，是把 version pony 从“已经有服务器本地 HF 后端”推进到“已经有结构化实验队列和可恢复 registry”。这一步不会直接给论文增加一个新指标，但它为后面所有指标的可信性打底：每个实验都有 batch 身份、任务身份、模型身份、方法身份、输出目录、日志位置和错误信息，后续 week7 的 literature-aligned baseline、week7 后段的中等规模验证、以及 week8 的多维矩阵都可以在这个结构上继续扩展。今天的边界也很清楚：没有新增 ranking family，没有运行 medium-scale，没有训练 LoRA，没有启动 vLLM，也没有把 API 重新变成主实验通道。当前默认主线仍然是 week6 收敛出的 structured risk family，pairwise 仍然是机制层证据线；今天做的是让这些已经确定的研究对象进入可调度、可追踪、可恢复的执行系统。做完这一步之后，后续上服务器跑真实配置时，我们不再是“开终端临时敲三条命令”，而是可以围绕 batch config 和 registry 有组织地推进。
