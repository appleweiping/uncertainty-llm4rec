今天的工作不是去追求多跑几个实验，也不是把旧的 pointwise 主链推翻重写，而是先把 version pony 在 week5 最关键的第一块地基钉死，也就是把“多决策任务统一框架”的概念真正压进数据结构层。现有仓库在这之前已经有一条相对稳定的旧主链：`main_preprocess.py -> main_build_samples.py -> main_infer.py -> main_eval.py -> main_calibrate.py -> main_rerank.py`，它围绕 pointwise yes/no recommendation 形成了完整闭环。这条链今天没有被破坏，因为在 version pony 里，pointwise 不会消失，而是要从“最终任务”重新定位为“uncertainty diagnosis 的基础观测层”。因此，今天真正完成的，不是替换旧样本构造，而是在旧 pointwise 样本之上建立新的多任务派生层，让 pointwise、candidate ranking、pairwise preference 三层任务第一次在同一用户、同一正例来源、同一时间戳上形成可追踪对齐。

具体实现上，我新增了 `src/data/candidate_ranking_builder.py` 和 `src/data/pairwise_preference_builder.py`，并补了统一主入口 `main_build_multitask_samples.py`。第一个 builder 直接从现有 `valid.jsonl`、`test.jsonl` 这类 pointwise eval split 聚合样本，不去重新发明数据来源，而是把同一 `(user_id, timestamp)` 事件上的候选集合重建成一条 ranking sample。这样做的好处非常直接：一方面，新任务天然继承了旧主链里已经被验证过的数据切分与负采样逻辑；另一方面，后面无论是 ranking prompt、ranking parser、ranking eval，还是 uncertainty-aware rerank，都能在统一的 `source_event_id` 上把 pointwise diagnosis 层和 ranking 主决策层对上。为了避免 ranking 候选顺序继续沿用 pointwise 文件里“正样本先写、负样本后写”的潜在位置偏差，我在 builder 层加入了可复现的稳定打乱逻辑，并把打乱后的 `positive_item_index` 明确写进 schema，这样后面既能保持任务公平性，也能保留可追踪性。

在 schema 上，我没有偷懒只保留最低限度字段，而是按后续 week5 与 week6 的主线需求直接预留了结构。ranking 样本现在至少包括 `user_id`、`history`、`candidate_item_ids`、`candidate_titles`、`candidate_texts`、`candidate_popularity_groups`、`positive_item_id`、`timestamp`、`split_name`，同时额外带上了 `candidate_labels`、`source_event_id`、`positive_item_index`、`source_pointwise_size`、`num_candidates`。这意味着它不只是一个临时给 prompt 吃的输入文件，而是一个后续可同时服务 ranking inference、ranking evaluation、ranking rerank 和多任务 compare 的标准接口。与之对应，pairwise builder 则不直接从原始 interactions 再造一条支线，而是明确从 ranking 样本继续派生，这一点对 version pony 非常重要，因为它把 pairwise preference 牢牢固定成了“中间机制层”，而不是另一个游离任务。当前第一版 pairwise 采用的是 `positive_vs_negative` 结构，但 schema 一开始就保留了 `pair_type`，并把 `source_positive_item_id`、`source_event_id`、`split_name`、`source_candidate_count` 这些字段写死，为后续扩展到 `hard_negative_vs_negative`、`head_vs_tail` 等更复杂 pairwise 结构留好了接口，不会到 week6 再返工数据层。

配置层今天也进入了主项目，而不是停留在脑图里。我新增了 `configs/task/candidate_ranking.yaml` 和 `configs/task/pairwise_preference.yaml`，把 source splits、输出前缀、候选打乱、pair 顺序打乱和 pair type 这类 task-level 决策从脚本内硬编码里抽离出来。同时我补了三份实验配置模板：`configs/exp/beauty_qwen_pointwise.yaml`、`configs/exp/beauty_qwen_rank.yaml`、`configs/exp/beauty_qwen_pairwise.yaml`。这三份模板现在已经开始显式包含 `data_config`、`task_type`、`model_config`、`input_path`、`output_dir`、`exp_name`、`max_samples` 这些关键字段。虽然 ranking 与 pairwise 的 prompt/parser 要到后续两天继续落地，但至少从今天开始，version pony 的任务线已经不是靠脚本里写死路径和默认值维持，而是开始进入“配置可切换、任务可区分、输入可追踪”的正式工程状态。

今天最关键的不是写了几个新文件，而是我已经用现有 Beauty processed data 把第一轮多任务输入真正生成出来。按照主入口设计，项目会从 `data/processed/amazon_beauty/valid.jsonl` 和 `test.jsonl` 中稳定产出 `ranking_valid.jsonl`、`ranking_test.jsonl`、`pairwise_valid.jsonl`、`pairwise_test.jsonl`，并且额外在 `outputs/summary/week5_day1_multitask_data_summary.csv` 里沉淀 split 级统计，包括 ranking 样本数、pairwise 样本数、平均候选数、平均 pair 数、pairwise 来源类型分布、正样本覆盖率和 pairwise 来源覆盖率。这个 summary 的意义不只是报数，它是我今天验收“多任务数据层是否真的成立”的第一张硬证据表。如果后面发现某个 split 的平均候选数失控、pair 数不合理、正样本覆盖率不是 1.0，问题就会立刻暴露在数据层，而不是等到后面 prompt 或 eval 才被动发现。

从 version pony 的 week5 总目标回看，今天真正完成的是第一步但也是最关键的一步：我已经把 Part5 里“pointwise 作为观察诊断层、pairwise 作为偏好机制层、candidate ranking 作为主决策层”的三层任务框架，第一次落到了统一的数据生成接口上。现在这三层任务不再停留在项目总览和每周计划的文字描述里，而是开始拥有同源、可复现、可聚合的输入文件体系。接下来 day2 的重点就会非常清晰，不再是思考“数据从哪里来”，而是把 `prompt_builder.py`、`parser.py` 和新 prompt 模板接上去，让 candidate ranking 和 pairwise preference 真正进入可被 LLM 消费、可被结构化解析的推理阶段。当前没有需要你立即拍板的阻塞点；我这边先按 `positive_vs_negative` 作为 pairwise 第一版主线推进，这是最稳也最符合 week5 day1 边界的选择。后面如果你想优先把 `head_vs_tail` 作为第二批 pairwise 结构，而不是 `hard_negative_vs_negative`，那会影响 week6 的机制分析重心，到时我会明确和你对齐，不会糊弄过去。
