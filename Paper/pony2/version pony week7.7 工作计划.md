# version pony week7.7 工作计划

week7.7 到当前这一步，我必须把这一阶段的定位重新收紧，而且要比最初写计划时更贴着真实结果。它已经不再只是一个“Beauty 上把 SRPD-v1 / v2 / v3 先各跑一遍”的训练型 framework 定型周，也不再适合继续沿用“默认 v3 会成为最终赢家，然后很快转 outer compare”的乐观写法。经过我在服务器上完成的真实 Beauty 单域 train/evaluate/compare 闭环，当前事实已经很清楚：`SRPD-v1` 证明了 structured-risk teacher distillation 这条 trainable 路线是成立的，`SRPD-v2` 证明了 uncertainty-aware weighting 在 trainable 设定中带来了真实增益，而 `SRPD-v3` 虽然在工程上跑通、在机制上仍然成立，但在当前 Beauty、Qwen3、300-sample server run 这一轮真实结果里并没有超过 `v2`。因此，week7.7 现在的核心任务，不再是预设某个版本已经封王，而是把当前已经出现的真实事实进一步放大、压实、验证，直到能够真正回答“我现在准备带去 outer compare 与后续 week8 的 trainable main line，到底是不是站得住”的问题。

这意味着，week7.7 的后半段必须改成一个更符合顶会节奏的形态。第一，我不能在 `v2 / v3` 还没有在更有说服力的样本规模上再次验证之前，就仓促打开 outer compare。第二，我也不能在 trainable winner 还只是 300-sample Beauty 结论时，就把整件事推入 week8 的四域统一矩阵。第三，我不能因为一开始的理论偏好更喜欢 `v3`，就在真实结果显示 `v2` 更强时还坚持把 `v3` 写成最终版本。week7.7 现在真正的价值，是把“DeepSeek compact evidence 已经给出的 method-family 收敛判断”与“Qwen3 local server 已经跑出的 trainable family 排序”对齐起来，然后通过更大样本的 Beauty 扩展，把这个排序从 demo 级判断推进到可防守的、接近封王级别的内部结论。也就是说，week7.7 的后半段不是发散，而是收敛；不是赶紧去打外部 baseline，而是先把内部王者用更强证据钉死。

基于当前真实结果，我要在计划开头明确写出一个新的方法边界判断。到目前为止，当前最可信的 candidate-ranking trainable line 不是 `SRPD-v3`，而是 `SRPD-v2_uncertainty_weighted_sft`。这一判断并不意味着项目偏离了 uncertainty 主线，反而恰恰说明 uncertainty 仍然是当前最有效、最稳的 trainable 改进来源。`SRPD-v1` 学的是 `structured risk teacher`，本质上已经继承了 uncertainty-aware decision family 的核心；`SRPD-v2` 把 uncertainty 进一步显式转化为 sample weighting，因此它仍然是标准的 uncertainty-aware trainable method；`SRPD-v3` 则尝试在 uncertainty 主线之上再叠加 pairwise mechanism enhancement。当前 `v3` 没赢，不是因为项目偏离了 uncertainty，而是因为 pairwise preference 这层增强还没有在当前训练规模与注入方式下转化成额外优势。由此，week7.7 的方法主线现在应该明确写成：当前 trainable main line 暂时由 `SRPD-v2` 承担，`SRPD-v3` 保留为 mechanism-enhanced retained variant，不删除、不否定，但不再预设它已经是 final winner。整个 week7.7 后半段要服务于一件事：在更大、更可信的 Beauty 训练规模上，确认 `SRPD-v2` 是否真的能成为 Beauty trainable line 的内部王者，并判断 `SRPD-v3` 是否还有翻盘空间。

因此，我对 week7.7 的总目标要重新收束成这样：在不推翻 week5、week6、week7、week7.5、week7.6 已有成果的前提下，围绕 Beauty 主域、Qwen3-8B 本地服务器 backbone、DeepSeek official API teacher evidence、structured risk strongest hand-crafted baseline 和当前 trainable family 的真实结果，先把 Beauty 的 DeepSeek compact teacher 从 100 样本扩展到 1000 样本，再把 Beauty 的 trainable line 从当前 300-sample demo / 80-train 微调输入推进到 1000 级别的正式训练验证规模，最终用更强的 Beauty 单域证据确认当前 trainable 王者是否真的是 `SRPD-v2`，以及它是否已经足够接近或超过 current best hand-crafted baseline，从而决定 outer compare 是否具备打开资格。week7.7 完成之后，我必须至少拿到四个硬结果。第一，Beauty 域的 DeepSeek official API compact evidence 已经不再只是 100-sample 最小证据，而是扩展成 1000-sample teacher 级证据，足以支撑更像正式微调任务的数据构造。第二，Beauty 上的 trainable line 不再停留在 80-train/20-valid 级别，而是推进到足以支持正式内部 winner 判断的 1000 级别训练规模。第三，`SRPD-v1 / v2 / v3` 在更高样本规模下的相对顺序已被重新确认，我可以更加自信地判断当前 trainable winner 是谁。第四，只有当上述三点成立，而且 winner 与 strongest hand-crafted baseline 的差距已经被重新测清之后，我才会决定是否在 Beauty 单域上打开 outer compare，而不是先入为主地直接进入外层对照。

在这一阶段，我还必须继续坚持 DeepSeek official API evidence 和 Qwen3 local server trainable execution 的双层分工，而且要写得更明确。DeepSeek 这一层的作用不是替代服务器训练，也不是和 Qwen3 local 的直接结果混成同一条表；它承担的是更强、更密集的 compact teacher 证据层。也就是说，把 DeepSeek Beauty 从 100 扩到 1000，并不是为了让官方 API 重新成为主实验执行后端，而是为了给 trainable line 提供更像真正 teacher corpus 的 structured-risk teacher 和 uncertainty-aware teacher 依据。Qwen3 local server 这一层则仍然承担 trainable main line 的真实训练与评估，它负责回答的是：当 teacher evidence 更强、训练样本更大时，当前 trainable family 的内部 winner 是否稳定，是否足够强，是否已经配得上去打 outer baseline。这两层必须继续分开：DeepSeek 1000 是 teacher-strengthening，不是主实验 back-end 回退；Qwen3 local 1000 是 trainable winner confirmation，不是重新回到 API 主线。

现在我必须把 week7.7 的日程重写成更贴着真实进展的五天结构，而且重点从 day4 开始转向“封王级 Beauty 扩展”，而不是早早进入 outer compare。week7.7 第一天到第三天的结果已经是真实发生过的：day1 完成了 `SRPD-v1` 在 Beauty 上的 `teacher-data -> startup_check -> train -> eval -> compare`，证明 structured-risk teacher SFT trainable path 成立；day2 通过修正 `left padding / dtype` 等执行层兼容问题，把 `SRPD-v1` 的 eval 从伪劣化状态纠偏回来，同时完成了 `SRPD-v2` 的训练与评估，并确认 uncertainty-weighted training 确实比 `v1` 更强；day3 则完成了 `SRPD-v3` 的训练与评估，并据此确认在当前 300-sample server run 下 `v3` 没有超过 `v2`。这三天现在不能再被写成“待做计划”，而要被视为已完成的前半段事实。week7.7 计划现在真正要安排清楚的，是后半段，也就是从“已有真实 SRPD family 排序”如何推进到“Beauty trainable 王者被更高强度证据确认”的路线。

week7.7 第四天，我要正式做的是“Beauty teacher 扩展到 1000 与 trainable 王者二次确认准备日”。这一天的第一件事，不是直接再跑 LoRA，而是先把 DeepSeek Beauty 的 structured-risk teacher 从 100 样本扩到 1000 样本。这里我要非常明确写清边界：这不是为了回到官方 API 做大规模主实验，而是为了让当前 trainable framework 的 teacher 不再只基于 100 条 compact evidence，而是基于更接近正式蒸馏任务的数据。具体工程上，我要新增或完善 Beauty 1000-sample 对应的数据构建与汇总配置，让 `main_rank.py`、`main_rank_rerank.py`、`main_build_srpd_rank_data.py` 能够在 DeepSeek API 1000-sample setting 下稳定生成新的 `structured-risk teacher` 结果与对应 summary。这个阶段至少要产出的文件包括新的 Beauty 1000-sample 结果目录、对应的 structured-risk rerank 表、以及新的 SRPD-v1/v2/v3 teacher-data 构建 summary。与此同时，day4 还要重新审视当前 trainable line 的样本入口：之前 `v1/v2/v3` 的 LoRA 训练实际只用了 `80 train + 20 valid`，这对于“跑通框架”是足够的，对“确认王者”是不够的。因此 day4 的第二件事，就是正式把 Beauty trainable line 的目标训练规模写死到 1000 级别，并对应调整 LoRA config 里的 `max_train_samples`、`max_valid_samples`、train/eval 状态 summary 和 batch/registry 入口。也就是说，day4 的验收标准不是“已经把 1000 全跑完”，而是 teacher 1000 扩展与 trainable 1000 扩展的入口、配置、输出路径、状态表头和比较口径全部已经定型，不再停留在 300-sample demo 心态。

week7.7 第五天，我要完成的是“Beauty 1000-sample trainable 王者正式确认日”。这一天的核心，不是再开四域，也不是先做 outer compare，而是拿已经扩到 1000 的 Beauty teacher 与 1000 级别的 trainable 输入，把 `SRPD-v2` 作为当前主线、`SRPD-v3` 作为 retained mechanism-enhanced variant，再做一轮正式的 train/eval/compare。这里我会优先押 `SRPD-v2`，因为今天的真实 300-sample 结果已经明确它当前是三条 trainable line 中最强的；`SRPD-v3` 仍然保留，但它在 day5 的角色已经从“默认 final winner”退回到“必须证明自己值得翻盘”的挑战者。也就是说，day5 最应该出现的结果，不是“v3 理所当然更强”，而是“在 1000-sample Beauty trainable line 下，当前 winner 是否仍然是 v2，如果是，它距离 structured risk 还差多少；如果不是，究竟是什么因素让排序发生了变化”。day5 的产物必须至少包括：1000-sample Beauty trainable compare 表、重新更新的 SRPD framework compare markdown、以及一份明确回答“当前 Beauty trainable 王者是谁、是否具备 outer compare 入场资格”的结论性总结。day5 的验收标准也必须写得非常硬：如果 1000-sample 下 `SRPD-v2` 仍然是 trainable winner，且与 strongest hand-crafted baseline 的差距缩小到可以被 defend，那么 outer compare 才具备打开资格；如果 1000-sample 下 trainable line 仍然明显打不过 structured risk，那么 week7.7 的后续结论就应该是“当前 trainable 路线还需增强，暂缓 outer compare，继续优化 trainable uncertainty line”，而不是自欺欺人地向前推进。

我还要把这一步为什么是“称王级别”的安排写清楚。真正称王，不是因为把某个版本叫成 final，而是因为它经得起更强样本规模和更真实 teacher 证据的二次验证。当前 300-sample server run 已经足以证明三条 trainable line 的方向与排序，但它不够作为最终封王依据；DeepSeek 100-sample compact evidence 足以做四域 inner compare 和 teacher-data alignment，但它不够作为 Beauty trainable 王者的最终支撑。因此，把 teacher 先扩到 1000，再把 Beauty trainable line 扩到 1000，本质上是在做“王者确认”而不是“无意义堆样本”。如果在这一轮更强证据下，`SRPD-v2` 仍然是 trainable family 的最优版本，而且它开始逼近甚至超过 structured risk，那么它才真正具备 outer compare 的资格；如果在这一轮里 `v3` 反而翻盘，那么我也必须诚实接受新的排序，而不是抱着今天这轮 300-sample 的结果不放。称王级别的安排，归根结底就是用更强证据重检当前排序，而不是靠偏好封王。

因此，这版更新后的 week7.7 结论应该写成这样：week7.7 的前半段已经真实完成了 Beauty 上 `SRPD-v1 / v2 / v3` 的服务器闭环，并第一次得到 trainable family 的真实排序，当前 300-sample setting 下 `SRPD-v2` 是暂时最强版本；week7.7 的后半段不再急着进入 outer compare 或 week8，而是要先把 DeepSeek Beauty teacher 从 100 扩到 1000，把 Beauty trainable line 从 demo 级规模扩到 1000 级别，再用这一轮更强证据正式确认当前 trainable 王者是否站得住。只有当这个问题被更强样本、更强 teacher 和更强训练结果回答之后，outer compare 才值得打开，week8 的四域统一矩阵才不会建立在一个尚未真正封王的方法对象之上。换句话说，week7.7 现在最顶会、最正确、也最不糊弄的推进方式，不是立刻对外宣战，而是先在 Beauty 上把王者用 1000 级别证据钉死。

## April 22 更新版：从四域 SRPD 收口转入 NH baseline 顶会级比较协议

到 April 22 这一步，week7.7 的真实进展已经超过了最初“Beauty 1000 级别确认”的单域设想，因此后半段计划必须再次按真实结果重排。现在我已经不再处在“还没确认 trainable family 是否能站住”的阶段，而是已经拿到了服务器真实输出并生成了四域正式矩阵：`Paper/pony2/results/week7_7_four_domain_srpd_final_matrix.csv` 和 `Paper/pony2/results/week7_7_four_domain_srpd_final_matrix.md` 已经把 Beauty full973、Books small500、Electronics small500、Movies small500 的 `direct / structured risk / SRPD-v2 / SRPD-v4 / SRPD-v5` 结果统一放进同一个 evidence table。与此同时，`Paper/pony2/April 22th week7.7 4th day 工作报告.md` 已经把这组结果的论文叙事初步收口：Beauty 和 Books 中 `SRPD-v2` 是 positive-domain winner，Electronics 中 `SRPD-v5` 是 direct-anchor repair winner，Movies 中 direct/structured-risk 仍然最强而 `SRPD-v4` 是 SRPD family 内部最佳修复。这意味着后续不应该再被写成“继续做 toy 版本、继续发明 v6/v7”，而应该进入真正顶会论文会被追问的阶段：我们如何和 NH baseline 里的强论文公平比较，如何证明我们的表不是只在自己定义的内部任务上好看，如何避免被审稿人用“different datasets / different candidate spaces / different evaluation protocols” 一句话击穿。

因此，从现在起，week7.7 的第 4 天到第 7 天要被重新定义为“外部比较边界固化与论文级实验协议收口”。这里我必须先立下一个原则：我们不是怕和 SOTA 比，也不是为了保住 SRPD 的结果而回避 NH baseline；相反，我们要做的是把比较边界做硬，把哪些论文能进入同协议主表、哪些只能做 same-schema proxy、哪些只能放进 related work 讲清楚。顶会级比较不是把所有 paper 的 reported number 复制到一张表里凑热闹，而是保证每一个进入主表的 baseline 都满足同一组基本约束：同一数据 split、同一候选集合、同一 top-k 评价指标、同一任务输出契约、同一 small-model 或 teacher-output schema。如果某篇论文原本做的是 full-item ranking、生成式 item ID、graph-enhanced decoding、agent shield、multimodal personalization 或 long-tail sequential backbone enhancement，而我们当前任务是 fixed candidate ranking with uncertainty-aware distillation，那么它们不能被粗暴地放进同一个 leaderboard；它们必须先被映射到可执行 proxy，或者被放入 conceptual/reference comparison。这不是退缩，而是严格实验设计。

week7.7 第四天的正式任务，应该命名为 **NH baseline -> 我们实验协议 mapping 日**。这一天我不再跑新模型，而是系统读取和整理 `Paper/BASELINE/NH/` 下的论文池，把每篇论文放进一个明确的 protocol mapping 表。这个表建议新建为 `Paper/pony2/results/week7_7_nh_baseline_protocol_mapping.csv`，并同时生成可读版本 `Paper/pony2/results/week7_7_nh_baseline_protocol_mapping.md`，列至少包括 `paper_file`、`paper_title`、`venue_year`、`core_problem`、`original_task`、`original_data_protocol`、`candidate_space`、`model_or_backend`、`main_signal`、`comparison_layer`、`same_schema_feasible`、`proxy_design`、`required_inputs`、`expected_outputs`、`metrics_to_report`、`risk_if_compared_directly`、`final_role_in_our_paper`。这一张表的贡献非常关键，因为它会把 NH baseline 从一堆“看起来都很强的论文”变成一组有边界、有优先级、有可执行策略的对照对象。比如 `Decoding Matters / D^3`、`SPRec` 这类方法可以被归入 decoding debias、homogeneity/fairness、preference regularization 方向，适合做 same-schema rerank 或 decoding-bias proxy，并重点观察 NDCG/MRR 与 coverage/head exposure/longtail coverage 的联合变化；`SLMRec`、`LLM-SRec`、`PAD`、`LLM-ESR` 这类方法可以归入 LLM-to-small 或 semantic-enhanced sequential recommendation 方向，它们和我们最接近的是“用大模型或语义信号帮助小模型”，但它们原始 pipeline 未必是 fixed candidate ranking，因此更适合被写成 distillation/semantic-enhancement proxy 或 related baseline；`OpenP5`、`CoVE`、`AGRec`、`ELMRec` 更偏 item grounding、graph reasoning、indexing 或 platform-level LLM4Rec，不适合直接拿 reported number 和 SRPD 四域表硬比；`Aligning LLMs for Controllable Recommendations`、`iAgent`、`UniMP` 则更偏 controllability、agent paradigm、多模态个性化，适合成为 related work 或 conceptual comparison，而不是主结果表 baseline。第四天的验收标准不是“读完所有论文就算了”，而是这张 mapping 表必须能回答每篇论文为什么进主表、为什么只做 proxy、或者为什么只写 related work。

第四天还要同步更新论文叙事文件，而不只是生成表。具体来说，`Paper/pony2/April 22th week7.7 4th day 工作报告.md` 已经写出第一版 baseline 比较原则，但 week7.7 工作安排里也要把它固化为执行计划，并且后续论文草稿应该引用这套 protocol。这里我会把 `Paper/pony2/results/week7_7_four_domain_srpd_final_matrix.md` 作为当前 internal same-schema evidence 的主输入，把 `Paper/BASELINE/NH/` 作为 external baseline pool，把新建的 `week7_7_nh_baseline_protocol_mapping.*` 作为桥。这样到第四天结束时，我不会再用模糊语言说“后面和 baseline 比”，而是能明确说：主表比较哪些方法，proxy 表比较哪些机制，related work 讨论哪些不能公平复现的强论文。这一步的贡献在于把 outer compare 从情绪压力变成工程协议，也就是把“我担心怎么和他们比”转化成“我知道每一类 baseline 应该站在哪一层”。

week7.7 第五天的正式任务，应该命名为 **same-schema proxy baseline 设计与最小实现日**。这里的“最小”不是 toy，而是最小可辩护实现：只实现那些能够在我们四域 candidate-ranking schema 下真实运行、真实输出 ranking、真实进入同一指标计算的外部机制代理。第五天首先要从第四天 mapping 表里选出 2 到 4 个最值得落地的 proxy，而不是贪多。优先级应放在和我们论文主问题最接近的三类：第一类是 debias/homogeneity/decoding 类 proxy，对应 D^3 或 SPRec 的精神，它们可以在 fixed candidate set 上设计为 popularity suppression、head-exposure penalty、homogeneity penalty 或 self-negative rerank proxy，但必须用真实 candidate features、真实 popularity/head-tail 划分、真实四域 ranking predictions 计算，不允许手写 toy score；第二类是 semantic/distillation 类 proxy，对应 SLMRec、LLM-SRec、PAD、LLM-ESR 的精神，它们可以作为 “semantic/teacher signal without uncertainty gate” 或 “direct teacher SFT without uncertainty weighting” 的对照，用来证明 SRPD 的贡献不是普通蒸馏，而是 uncertainty-aware conditional distillation；第三类是 long-tail/fairness 类 proxy，对应 LLM-ESR 或 SPRec 的长尾收益问题，它不只看 NDCG/MRR，还必须同时看 `coverage@10`、`head_exposure_ratio@10`、`longtail_coverage@10`、`out_of_candidate_rate`。第五天要修改或新增的文件应围绕执行闭环，而不是论文空话：如果已有 baseline runner 能扩展，就优先扩展 `main_rank_rerank.py` 的 rerank variant 或新增一个独立的 `main_literature_proxy_rank.py`；如果需要配置层，则新增 `configs/exp/{domain}_literature_proxy_*.yaml`；如果需要统一汇总，则新增 `main_compare_literature_proxies.py` 或扩展现有 compare script，让 proxy 输出进入 `outputs/summary/week7_7_literature_proxy_compare.csv` 和 `outputs/summary/week7_7_literature_proxy_compare.md`。第五天的验收标准是至少有一到两个 proxy baseline 在一个域上真实跑通，并能通过同一 evaluator 生成 ranking metrics，而不是停留在方法描述。

第五天在论文贡献上的意义，是为 SRPD 找到真正有压力的同协议对照，而不是只和 direct/structured risk 这种内部 baseline 比。当前四域矩阵已经证明 SRPD family 内部有清楚结构，但顶会审稿人会继续问：相比已有 LLM4Rec 的 debias、distillation、semantic enhancement、long-tail repair，你的 uncertainty-aware conditional distillation 到底新在哪里？第五天的 proxy baseline 就是为这个问题准备证据。比如如果一个 non-uncertainty distillation proxy 在 Beauty/Books 不能达到 SRPD-v2，那么我就能说明不是“任何 teacher SFT 都可以赢”，而是 uncertainty-weighted teacher construction 带来了额外价值；如果一个 head-exposure penalty proxy 改善 coverage 却损害 NDCG/MRR，而 SRPD-v5 在 Electronics 上同时改善或至少保持 utility，那么我就能说明 direct-anchor uncertainty repair 不是简单的 popularity debias；如果 semantic proxy 在 Movies 仍然不稳定，那么 Movies 的失败就更像域属性与 teacher gap 问题，而不是 SRPD 单点缺陷。第五天不是为了让 proxy 一定输，而是为了让比较变得严肃。

week7.7 第六天的正式任务，应该命名为 **四域 proxy compare 执行与反质疑审计日**。这一天要把第五天跑通的 proxy 从单域扩展到四域，至少覆盖 Beauty、Books、Electronics、Movies 中已经完成 direct/structured/SRPD 主矩阵的那些数据入口。执行顺序必须遵循“不覆盖、不污染、不混表”的原则：proxy 的输出目录不能覆盖已有 `outputs/{domain}_qwen3_*` 或 SRPD 主结果，建议统一命名为 `outputs/{domain}_literature_proxy_{method}_small500` 或 `outputs/{domain}_literature_proxy_{method}_full973`，所有 summary 输出进入 `outputs/summary/week7_7_literature_proxy_compare.*`，最终再由 `Paper/pony2/results/week7_7_literature_proxy_compare.md` 固化为论文辅助结果。第六天还要做一件非常重要的事：写一份反质疑审计说明，建议文件名为 `Paper/pony2/results/week7_7_outer_compare_fairness_audit.md`，逐条回答为什么某些 NH 方法没有进入主表，为什么某些方法只做 proxy，为什么 reported SOTA number 不直接横向比较，以及我们的 fixed candidate ranking schema 与它们原始 setting 的差别在哪里。这个文件会在写论文时非常有用，因为它可以提前把审稿人最容易质疑的问题拆掉。

第六天的结果分析不能只看哪个 proxy 分数高，而要围绕“我们的贡献是否仍然成立”来写。四域主矩阵已经告诉我，SRPD-v2 在 positive domains 强，v4/v5 在 failed-domain repair 中有作用，Movies 仍然暴露出 teacher gap 边界。proxy compare 要进一步验证这些解释是否稳。如果 debias proxy 在 Electronics 也能提高 exposure 但不如 v5 的 NDCG/MRR，那么 v5 的 direct-anchor repair 叙事更强；如果 semantic/distillation proxy 在 Beauty/Books 有提升但低于 v2，那么 v2 的 uncertainty weighting 贡献更强；如果所有 proxy 在 Movies 都很难超过 direct/structured risk，那么 Movies 就可以被写成一个非常重要的 negative domain evidence，说明 fixed candidate ranking 中有些域的 direct baseline 已经非常强，teacher correction 的可蒸馏性不足，继续盲目蒸馏反而会损害 student。顶会论文不怕有失败域，怕的是不能解释失败；第六天的重点就是让失败域变成边界贡献，而不是结果缺陷。

week7.7 第七天的正式任务，应该命名为 **论文级主表、辅助表与故事线冻结日**。这一天不再新增实验，除非第六天发现某个 proxy 缺关键文件必须补跑。第七天的核心是把所有结果整理成论文可以使用的结构：第一张主表是 `four-domain same-schema main table`，来源是 `Paper/pony2/results/week7_7_four_domain_srpd_final_matrix.csv/md`，它负责证明 SRPD family 在四域下的内部结构与最终 winner；第二张辅助表是 `NH literature proxy compare table`，来源是 `Paper/pony2/results/week7_7_literature_proxy_compare.csv/md`，它负责证明我们不是只和自己的 baseline 比，而是在可公平执行的外部机制代理下仍然有独特价值；第三张方法定位表是 `NH protocol mapping table`，来源是 `Paper/pony2/results/week7_7_nh_baseline_protocol_mapping.csv/md`，它负责证明我们认真看了外部论文，并且没有把不可比的 reported numbers 硬塞进主表；第四份审计文件是 `week7_7_outer_compare_fairness_audit.md`，它负责保护论文免受 protocol mismatch 的质疑。第七天还要把 `Paper/pony2/April 22th week7.7 4th day 工作报告.md` 或后续日报继续补全，把“我们的方法是什么、和 NH baseline 的差别是什么、为什么不是 toy、为什么是顶会级比较协议”写成可以迁移到论文 introduction/experiment section 的密集段落。

第七天的最终论文叙事应该冻结成一个清楚版本：我们的方法不是一个普通 LoRA baseline，也不是随便拿 uncertainty 当辅助指标，而是把 uncertainty 从 pointwise diagnosis 推进到 structured-risk ranking，再推进到 conditional preference distillation。我们的主发现也不是“某一个版本永远赢”，而是更科学的条件式结论：当 teacher correction 有足够可蒸馏性时，uncertainty-weighted SRPD-v2 可以显著提升 small-model ranking；当 teacher-direct gap 小或 teacher correction 不稳定时，gap-gated v4 和 direct-anchor v5 可以缓解过度蒸馏，但并不保证所有域反超 direct；当 direct/structured risk 已经非常强时，Movies 这种失败域反而揭示了 uncertainty-aware distillation 的适用边界。和 NH baseline 相比，我们的差异不在于又做了一个更大的 LLM recommender，而在于系统研究了“哪些 teacher correction 值得被小模型学习”，并把这个判断落实为可训练的数据构造、样本权重、target fallback 和跨域验证。这个贡献比 toy 更硬，因为它有真实四域结果、真实服务器训练、真实 GitHub 固化文件、真实 NH baseline protocol mapping，也有正例、负例和修复机制三类证据。

因此，week7.7 第四到第七天的最终安排应该被压成一句执行原则：**第四天定比较协议，第五天做可辩护 proxy，第六天跑四域并做公平性审计，第七天冻结论文主表和故事线。** 这四天不做 toy，不做“为了表格好看”的伪 baseline，也不继续无边界发明新版本。所有新增工作都必须服务于顶会论文最核心的两个问题：第一，我们的 uncertainty-aware conditional distillation 是否在 same-schema 主结果中成立；第二，它和 NH baseline 中已有的 debias、distillation、semantic enhancement、long-tail repair、controllability、agent/multimodal 方法相比，究竟站在什么不同问题上，哪些可以公平比较，哪些不能直接比较。只要这两件事被写清楚、跑扎实，week7.7 就不再只是一个实验周，而会成为整篇论文从内部方法探索走向外部比较防守的关键转折周。
