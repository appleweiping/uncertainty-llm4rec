**version pony 项目总览**

version pony 不是我对 version 1.2 的简单续写，也不是把 Part5 和 Part6 机械拼接成更长的阶段计划，而是我对整个项目研究对象、方法形态、工程底座与验证标准的一次系统升级。version 1.2 已经帮我建立了一条非常清楚、也非常有价值的主线：我先在 pointwise recommendation 上确认 LLM 自报 confidence 不是随机噪声，而是一个“有信息量但不可靠”的信号；再通过 calibration 把这个原始分数修正成更接近真实正确率的 calibrated confidence；再把 uncertainty 作为风险项引入 reranking，使它从一个分析变量变成真正参与推荐决策的变量；随后我又把这条链扩展到多模型、多 estimator 与 robustness setting，逐步把项目从 prototype 推向系统研究。这个起点非常重要，因为它证明了我研究的不是空洞概念，而是一个已经被真实实验支撑、并且可以持续向上扩展的问题链条。

但我也很清楚，顶会级工作不会因为“主线闭环了”就自然成立。真正决定说服力的，不是我有没有把 diagnosis、calibration、reranking 这条旧链跑通，而是这条链能不能迁移到更主流、更接近真实推荐场景的任务 formulation，能不能和文献中的强 baseline 在统一 setting 下做公平对照，能不能摆脱 API 速率与成本的外部瓶颈，能不能在多任务、多模型、多域、多噪声条件下维持结论稳定。也正因为如此，version pony 的核心目标，不再只是证明“uncertainty 有用”，而是要把整个项目推进成一个完整、具体、可执行、可写论文、可复现、可持续扩展的顶会导向研究框架。在这个框架里，我不仅要说明 uncertainty 在推荐里存在，还要说明它如何被测量、如何被校准、如何跨决策粒度迁移、如何进入偏好比较与候选排序决策，以及它是否能在强 baseline、本地可控模型、四个真实数据域和噪声扰动环境下持续体现独立贡献。

从研究问题上说，version pony 回答的已经不是单一的“LLM 在推荐里会不会 overconfident”。我真正要回答的是这样一个更完整的问题：在现代推荐场景中，不确定性究竟如何被定义、提取、比较、校准并转化为可扩展的决策能力。这个问题比旧版更强，因为它不再停留在 observation-level 的现象分析，也不再局限于单一任务、单一公式或单一点式方法，而是进入了“uncertainty 作为统一决策变量”的系统研究。换句话说，我不再只研究一个模型会不会把错误答案说得很自信，我要研究的是：当任务从 pointwise 判断扩展到 pairwise preference，再扩展到 candidate ranking 时，不确定性是否仍然是一个可测量、可迁移、可利用的核心变量；当实验从单个数据集扩展到多个 Amazon 子域，从单一 API 扩展到五个官方 API 与本地 Hugging Face / 服务器后端时，这个变量是否仍然成立；当环境从 clean 变成 noisy 时，这个变量是否依然具有决策与稳健性价值。

在任务结构上，version pony 不再把 pointwise yes/no recommendation 视为最终任务主体，而是把它明确降级为 pointwise diagnostic layer，也就是整个项目的观测与诊断层。这一层我不会删，也不能删，因为它承担的是整个项目的证据起点。它的职责不是再去竞争最终推荐任务，而是持续负责 uncertainty elicitation、confidence-correctness correlation、high-confidence error、reliability diagram、ECE、Brier、AUROC、popularity bias 与 exposure shift 等诊断分析。前面阶段我已经证明，LLM 的 confidence 的确包含判别信息，但它的概率刻度并不可信，存在明显 miscalibration，而且在不同 popularity group 与 exposure 分布中呈现结构性差异。因此，它不能直接作为下游决策依据，必须经过校准与进一步建模。这个结论本身，就是我整篇论文的研究动机，也是后面 pairwise 和 ranking 层为什么需要 uncertainty 的理论前提。

在此之上，我把第二层定义为 pairwise preference layer。这个层级的研究意义，不是取代 pointwise，也不是与 candidate ranking 并列竞争主任务，而是承担“机制层”和“中介层”的角色。在 pairwise task 中，我让模型面对的不再是“是否推荐 item A”这种绝对判断，而是“对于同一个用户，是更偏好 A 还是 B，以及我对这个偏好判断有多确定”。这个 setting 比 pointwise 更接近偏好建模本身，也比 candidate ranking 更容易控制和解释，所以它特别适合用来研究 uncertainty 如何作用于局部偏好比较。更重要的是，pairwise 并不是一个孤立层，因为这些局部偏好关系可以继续被聚合成 candidate ranking 的底层分数基础。也就是说，在 version pony 中，pairwise preference 不只是一个辅助实验，而是 candidate ranking 的细粒度机制来源，它帮助我回答：uncertainty 到底是在绝对判断里有价值，还是在局部比较和偏好冲突里更有价值。

第三层则是 candidate ranking layer，它是 version pony 的新的主任务层。真实推荐系统最终面对的，不是单个 yes/no 判断，而是在一个候选集中做列表级决策、排序和资源分配。旧版的不确定性方法已经在 reranking 上说明了一件很重要的事：当我把 uncertainty 作为 penalty 引入排序时，它会改变 top-k 的结构分布，影响曝光、长尾覆盖与风险控制。version pony 把这个逻辑继续往前推进。我不再满足于在 pointwise 分数之后附加一个 rerank，而是把 candidate ranking 本身提升为主决策载体。这样一来，我研究的问题就从“这个 item 是否推荐且我有多确定”，升级成“当多个候选同时竞争时，uncertainty 是否能帮助我识别高分但高风险的候选，并改善最终 top-k 的可靠性、分布结构和资源分配方式”。这才是更符合当前推荐系统与 LLM4Rec 研究趋势的任务升级。

因此，version pony 的整体结构不是单线式的，而是三层递进结构。pointwise 是观测与诊断层，负责 uncertainty 的原始提取与质量分析；pairwise 是局部偏好与机制分析层，负责研究 uncertainty 在成对比较中的作用，并为 ranking 提供细粒度偏好信号；candidate ranking 是主决策层，负责把 uncertainty 真正落实到列表级推荐中。这样一来，我的项目就从旧版“diagnosis → calibration → reranking”的单任务链条，升级成了“observation → preference → ranking”的多决策任务统一框架，而 calibration、多源 uncertainty、decision、robustness 则贯穿其中，成为连接三层的共同方法主线。

从方法主线看，version pony 会保留并强化我旧版中最重要的三件事。第一是 calibration。原始 verbalized confidence 仍然必须严格在 valid 上拟合，在 test 上应用，避免信息泄漏，并统一得到 calibrated confidence 与 uncertainty \= 1 − calibrated\_confidence。只有 uncertainty 经过校准，它才有资格进入后续决策层。第二是多源 uncertainty estimator。我旧版已经提出 verbalized raw、verbalized calibrated、self-consistency 与 fused uncertainty 的比较空间，version pony 不会推翻这一点，而是会把这些 estimator 从 pointwise setting 迁移到 pairwise 与 candidate ranking setting 中，让它们在不同决策粒度下都进入统一 compare。第三是 uncertainty-aware decision。旧版的 rerank 公式可以写成 final\_score \= calibrated\_confidence − λ · uncertainty，version pony 会保留这个精神，但把它推广成跨任务统一机制：在 pointwise 层，它是 calibrated decision signal；在 pairwise 层，它是局部偏好比较的 reliability weight；在 ranking 层，它是列表级风险惩罚项、排序约束，甚至是 selective recommendation 的依据。换句话说，我不再把 uncertainty 看成某个脚本里的后处理分数，而是把它重构成一个能够跨 observation、preference 和 ranking 三个层级被建模、比较和利用的统一变量。

但 version pony 真正与旧版拉开差距的，不仅是多任务，还在于 baseline 体系必须全面升级。旧版更多是 estimator baseline 和方法内部 baseline，例如 raw confidence、calibrated confidence、self-consistency、fused，以及 baseline ranking 与 uncertainty-aware reranking 之间的比较。这些比较对于证明方法链条成立是必要的，但对于顶会并不够，因为审稿人很容易把它理解成“我在自己定义的世界里和自己过去版本比”。因此，在 version pony 中，我会把 baseline 明确分成三个层级。第一层是 uncertainty-source baseline，也就是在同任务、同模型、同数据条件下比较 raw verbalized、calibrated、self-consistency、fused 等 estimator 的校准质量、判别能力和决策效果。第二层是 decision-formulation baseline，也就是在 pointwise、pairwise、candidate ranking 这些任务中分别引入“不使用 uncertainty 的同任务方法”做正面对照，例如 candidate ranking 的 direct ranking output、pairwise 的 plain win-count 或 Borda-style aggregation、pointwise 的 calibrated rerank baseline。第三层是 literature-aligned baseline，也就是将文献中与当前任务设置最接近的方法纳入统一候选集、统一划分、统一指标下的公平比较。这一层我不追求 baseline 数量，而是追求 baseline 的对齐质量与审稿防御力。

结合我对你给的压缩包中论文与代码线索的逐个阅读，我对 literature-aligned baseline 的理解已经不能再停留在泛泛口号上，而必须具体落地。对 generative / ranking platform 层，我从 OpenP5 得到的最重要启发，不是简单“再引一个模型”，而是顶会级 LLM4Rec 工程应该围绕统一数据处理、统一 task schema、统一评测口径与可扩展执行后端来组织，而不能把推理、评估、对照散成零碎脚本。对 embedding / transition / local model 层，我从 LLM2Rec、DEALRec、TransRec 这些工作里看到的不是“名字上像推荐”，而是一个很清晰的工程方向：LLM 不一定只能作为黑盒 API 的文本生成器，它可以被当作 embedding model、transition model、preference model，甚至是 fine-tuning base model 来使用。这直接支撑了我把本地 Hugging Face 小模型与服务器推理升级为主实验后端的路线，而不是继续依赖 API。对评测与强对照方向，我从 EvaLLMasRecommender 看到的是更强的信号：ranking 与 reranking 完全可以放进同一评测体系中，多个 LLM 与传统方法也可以在统一指标下做直接比较，同时还可以讨论 candidate position bias、hallucination、grounding 等 LLM 特有维度。这说明我把 candidate ranking 作为主任务、并搭配统一评测矩阵，是完全对路的。对偏好推理与多任务对齐方向，我从 Exp3rt 这类工作里看到的是，pairwise / preference / reasoning 并不只是解释性附属品，而是可以真正进入候选 reranking 和偏好建模主线的任务形式。这说明我的多任务 baseline 体系不是人为发散，而是和近年的论文组织方式一致。再加上你给的材料里那篇 uncertainty quantification and decomposition for LLM-based recommendation 的工作，也进一步表明，把 uncertainty 作为独立研究对象本身就是合理的，而且它不应只停留在 calibration 指标，而应该进入 decomposition、decision 与 robustness。也正因为我有了这些文献和仓库的具体参照，version pony 的 baseline 升级才不是凭空硬写，而是建立在真实论文生态之上的。

进一步说，我在 version pony 中提出“把本地 Hugging Face 小模型与服务器推理升级为主实验后端，使实验规模、吞吐和批量调度能力真正摆脱 API 速率与成本瓶颈”，这句话必须具体到实现路径，而不能停留在愿景层。这个实现路径，我会明确落成“统一后端接口 \+ 本地主实验后端 \+ API 观测型后端 \+ 批处理调度层”四件套。第一，我要把 `src/llm/base.py` 升级成统一后端抽象层，让所有任务都只依赖一个稳定接口，例如 `generate()` 或 `batch_generate()`，并统一返回 `raw_text`、`model_name`、`latency`、`parse_success`、必要时还有 `usage` 等字段。这样，pointwise、pairwise、candidate ranking 三条任务线就不再关心底层到底是 API 还是本地模型。第二，我要把 `src/llm/api_backend.py` 和 `src/llm/local_hf_backend.py` 分开，并明确重置二者角色。API backend 不再承担主实验，而是降级为“观测型、分析型、对照型后端”，主要用于五个官方 API——Doubao、Qwen、GLM、DeepSeek、Kimi——的 cross-model 观察、小规模对照、case study、自一致性抽样和特定现象验证。真正承担主实验吞吐的，将是本地 Hugging Face / 服务器后端。第三，我要把 `local_hf_backend.py` 做成真正可用于主实验的执行后端，而不只是两行 `AutoTokenizer.from_pretrained` 和 `AutoModelForCausalLM.from_pretrained`。这个文件必须明确解决四类问题。第一类是本地 cache 与路径解析，我要优先命中本地 HF cache 或服务器模型目录，而不是运行时强依赖网络。第二类是设备与显存管理，我要支持 `device_map`、`dtype`、`max_new_tokens`、`batch_size`、必要时的量化或半精度，并对无 accelerate 环境给出兼容逻辑。第三类是批量推理，我要支持把 pointwise、pairwise、candidate ranking 三类 prompt 统一打包进 batch，而不是像 API 一样逐条调用。第四类是容错与恢复，我要记录失败样本、空输出、解析失败、OOM、backend error，并支持续跑和跳过已完成样本。只有做到这四点，本地小模型后端才不是“本地也能跑一下”，而是真正成为主实验主线。第四，我还必须增加一层 batch 调度体系，也就是 `main_batch_run.py`、`src/utils/exp_launcher.py`、`src/utils/exp_registry.py` 与 `configs/batch/*.yaml`。这层的意义，是把 task、model、domain、uncertainty\_source、lambda、noise\_setting 这些维度从手工命令提升成结构化实验队列，让 week7 / week8 那种多维统一矩阵真正可执行。Part6 说的“实验能力由本地算力和批处理设计决定”，落到工程上就是这一整套，而不是别的。

因此，在后端层的具体设计上，我会在 `configs/model/` 下显式区分 `api_doubao.yaml`、`api_qwen.yaml`、`api_glm.yaml`、`api_deepseek.yaml`、`api_kimi.yaml`，以及 `qwen_local_7b.yaml`、`qwen_local_8b.yaml` 或其他本地主模型配置。每个配置文件至少包括 `backend_type`、`model_name_or_path`、`tokenizer_name_or_path`、`device`、`dtype`、`batch_size`、`max_new_tokens`、`temperature`、`top_p`、`api_key_env`、`base_url` 等字段。`main_infer.py`、`main_rank.py`、`main_pairwise.py` 这些入口不允许再默认某一个 API，而必须完全通过 `model_config` 或 `backend_name` 切换后端。对于 pointwise、pairwise、candidate ranking 三条任务线，我必须保证输出 schema 在 API 与本地后端上一致，这样后面的 parser、metrics、aggregation 才能完全复用。然后，我还要在 `configs/batch/` 下定义如 `week7_local_scale.yaml`、`week8_domain_expand.yaml`、`week8_model_compare.yaml`、`week8_noisy_run.yaml` 这样的批处理清单，把要跑的实验显式列成队列。这样，本地小模型与服务器推理的真正价值，就不只是“便宜一些”，而是我终于可以跑中等规模甚至更大规模的 ranking / pairwise / pointwise 统一实验，可以扫描多个 λ、多个 uncertainty source、多个 noise level，而这些事在 API-only 阶段因为速率与成本问题根本不现实。

从数据与实验场景上看，version pony 的主实验不再围绕 MovieLens，而是明确建立在四个已经 processed 好的 Amazon 子域之上，也就是 Amazon Beauty、Amazon Electronics、Amazon Books 和 Amazon Movies。这四个数据集已经不是“准备中”，而是我的正式主实验域。Beauty 可以代表商品推荐主域，Movies 代表内容型推荐域，Books 代表语义驱动更强的文本域，Electronics 代表标题较短、语义更弱、噪声更高的功能性商品域。这个四域设计非常关键，因为它让我不再只能在一个 benchmark 上描述现象，而可以明确回答 reviewer 最关心的问题之一：这个现象是不是跨 domain 稳定成立。也正因为四个域都已经 processed，我在 version pony 中的重点就不再是“能不能接数据”，而是如何把 pointwise、pairwise、ranking、baseline、本地后端和多维统一验证真正落在这四个域上。

从模型维度上看，version pony 的官方 API 组已经明确，不再模糊。我的官方 API 一共有五个，分别是 Doubao、Qwen、GLM、DeepSeek 和 Kimi。它们会继续保留，但角色会被重置。我不会再让它们承担大规模主实验，而是把它们视作商业黑盒模型的观察窗口和 cross-model 对照组，用来支持少量高价值对比、case study、自一致性样本采样、特定任务下的现象验证，以及回答“这个现象是不是单一模型特例”这类问题。与此同时，我会把本地 Hugging Face 小模型与服务器推理升级为主实验主线，让实验的主吞吐不再受官方 API 速率与成本限制。这样一来，version pony 在 model dimension 上就不是“只有一个模型能跑”，而是形成了“五个官方 API 作为观测型组 \+ 本地可控模型作为主实验组”的双层结构。这个结构非常符合顶会口径，因为它既保留了商业 LLM 行为观察，也保留了大规模、可控、可复现实验能力。

从工程结构上讲，version pony 的代码组织必须比旧版更彻底地正交化。数据维度、任务维度、模型维度、方法维度、结果维度必须分离。数据维度由 `configs/data/` 与 `src/data/` 管；任务维度由 `configs/task/`、`main_infer.py`、`main_rank.py`、`main_pairwise.py`、`main_build_multitask_samples.py` 管；模型维度由 `configs/model/` 和 `src/llm/*backend.py` 管；方法维度由 `src/uncertainty/`、`src/methods/` 管；结果维度由 `outputs/{exp_name}/predictions/`、`calibrated/`、`reranked/`、`figures/`、`tables/` 和 `outputs/summary/` 管。这样，一个实验如果命名为 `beauty_qwen_local_rank_sc_noisy`，它的数据来源、任务类型、模型后端、uncertainty source 和 noise setting 都能从名字与配置中自己闭合，后面的聚合脚本也能自动解析，不需要我再手工记忆。这个正交化目录结构，本质上就是旧版 `exp_name` 隔离机制在顶会验证要求下的继续升级。

具体到文件职责，version pony 的完整文件体系我会这样理解和推进。`main_preprocess.py` 继续负责把四个 Amazon 域接入统一 processed schema。`main_build_samples.py` 继续生成 pointwise train/valid/test。新增的 `main_build_multitask_samples.py` 则从 processed data 和 split 信息中派生 `ranking_valid.jsonl`、`ranking_test.jsonl`、`pairwise_valid.jsonl`、`pairwise_test.jsonl` 等新任务输入。`src/data/candidate_ranking_builder.py` 负责把同一用户的一组候选组织成 ranking sample，字段至少包括 `user_id`、`history`、`candidate_item_ids`、`candidate_texts`、`candidate_popularity_groups`、`positive_item_id` 等；`src/data/pairwise_preference_builder.py` 负责把 ranking 候选进一步转成 pairwise sample，字段至少包括 `item_a_id`、`item_b_id`、`preferred_item`、`pair_type` 等。`src/llm/prompt_builder.py` 必须扩展 `build_candidate_ranking_prompt()` 与 `build_pairwise_preference_prompt()`，`src/llm/parser.py` 必须扩展 `parse_candidate_ranking_response()` 与 `parse_pairwise_preference_response()`。`main_rank.py` 和 `main_pairwise.py` 则分别完成 ranking / pairwise 推理落盘。`src/eval/ranking_task_metrics.py` 负责 HR@K、NDCG@K、MRR、coverage、head\_exposure、long-tail coverage；`src/eval/preference_metrics.py` 负责 pairwise accuracy、一致性、confidence bin 统计。`main_eval_rank.py` 和 `main_eval_pairwise.py` 则把它们落成稳定结果表。方法层新增 `src/methods/uncertainty_ranker.py` 和 `main_rank_rerank.py`，把 pointwise 或其他 estimator 的 uncertainty 迁移进 candidate ranking；新增 `src/methods/uncertainty_pairwise_aggregator.py` 和 `main_pairwise_rank.py`，把 pairwise 层的局部偏好与 uncertainty 聚合成列表排序；再通过 `main_uncertainty_compare_multitask.py`、`main_baseline_multitask.py`、`main_compare_baselines.py` 完成多任务 compare 与 baseline matrix。后端与调度层则由 `src/llm/base.py`、`src/llm/api_backend.py`、`src/llm/local_hf_backend.py`、`main_batch_run.py`、`src/utils/exp_launcher.py`、`src/utils/exp_registry.py` 组成。所有这些文件加起来，才构成一个真正完整的 pony，而不是仅仅多几个脚本。

version pony 的输出体系也必须比旧版更具体、也更论文导向。我不能只说“会生成结果”，而要明确每层产出什么，以及这些产出服务哪一类论文 claim。数据层要产出 `interactions.csv`、`items.csv`、`users.csv`、`popularity_stats.csv` 以及 `train.jsonl`、`valid.jsonl`、`test.jsonl`，再进一步产出 `ranking_valid.jsonl`、`ranking_test.jsonl`、`pairwise_valid.jsonl`、`pairwise_test.jsonl`。推理层要产出 `outputs/{exp_name}/predictions/test_raw.jsonl`、`rank_predictions.jsonl`、`pairwise_predictions.jsonl`。校准层要产出 `outputs/{exp_name}/calibrated/test_calibrated.jsonl` 与 `calibration_comparison.csv`。pointwise 评估层要产出 `diagnostic_metrics.csv`、reliability diagram、confidence histogram、popularity / exposure 图表。ranking 评估层要产出 `ranking_metrics.csv`，至少包含 HR@10、NDCG@10、MRR、coverage、head\_exposure\_ratio、longtail\_coverage。pairwise 评估层要产出 `pairwise_metrics.csv` 和 `preference_confidence_bins.csv`。多任务 compare 层要产出 `week5_multitask_summary.csv`、`part5_multitask_final_results.csv`、`part5_multitask_final_summary.md`。baseline 体系要产出 `week7_day4_baseline_matrix.csv`。多维统一验证层要产出 `part6_main_results.csv`、`part6_ablation_results.csv`、`part6_final_summary.md` 与 `docs/part6_experiment_claims.md`。也就是说，version pony 不是“跑出很多 csv”，而是每一层都要有明确的论文服务对象：哪些表支撑 diagnosis，哪些表支撑 ranking，哪些表支撑 baseline，哪些表支撑 robustness，最终对应哪一条核心 claim。

从实验维度上，version pony 最终要形成的不是一串零散实验，而是一个五维实验矩阵。第一维是 task，包括 pointwise、pairwise、candidate ranking，其中 pointwise 负责诊断与校准，pairwise 负责偏好比较与机制分析，candidate ranking 负责主决策验证。第二维是 model，包括五个官方 API 组成的观测型商业模型组、本地 Hugging Face 小模型组成的主实验组，以及后续可能接入的 LoRA-adapted model，用来区分黑盒行为与可控模型行为。第三维是 uncertainty，包括 raw verbalized、calibrated、self-consistency、fused 以及后续可能扩展的 proxy-based uncertainty。第四维是 domain，包括 Beauty、Electronics、Books、Movies 四个 Amazon 子域。第五维是 robustness，包括 clean 和不同类型、不同强度的 noisy setting。只有当这五个维度在统一 schema 与统一 summary 下真正串起来，我的论文才不再是“几个 case 说得过去”，而是能系统回答：我的方法是否跨任务成立，是否跨模型成立，是否跨域成立，是否在噪声 setting 下仍成立。

如果进一步从时间主线去看，version pony 可以分为两大执行阶段。第一阶段是 pony week5 和 week6，对应 Part5 的正式落地。week5 的核心是架构层跃迁，把项目从单一 pointwise pipeline 升级为三层多任务框架，并让三层都进入可配置、可推理、可评估、可聚合状态。week6 的核心是方法层跃迁，把 uncertainty 从 pointwise diagnosis 真正迁移到 pairwise 与 candidate ranking 决策中，补齐 decision baseline 与多任务 compare，最终生成可以直接写论文 method / experiment 的 Part5 总表和总结。第二阶段是 pony week7 和 week8，对应 Part6 的正式落地。week7 的核心是基础设施升级，即完成强 baseline 接入、本地后端主线建立、批量调度体系成型和中等规模主域实验。week8 的核心是统一验证升级，即把 task、model、uncertainty、domain、robustness 五个维度真正串起来，并输出主表、消融表和 claim 对齐文档。这样划分的意义在于，Part5 解决“多任务框架是不是成立”，Part6 解决“这个框架是否经得起顶会审稿标准的验证”。

因此，version pony 的贡献可以压缩成五类，而且每一类都比旧版更完整。第一类是任务贡献。我首次将 pointwise、pairwise、candidate ranking 放在统一 uncertainty-aware recommendation 框架下，使 uncertainty 不再绑定在单一任务上，而成为跨决策粒度的统一变量。第二类是方法贡献。我不仅延续了 diagnosis、calibration、multi-source uncertainty、reranking 这条方法链，还把这条链迁移到 preference 与 ranking setting，使 uncertainty-aware decision 从单公式后处理扩展为跨任务机制。第三类是 baseline 贡献。我把 baseline 从方法内部 compare 升级成 uncertainty-source、decision-formulation 和 literature-aligned 三层结构，使项目具备与现有工作做公平对照的能力。第四类是系统贡献。我把执行框架从 API 驱动的小样本实验推进到本地 Hugging Face / 服务器推理驱动的批量可扩展平台，使实验规模、吞吐和调度真正可控。第五类是验证贡献。我把实验从单域、单模型、单设置扩展到四个 Amazon 域、五个官方 API、多任务、多模型、多噪声的统一矩阵，并通过自动聚合脚本沉淀成论文级证据系统。

最后，如果我把 version pony 压缩成一句真正准确、也最适合放在项目首页或 proposal 开头的话，它已经不再是一个围绕 pointwise yes/no confidence 的 prototype，而是一个以 pointwise diagnosis 为观测底座、以 pairwise preference 为机制层、以 candidate ranking 为主决策层、以分层 baseline 与本地可扩展后端为验证支撑、以四个 Amazon 子域、五个官方 API 与多任务多模型多噪声统一矩阵为证据系统的顶会导向不确定性感知推荐框架。我真正要证明的，不只是“uncertainty 有用”，而是“uncertainty 可以在现代 LLM recommendation 中作为一个可测量、可校准、可迁移、可比较、可扩展的核心决策变量长期成立”。这才是 version pony 相比旧版最本质、也最值得写进论文的升级。

**补充说明：ranking 决策层的方法升级线**

在当前 version pony 的多任务统一框架中，candidate ranking 已经被明确提升为主决策层，因此它的方法升级不能长期停留在“从 pointwise uncertainty 线性迁移一个 penalty”这一层。现阶段已经拿到的真实结果表明：简单 linear transfer 足以证明 uncertainty 可以跨任务进入 ranking 决策，但它并没有自动带来收益；coverage-aware 更像是在 partial coverage 条件下保持方法诚实性的稳健约束；top-k gated 则说明前排位置确实是更值得干预的区域，但单独使用这一机制仍不足以建立 NDCG 与 MRR 收益。因此，项目总览中需要额外补进一条更加明确的方法升级线：在 ranking 主决策层，将 uncertainty 从“统一减法惩罚项”升级为“结构化复合风险信号”。

这条方法升级线包含三个正式 variant。第一，`nonlinear_structured_risk_rerank` 作为主 variant，用位置门控、阈值截断、非线性风险放大、coverage 感知、前排强化与高相关低风险保护项共同构成新的 ranking rerank 公式。它的研究定位非常明确，不是追求复杂外观，而是针对已经暴露出的失败模式逐层建模：前排更敏感、低于阈值的不确定性不该被乱罚、缺失真实 uncertainty 覆盖时不应和真实覆盖项同等对待、高相关低风险候选不应被过度误伤。第二，`local_margin_swap_rerank` 作为局部修正 variant，不负责全局风险塑形，而专门解决“局部相关性差距很小但高风险项挡在前面”的排序失败模式。第三，`structured_risk_plus_local_margin_swap_rerank` 作为组合 variant，用来明确检验全局结构化风险与局部排序修正在同一 ranking setting 中叠加后是否真的产生额外收益。

这意味着 version pony 的 ranking 方法空间已经从“baseline ranking + simple rerank”扩展成一个可消融、可比较、可回退的 family，而不是若干零散实验脚本。工程上，所有 variant 继续共享 `src/methods/uncertainty_ranker.py` 与 `main_rank_rerank.py` 的统一入口；实验上，它们继续共享同一套 `rerank_results.csv`、同一套 summary compare schema；论文叙事上，它们共同服务于一个更清楚的结论：uncertainty 进入 ranking 决策是成立的，但收益建立并不自动发生，因此必须通过结构化风险塑形与局部顺序修正来检验 ranking 层最适合的 uncertainty 使用方式。也正因为这条方法线已经被正式纳入，version pony 的 candidate ranking 层不再只是多任务框架里的一个 listwise baseline，而开始成为真正的方法创新承载层。

**补充说明：week6 之后的主线收敛与 week7-week9 的承接关系**

在当前版本里，我还需要把一条非常关键的时间逻辑写得更明确：week6 的意义已经不再只是“把 uncertainty 接进 ranking 和 pairwise”，而是**在统一 setting、统一指标、统一总表下，并行比较 ranking decision family 与 pairwise-to-rank mechanism family，并最终收敛出 current best family**。这一步决定了后续 week7、week8、week9 不应再围绕一个模糊的方法集合展开，而应围绕一个已经被比较过、已经被收紧过的主线 family 展开。也就是说，version pony 在时间结构上不是“先做很多方法，再随便往后推”，而是“先在 week6 完成有组织的并行 compare 与 family 收敛，再在 week7-week9 持续检验和加固这一主线”。

因此，我会把后续三周的承接关系理解得更精确。week7 的价值，是把 week6 收敛出的 current best family 推进到本地主实验后端、batch 调度和 literature-aligned baseline 的统一对照体系中，并把 pairwise-to-rank 保留为机制层验证线，而不是继续开放式方法搜索。week8 的价值，是把 current best family 正式写进五维统一验证矩阵，也就是让主表明确区分 direct ranking baseline、current best ranking family、pairwise mechanism line 和必要的 exploratory evidence，而不是把所有方法混成一个含糊的“uncertainty-aware rerank”。week9 的价值，则是在这个已经收敛的主线之上做最有价值的增强，也就是让 LoRA-adapted model 与 reviewer-gap 补实验优先挂到 current best family 上，验证它在参数适配和审稿追问之下是否仍然成立。

这里还要特别补一句，避免后续误读：current best family 的收敛，并不等于把复杂融合版提前删除。像 `structured_risk_plus_local_margin_swap_rerank` 这样的完全融合版，我会继续把它保留在统一 compare、统一总表和后续消融矩阵里，只是现阶段不让它占用默认主实验资源。它的身份更像一个带触发条件的保留 family：当前不作为主线，但如果后面样本规模扩大、uncertainty coverage 提升、候选复杂度变高，或者 cross-domain / robustness 结果显示它更稳，那么它可以被重新提升为重点比较对象。也正因为我把这条规则写死了，version pony 的方法收敛才不是粗暴删枝，而是有组织地保留潜在高价值复杂版本。

这样补完之后，整个 version pony 的项目逻辑会更像一篇真正成熟的顶会论文而不是开发日志。我的主线不再是“我尝试了很多 uncertainty 公式”，而是“我先在 week6 用统一 compare 收敛出 candidate ranking 主任务上的 current best family，再用 week7 的强执行环境、week8 的统一大矩阵、week9 的适配与补洞，把这条主线逐层压实”。对应到论文叙事上，candidate ranking 主决策层的价值也因此更清楚了：它不是一个任意选定的 listwise 任务，而是整个项目在系统比较之后收敛出的主要承载层；pairwise-to-rank 则不是失败备选，而是机制层证据与 preference signal 的来源。这样写，version pony 的整体方法贡献、系统贡献和验证贡献会更集中，也更符合“先比较、后收敛、再扩展验证”的顶会研究节奏。
