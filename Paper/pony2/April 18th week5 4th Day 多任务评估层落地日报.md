今天的核心任务，不再是让多任务系统“能跑”，而是让它真正进入“能算”的研究状态。按照 version pony 的 Part5 结构，week5 第四天必须把前三天已经搭好的多任务数据层、prompt/parser 层和 inference 层，进一步推进到正式评估层。原因非常直接：如果 pointwise、candidate ranking、pairwise preference 只是各自生成了一批 prediction jsonl，却没有稳定、统一、可复现的评估口径，那么这个多任务框架仍然只是工程扩展，而不是研究系统。今天的重点因此不是加更多模型，也不是继续扩展 prompt，而是把三种任务都接入各自正确的指标空间，并且让输出稳定落盘为表格，能够直接服务后续 week6 的 compare。

我今天首先确认了 pointwise 评估主链的边界，不去推翻旧版 main_eval.py，而是将它明确保留为 pointwise diagnostic layer 的正式评估入口。这个决策和项目总览完全一致，因为 pointwise 在 version pony 里的职责并不是消失，而是被重新定位为 uncertainty diagnosis 的基础观测层。也就是说，它要继续承担 calibration quality、confidence discrimination 和 exposure-related bias 的分析任务，而不是被 ranking 或 pairwise 取代。基于这个判断，我今天没有去改写 main_eval.py 的核心逻辑，而是直接复用它来产出 pointwise 的 diagnostic_metrics.csv、confidence_bins_accuracy.csv、reliability_bins.csv、popularity_group_stats.csv、high_confidence_exposure.csv，以及 reliability diagram、confidence histogram、popularity confidence boxplot 等图。这一步的意义在于，旧主链没有被破坏，而是被保留为新多任务框架中的诊断层支柱。

在此基础上，我正式新增了 ranking 与 pairwise 的评估模块，而不是继续把所有逻辑堆在一个文件里。candidate ranking 侧，我新增了 src/eval/ranking_task_metrics.py，把 ranking prediction 重新整理成可评估的 event-level 表达，再实现了 HR@10、NDCG@10、MRR、coverage、head_exposure_ratio、longtail_coverage 等指标。这里的关键，不只是把公式写出来，而是先定义清楚什么叫“ranking 任务的正确评价对象”。因为当前 ranking 输出不是旧 pointwise 那种一条样本只对应一个候选，而是一条样本内部已经包含整组候选及其预测排序，所以我今天先在 ranking_task_metrics.py 里做了结构转换，把 positive_item_id 在 pred_ranked_item_ids 中的位置提取出来，形成 positive_rank，再基于它计算 HR、NDCG 和 MRR；与此同时，把 topk_item_ids 与 candidate_popularity_groups 对齐起来，用来计算 exposure 和 coverage。这一步如果偷懒，直接拿旧 ranking_metrics.py 去硬套，结果会在数据粒度上完全错位，后面所有数字都会失真。

pairwise 侧，我新增了 src/eval/preference_metrics.py，用于把 pairwise_predictions.jsonl 转成明确的偏好评价表，并实现 pairwise_accuracy、confidence-binned accuracy、preference_consistency、strict_preference_consistency、ambiguous_preference_rate 等指标。这里我今天特别小心的地方，是没有把 pairwise 简单当成一个普通二分类任务，而是保留了它“局部偏好判断”的结构语义。pairwise_accuracy 仍然以 preferred_item_pred 是否等于 preferred_item_true 为基础，但 preference_consistency 则按 source_event_id 聚合，观察同一正例事件下多组 pairwise 判断是否稳定指向锚定正例；strict_preference_consistency 则更严格地要求一个事件下的所有 pair 都和锚定正例保持一致。这个定义并不意味着 pairwise 研究已经完成，而是意味着我们从今天开始不再只看单对正确率，而开始尝试从局部偏好集合的一致性去看模型稳定性，这对后续 uncertainty 是否能迁移到 pairwise 层是有价值的。

为了让这些指标不只是模块函数，而是真正进入项目主链，我今天同步新增了 main_eval_rank.py 和 main_eval_pairwise.py 两个正式入口。main_eval_rank.py 的职责是读取 outputs/{exp_name}/predictions/rank_predictions.jsonl，将 ranking 结果转成稳定表，并落盘 ranking_metrics.csv、ranking_exposure_distribution.csv、ranking_eval_records.csv；main_eval_pairwise.py 则读取 outputs/{exp_name}/predictions/pairwise_predictions.jsonl，落盘 pairwise_metrics.csv、preference_confidence_bins.csv、pairwise_eval_records.csv。这样做以后，ranking 与 pairwise 的评估终于和 pointwise 一样，进入了可命令行调用、可重复运行、可直接生成表格的正式工程状态，而不是 notebook 临时算一遍就结束。今天这一步非常关键，因为没有正式入口，后面 week6 做多任务 compare 时就会不断回退到“手工读取 jsonl 再临时算”的低可靠流程，这和顶会级别项目管理是冲突的。

在完成代码搭建之后，我直接对第三天已经生成的三类 Beauty Qwen 预测结果跑了一轮真实评估。这里我依然保持和前一天相同的诚实边界：当前评估对象是第三天那批 30 样本级别的 inference 闭环结果，它的任务是检验评估链是否稳定、指标是否有定义、表头是否统一，而不是把这组小样本数字包装成最终性能结论。pointwise 评估顺利生成了 diagnostics 与图表，其中 diagnostic_metrics.csv 显示当前 30 条样本上的 accuracy 为 0.5，avg_confidence 为 0.936，Brier 为 0.42344，ECE 为 0.436，AUROC 为 0.7778。这组结果本身不漂亮，但恰恰说明 pointwise 作为诊断层仍然有价值：它把当前 verbalized confidence 的过度自信问题明确暴露出来，而不是被掩盖在后续 ranking 指标里。

ranking 侧，当前 30 条样本的 ranking_metrics.csv 显示：HR@10 为 1.0，NDCG@10 为 0.6613，MRR 为 0.5522，coverage@10 为 1.0，head_exposure_ratio@10 为 0.15，longtail_coverage@10 为 1.0，parse_success_rate 为 1.0，avg_latency 为 6.0547 秒。这里我不会误导性地强调 HR@10=1.0，因为当前每个 ranking 样本只有 6 个候选，且 top-k 设为 10，HR@10 在这种设定下天然宽松，真正更有信息量的是 NDCG 与 MRR，它们显示模型虽然基本能把正例留在榜单中，但排序位置并不稳定，这与 pointwise 那边暴露出的 confidence 失真一起，构成了很清楚的研究信号：当前系统已经进入 listwise 决策空间，但远未达到高质量排序阶段。

pairwise 侧，当前 30 条样本的 pairwise_metrics.csv 显示：pairwise_accuracy 为 0.6，parse_success_rate 为 1.0，avg_confidence 为 0.8713，avg_latency 为 2.7445 秒，ambiguous_preference_rate 为 0.0，preference_consistency 为 0.6，strict_preference_consistency 为 0.3333。同时，preference_confidence_bins.csv 已经稳定生成，可以看到高置信区间并不自动对应高正确率，例如 0.9–1.0 置信区间的 accuracy 只有 0.5。这一点非常重要，因为它说明 pairwise 层现在虽然已经有了正式推理和正式评估，但在“偏好信心是否可信”这个问题上并不成熟，甚至存在明显的高置信错误。这不是坏消息，而是项目真正进入研究空间的标志：如果 pairwise 一上来就全对，项目反而失去了不确定性研究的必要性。现在的结果更真实，也更适合作为 week6 继续推进 uncertainty compare 的起点。

为了让三类任务在后续 compare 时共享同一张高层摘要表，我今天还新增了 main_summarize_multitask_eval.py，并生成 outputs/summary/week5_day4_multitask_eval_summary.csv。这个表并不是为了强行把三类任务揉成一种指标，而是为了把它们放进统一 schema 下，提前为后面的 compare 保留结构接口。因此，它统一提供了 task、sample_count、HR@10、NDCG@10、MRR、pairwise_accuracy、ECE、Brier、coverage、head_exposure、longtail_coverage、parse_success_rate、avg_latency、source_path 等字段。对于不适用的列保留空值，而不是为了“表好看”去乱填代理指标。这个选择非常重要，因为多任务框架真正需要的是统一结构，而不是伪统一语义。今天这张表已经证明，pointwise、candidate ranking、pairwise preference 三种任务粒度第一次被放进了同一张可扩展的研究摘要空间中。

我今天也同步更新了 README，但依然遵守科研项目主页的节制原则。README 现在已经在高层结构上明确说明：推理层按任务拆分之后，评估层也开始按任务拆分；main_eval.py 保留 pointwise 的诊断职能，main_eval_rank.py 负责 ranking 的推荐指标空间，main_eval_pairwise.py 负责 pairwise 的局部偏好一致性空间。我没有把 README 写成“今天新增了什么、明天干什么”的项目日记，也没有把具体指标定义和全部产物路径铺得太满，而是只在高层说明“多任务评估层已经成形”。这样既符合科研仓库应有的专业表达，也避免把工程细节和实验套路过度公开。

从今天的工作结果来看，第四天的验收标准已经成立。第一，三种任务的评价都能稳定跑通，不是只存在模块函数，而是能形成正式入口和标准表文件；第二，结果里没有出现大面积 NaN 或 schema 断裂，pointwise、ranking、pairwise 三种任务的关键指标都成功落盘；第三，统一摘要表已经具备 week6 compare 所需的基础列头，说明多任务框架不再只是“输出文件多了”，而是真正进入了“有可比较指标”的研究状态。更重要的是，三类任务的结果已经清楚体现出它们各自的定位：pointwise 主要暴露校准与高置信错误问题，ranking 主要暴露排序位置与曝光结构问题，pairwise 则开始暴露局部偏好与信心一致性问题。

当然，今天我也必须把仍然存在的边界与风险写清楚。第一，当前 day4 的评估仍然建立在 day3 的 30 样本级闭环上，因此它证明的是评估体系已经建立，不是最终实验性能已经稳定。第二，ranking 的 HR@10 因为候选数较小而偏宽松，后面必须更重视 NDCG、MRR 甚至更难的 ranking 设定，而不能被单一 hit 指标误导。第三，pairwise 当前只有 0.6 accuracy 且高置信区间仍有明显错误，这说明后续如果要讨论 pairwise uncertainty transfer，就必须非常谨慎，不能预设“pairwise 一定比 pointwise 更稳定”。第四，统一摘要表虽然已经成形，但它现在仍是第一轮最小闭环，后面还需要接入更大样本、更稳定 backend，以及 week6 的多任务 compare 逻辑。

因此，我对今天的最终结论是：Part5 所要求的多任务评价体系已经完成第一轮最小闭环。pointwise、pairwise、candidate ranking 三个决策粒度第一次进入统一评估空间；pointwise 保住了诊断层职责，ranking 获得了正式推荐指标空间，pairwise 获得了局部偏好正确率与一致性指标空间；项目正式从“有多任务输出文件”推进到“有多任务可比较指标”的研究阶段。也就是说，到 week5 第四天结束，version pony 已经不再是一个概念性的多任务框架，而是一套能够同时生成、推理、评估三种决策粒度的执行系统，后续可以进入更严肃的 uncertainty compare 与跨任务分析，而不是继续在接口层徘徊。
