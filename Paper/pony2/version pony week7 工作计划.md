**version pony week7 工作计划**

week7 的总目标，是完成 version pony 从“多任务方法框架”到“顶会级验证平台”的第一次正式跃迁。这一周我不会再把主要精力放在任务定义本身，因为 pointwise、pairwise、candidate ranking 三层结构在 week5 和 week6 已经成立；我这一周要解决的，是这个框架能不能支撑更强的对照、更大的吞吐和更稳定的实验组织。具体来说，week7 我必须完成三类核心升级。第一类是后端升级，我要把五个官方 API——Doubao、Qwen、GLM、DeepSeek、Kimi——从主实验后端重新定位为观测型、分析型和对照型后端，把本地 Hugging Face 小模型与服务器推理升级为主实验主线，这样实验规模、吞吐和批量调度才不会继续受 API 速率与成本限制。第二类是 baseline 升级，我要把旧版多任务内部 baseline 与真正 literature-aligned baseline 开始放进同一坐标系，让 reviewer 无法再轻易把项目视作“封闭式内部 compare”。第三类是执行框架升级，我要让实验从“我手工逐个 exp 跑”升级成“由批处理配置驱动、能统一调度、失败可恢复、结果可聚合”的系统平台。到 week7 结束时，我必须拿到的，不只是几个 backend 文件或 batch 脚本，而是一套真正能支撑 week8 多维统一验证的基础设施。这一周如果做不扎实，week8 的多任务、多模型、多域、多噪声矩阵就会变成表面计划，而无法可靠执行。

这一周还有一个必须提前写死的工程原则，就是我对“本地 Hugging Face 小模型与服务器推理”这件事的理解必须具体且不再含糊。这里所谓“本地”并不是指我个人电脑本地，而是指**真正执行训练或推理的那台机器的本地存储与本地运行环境**。也就是说，7B/8B 底座模型不会下载到我自己的笔记本上去折腾，而是下载到服务器上，固定放在服务器的 HF cache 或模型目录中，由服务器承担模型保存、加载、推理与后续可选的 LoRA/QLoRA 训练。本地电脑的职责，是维护代码仓库、修改配置、提交 GitHub；服务器的职责，是保存 base model、运行主实验、保存输出结果、在需要时保存 LoRA adapter。这种分工不是个人习惯，而是目前论文和工程里最稳的做法。Hugging Face 的缓存机制、PEFT/LoRA 的 adapter 设计、以及 vLLM 的服务式推理范式，本质上都默认了这种“代码和实验可分离、模型驻留在执行机器”的工作方式。对应到我的项目里，week7 之后我必须把这种分工彻底写进 backend、config 和目录结构里，以免后面本地代码、服务器模型、实验输出三者缠在一起搞不清楚。

从论文做法上看，week7 的后端升级并不是我自己发明一条奇怪路线，而是顺着近几年本地部署 7B/8B 与 LoRA 适配的主流实验范式往下走。很多近期推荐与 LLM 论文并不会全参重训一个 7B/8B 模型，而是先选一个可本地部署的 instruct 底座模型，冻结大部分参数，然后用 LoRA 或 QLoRA 只训练很少的 adapter 参数。这样做有两个非常现实的意义。第一，它显著降低显存需求，使 7B/8B 量级模型能够在单卡或较小服务器资源上完成任务适配。第二，它让训练产物从“整份底座模型副本”变成了一个小得多的 adapter 目录，从而非常适合研究型项目的管理和复现。因此，我在 week7 的后端设计上会明确遵循这条路线：先把 7B/8B 本地底座作为主实验 backbone 跑通，再决定要不要加 LoRA；如果加 LoRA，也不是重训全模型，而是做可插拔的 adapter；如果后面吞吐继续增大，再考虑切到 vLLM 常驻服务。也就是说，week7 的目标不是一上来把 LoRA 全做完，而是先把“服务器上 base model 常驻、本地代码驱动配置、支持 base-only 与 base+adapter 两种状态、未来可切 vLLM”的工程接口铺好。这样，后续 LoRA-adapted model 才是自然扩展，而不是重新推翻后端。

week7 第一天，我要完成的是**统一后端抽象层、服务器本地主实验后端、以及本地/服务器职责分工的正式建立**。旧版和前两周的实验虽然已经能跑，但核心限制仍然非常明显：pointwise、pairwise、candidate ranking 三条任务线在推理层仍然强依赖外部 API 作为主要执行后端，这导致样本规模受限、实验 latency 高、并发受限，而且重复实验和参数扫描都很难做。Part6 已经非常明确地指出，顶会级实验不能继续把 API 当作主实验执行层，而必须把 API 降级为观测型后端，把本地 Hugging Face 与服务器推理升级为主实验后端。所以第一天我最重要的工作，不是“再加一个模型配置”，而是把整个后端抽象真正做对。具体来说，我会重点改 `src/llm/base.py`、`src/llm/api_backend.py`、`src/llm/local_hf_backend.py`。`base.py` 必须明确统一接口，至少对 pointwise、pairwise、candidate ranking 三条任务线都只暴露 `generate()` 或 `batch_generate()` 这一类稳定入口，并统一返回 `raw_text`、`model_name`、`latency`、`parse_success`、必要时还有 `usage` 等字段。`api_backend.py` 保留，但我要明确重置它的角色：它不再是主实验通道，而是五个官方 API 的少量 cross-model 对照、case study、自一致性抽样和特定现象观察通道。真正的主实验后端，将落到 `local_hf_backend.py`。

而 `local_hf_backend.py` 必须从“本地能加载一下模型”升级成“可承担主实验”的执行后端。我会在这个文件里显式解决四类问题。第一类是服务器本地 cache 和路径命中问题，我必须优先命中服务器的 HF cache 或服务器模型目录，而不是运行时联网拉取；也就是说，`model_name_or_path` 必须既支持 Hugging Face repo 名，也支持服务器上的本地路径，例如 `/root/autodl-tmp/models/...`。第二类是设备与显存问题，我要支持 `device_map`、`dtype`、`batch_size`、`max_new_tokens`，必要时还要兼容半精度或量化，并且对没有 accelerate 的环境做兼容逻辑，避免脚本一跑就因设备映射报错。第三类是批量推理问题，我要让 pointwise、pairwise、candidate ranking 三类 prompt 都能被统一打包进 batch，而不是逐条调用。第四类是容错问题，我要记录失败样本、解析失败、OOM、空输出或 backend error，并为续跑预留机制。为了配合这一层，我还要在 `configs/model/` 下补齐或规范化 `api_doubao.yaml`、`api_qwen.yaml`、`api_glm.yaml`、`api_deepseek.yaml`、`api_kimi.yaml`，以及 `qwen_local_7b.yaml`、`qwen_local_8b.yaml` 之类本地模型配置。每个配置必须包含 `backend_type`、`model_name_or_path`、`tokenizer_name_or_path`、`device`、`dtype`、`batch_size`、`max_new_tokens`、`temperature`、`top_p`、`api_key_env`、`base_url` 等关键字段。同时，为了避免后面搞不清楚，我还要补一份明确的部署说明文档，例如 `docs/week7_local_backend_and_server_workflow.md`，把“模型下载到服务器、代码在本地维护、服务器 pull 代码后运行、后续 LoRA 也只在服务器保存 adapter、本地不保存底座模型”写清楚。第一天我不会直接跑全量实验，而是做三条任务线的最小本地后端验证，确保 `main_infer.py`、`main_rank.py`、`main_pairwise.py` 都已经支持通过 `model_config` 切本地模型。第一天的输出必须有 `outputs/summary/week7_day1_backend_check.csv`，至少记录 `task`、`backend`、`model`、`load_success`、`avg_latency`、`parse_success`，以及一份 `docs/week7_day1_backend_notes.md`，写清当前服务器模型目录、cache 路径、最小加载命令与失败情况。第一天的验收标准是：三条任务线都能在服务器本地后端上跑通最小样本，并且和 API 路径保持一致输出 schema；同时，我已经把“模型放服务器、代码在本地维护、实验在服务器执行”这条工程分工固定下来。第一天的贡献，不是“本地也能跑了”，而是我正式把 version pony 从 API 单后端实验模式切换成“API 观测型后端 \+ 服务器本地主实验后端”的双层执行框架。第一天结束时，我的结论应当是：项目已经完成主实验执行层的第一次重构，后端不再由外部 API 决定，而开始由我自己的模型配置、服务器缓存、设备和批处理能力决定。

week7 第二天，我要推进的是**批处理调度层、实验注册机制，以及服务器运行流程的正式落地**。如果第一天只是让服务器本地后端“能用”，那么第二天要解决的是“它能不能支持真正的大规模、多配置、多任务实验执行”。因为 Part6 要求的不是一个个手工命令，而是一个五维实验矩阵；如果没有统一调度和注册机制，后面 week8 只要一扩维，目录就会乱、失败实验就会难以恢复、结果也很难追踪。因此，这一天我会新增 `main_batch_run.py`，并配套补 `src/utils/exp_launcher.py`、`src/utils/exp_registry.py`，同时在 `configs/batch/` 下开始建立 batch 清单，例如 `week7_local_scale.yaml`。`exp_launcher.py` 的职责，是把单个 experiment spec 转成标准化执行流程，让 pointwise、pairwise、candidate ranking 三条任务线都能通过统一入口被调度，而不是各自写一套脚本壳子。`exp_registry.py` 则负责记录实验状态，至少要跟踪 `exp_name`、`task`、`model`、`status`、`output_dir`、`error_message`、`retry_count` 这些字段，这样后面失败恢复、重试和统一 summary 才真正有依据。

第二天我最强调的不是“自动化看起来高级”，而是这层对顶会实验组织是必要条件。因为从 week7 开始，我已经不再是跑一个 Beauty 上的 pointwise 小实验，而是要考虑三类任务、多个 uncertainty source、多个 λ、多个 domain、多个后端和后面的 noisy 设置。这些东西如果没有 batch 层，所有结果最终都会重新回到人工维护状态。考虑到我现在的真实工作流是“本地改代码，服务器跑实验”，所以第二天我还必须把这条链条写成清楚的服务器执行约定。我会补一个脚本级说明，例如 `scripts/run_sync_and_batch.sh` 或 `docs/week7_server_execution.md`，把本地 push、服务器 pull、检查 cache、启动 batch、查看 registry、续跑失败实验这一套流程明确下来。第二天我会先用小规模的 batch 验证，不追求多，而追求稳定。我至少要让 pointwise、pairwise、candidate ranking 各调起一组本地实验，并且验证失败时不会把整个队列打崩。第二天的核心输出是 `outputs/summary/week7_day2_batch_status.csv`，里面至少要记录每个 `exp_name` 的 `task`、`model`、`status`、`latency`、`output_dir`、`error_message`；同时还要有一份 `docs/week7_day2_batch_workflow.md`，把当前 batch 清单写法、服务器执行入口和失败恢复逻辑写清。第二天的验收标准是：批处理系统能够稳定启动三类任务，不同实验有独立输出目录，失败实验可以被识别和单独重试，而且整套流程已经不再依赖我手工逐个记命令。第二天的贡献，不是“多了一个调度脚本”，而是我正式把实验从手工串命令推进成了结构化实验队列，为 week8 的多维矩阵奠定了真正的工程基础。第二天结束时，我的结论应当是：version pony 已经不再依赖人工记忆和手动逐个跑实验，而是开始具备系统性、大规模、可恢复的执行能力。

week7 第三天，我要进入 **Part6 的 literature-aligned baseline 第一轮接入**，并且把“本地 7B/8B、之后可接 LoRA-adapted model”的路线正式和 baseline 体系对齐。week5 和 week6 里，我已经完成了 uncertainty-source baseline 和 decision-formulation baseline 的雏形，也就是 raw / calibrated / self-consistency / fused 之间的 estimator compare，以及 pointwise、pairwise、candidate ranking 各自的 non-uncertainty 对照版本。但这还不够，因为顶会审稿最容易攻击的一点，就是“你只在自己定义的框架里和自己比较”。所以第三天我要开始接入真正的 literature-aligned baseline，不过我要控制范围，做“最小但严格”的第一版，而不是一口气堆很多方法。我会新增 `src/baselines/literature_rank_baseline.py`、`src/baselines/literature_pairwise_baseline.py` 和统一入口 `main_run_literature_baselines.py`。这两个 baseline 文件的职责，不是把所有文献完整复现，而是在当前的候选集、数据划分、评测口径下，对接最接近 candidate ranking 与 pairwise preference 的文献范式。也就是说，我强调的是对齐质量，不是 baseline 数量。

这里我的依据并不是空想，而是你之前让我解压阅读的那些论文和代码线索。OpenP5 告诉我，generative recommendation 平台应该围绕统一 task schema 和标准评测组织；EvaLLMasRecommender 告诉我，ranking / reranking 可以在同一评测体系下统一对照多个 LLM 与传统方法；Exp3rt 这类工作则说明 pairwise / preference / reasoning 本身就是合理的任务载体，而不是解释性噱头。除此之外，本地 7B/8B 与 LoRA 适配这一条，也不是我自己杜撰出来的工程梦想，而是论文里非常常见的路线：先在服务器上固定保存 7B/8B instruct 底座模型，先跑 base-only inference baseline；如果确认任务上需要适配，再用 PEFT 做 LoRA 或 QLoRA，只训练 adapter，不重训全模型；推理阶段则选择 Transformers 直接加载 `base + adapter`，或者后续切到 vLLM 常驻服务。这意味着，我在第三天接 baseline 时，也必须把 model family 这一层考虑进去。我不会当天就把 LoRA 训练做完，但我会把 literature baseline 的文件接口与模型配置接口做成兼容 `base-only` 和 `base+adapter` 两种状态的样子，例如在 baseline config 里预留 `adapter_path` 字段，并在 `docs/week7_day3_literature_baseline_notes.md` 里明确说明：week7 的 literature baseline 先以 base-only 本地 7B/8B 模型为主，后续如有必要再扩展到 LoRA-adapted variant。第三天我会先在 Beauty 域和一个本地主模型上接入最小严格对齐版 baseline，各做一个 ranking 和一个 pairwise baseline。第三天的核心输出是 `outputs/summary/week7_day3_literature_baseline_summary.csv`，至少列出 `task`、`baseline_name`、`domain`、`model`、`HR@10`、`NDCG@10`、`MRR`、`pairwise_accuracy`；同时补一份 `docs/week7_day3_literature_baseline_notes.md`，写清当前 baseline 的文献来源、为什么这样对齐、以及后续是否扩成 LoRA 版。第三天的验收标准不是这些 baseline 的指标一定多高，而是它们已经在完全相同的候选集、相同划分、相同指标下跑通，并且可以被纳入后续统一 summary；同时，我已经把“base-only 本地小模型 → 可选 LoRA adapter → 后续可选 vLLM”的路线和 baseline 体系对齐起来。第三天的贡献，是把 version pony 从“方法内部自洽”正式推进到“可以与文献范式在同一 setting 下正面对照”。第三天结束时，我的结论应当是：Part6 所要求的 literature-aligned baseline 已经完成第一轮接入，项目开始具备应对“封闭式内部实验”质疑的基础能力，而且本地 7B/8B 与未来 LoRA-adapted model 的接口也已经提前对齐。

week7 第四天，我要做的是 **三层 baseline 体系的统一聚合与正式表达**，并把“官方 API 组、本地主模型组、后续 LoRA-adapted 组”的实验身份关系也写清楚。第三天只是接入了 literature baseline 的第一版，但如果它继续和 week6 里的 uncertainty-source baseline、decision-formulation baseline 分散存在，那仍然无法形成顶会级说服力。因此，这一天我不会再写新方法，而是要把三层 baseline 全部放进同一张矩阵里。我会新增或完善 `src/analysis/aggregate_baseline_results.py`，并配一个入口 `main_compare_baselines.py`。这套分析层必须至少收集三类 baseline。第一类是 uncertainty-source baseline，例如 raw verbalized、calibrated、self-consistency、fused。第二类是 decision-formulation baseline，例如 candidate ranking 的 direct ranking output、pairwise 的 plain aggregation、pointwise 的 calibrated rerank。第三类是 literature-aligned baseline，也就是第三天接入的 rank / pairwise 文献对齐方法。所有这些 baseline，都要在统一 schema 下被表达，而不是分散在不同风格的 csv 里。
