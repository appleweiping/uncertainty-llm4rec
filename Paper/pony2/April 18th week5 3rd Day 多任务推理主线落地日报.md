今天的工作目标，是把 week5 前两天已经搭好的多任务数据层与多任务语言接口层，真正推进到可运行的推理闭环层。到昨天为止，项目虽然已经能构造 ranking / pairwise 样本，也已经能对 candidate ranking 与 pairwise preference 做 prompt 构造和 parser 解析，但这些能力还停留在“接口可用”的阶段，并没有真正进入统一 backend 驱动下的正式推理执行。因此，今天的核心不是再写更多 prompt，也不是先做评估，而是把 pointwise、candidate ranking、pairwise preference 三条任务线都接进实际 inference 主流程，让版本结构从单一 pointwise inference 升级成并行多任务 inference。

我今天首先处理的是入口职责边界。按照项目总览与 week5 第三天计划，pointwise 主链不能被新任务稀释，所以我保留了 main_infer.py 继续专门服务 pointwise diagnostic layer，没有把 ranking 与 pairwise 硬塞回旧入口。与此同时，我新增了 main_rank.py 和 main_pairwise.py，分别承担 candidate ranking 与 pairwise preference 的正式推理入口。这样处理之后，三条任务线在工程语义上已经清晰分层：main_infer.py 负责点式诊断推理，main_rank.py 负责列表决策推理，main_pairwise.py 负责局部偏好判断推理。这一步非常重要，因为如果今天继续用一个总入口兼容所有任务，短期看似省代码，实际上会让后续 week6 的跨任务比较、week7 的 backend 切换、week8 的统一统计全部混乱。

在入口拆分之外，我同步重构了 src/llm/inference.py，使它不再默认 pointwise，而是开始真正支持按 task_type 分发推理逻辑。今天的实现不是简单把几个 if/else 粘起来，而是把共享部分和任务特有部分尽量拆开：共享部分负责调用 backend、记录 latency、保留 raw_response、附加 model_name/provider/usage 等元数据；任务特有部分分别负责 pointwise 的 yes/no 决策解析、candidate ranking 的排序结果解析、pairwise preference 的 A/B 偏好解析。这样做的结果是，三条任务线虽然输出 schema 不同，但都共享同一层 generation 元信息记录逻辑，因此后面要做 summary 聚合或 appendix 故障排查时，不会出现“有的任务有 latency，有的任务没有；有的任务保留 raw_response，有的任务不保留”这种结构性缺口。

今天新增或实质改造的核心文件主要包括 main_rank.py、main_pairwise.py、src/llm/inference.py、src/utils/exp_io.py，以及对 main_infer.py 的整理。src/utils/exp_io.py 的意义在于把实验入口层反复出现的 YAML 读取、JSONL 读写、prompt 模板读取逻辑收束成统一小工具，避免三个入口脚本各自复制一套读取函数，后面改配置接口时反复返工。main_infer.py 经过整理后仍然保持 pointwise-only，但已经不再内联一堆重复的 IO 逻辑。main_rank.py 与 main_pairwise.py 则复用了同样的配置合并方式、输出目录解析方式和随机种子处理方式，使三条推理链在命令行使用体验上也保持一致。

在 prediction schema 方面，今天的原则是宁可多保留关键排障字段，也不能为了“输出简洁”把后续研究分析需要的信息提前抹掉。因此，pointwise 输出除了原有 response_text、response_label、response_confidence、parsed_success 之外，我额外统一补上了 parse_success、latency、model_name、provider、usage 等元信息字段；candidate ranking 输出中固定保留了 user_id、candidate_item_ids、pred_ranked_item_ids、topk_item_ids、raw_response、latency、model_name、parse_success，并进一步记录了 positive_item_id、positive_item_index、candidate_titles、candidate_popularity_groups、contains_out_of_candidate_item、out_of_candidate_item_ids、parse_mode 等信息；pairwise preference 输出则固定保留了 user_id、item_a_id、item_b_id、preferred_item_pred、confidence、raw_response、parse_success，同时也保留了 preferred_item_true、pair_type、reason、ambiguous_preference、parse_mode 等后续分析真正会用到的字段。今天之所以坚持保留 raw_response 和 parse_success，是因为 week6 以后我们做的已经不只是指标汇总，还要能够反查模型失败样例、比较任务间出错机制，并且要为 appendix 准备可解释的失败证据。

完成代码接线之后，我没有停留在“理论上能跑”，而是直接用 Beauty 当前的 week5 输入跑了一轮真实推理闭环。这里我有意采用了轻量但诚实的工程验证策略：pointwise、candidate ranking、pairwise preference 三条线各跑 30 条样本，不做伪装成全量实验的大规模结论，而是把第三天定位为 inference 闭环验收日。pointwise 通过 main_infer.py 输出到 outputs/beauty_qwen_pointwise/predictions/test_raw.jsonl；candidate ranking 通过 main_rank.py 输出到 outputs/beauty_qwen_rank/predictions/rank_predictions.jsonl；pairwise preference 通过 main_pairwise.py 输出到 outputs/beauty_qwen_pairwise/predictions/pairwise_predictions.jsonl。三类任务都成功落盘，没有出现大面积空响应、结构断裂或关键字段缺失，这说明多任务推理主线已经从接口测试进入正式执行状态。

为了避免“跑出来了但没有结构化记录”，我今天还新增了 main_summarize_multitask_inference.py，并生成 outputs/summary/week5_day3_inference_summary.csv，统一统计三类任务的 sample_count、avg_latency、parse_success_rate、avg_response_length、failure_ratio、empty_output_ratio。当前这轮轻量闭环的结果是：pointwise_yesno 在 30 条样本上平均 latency 为 2.1667 秒，parse_success_rate 为 1.0，平均响应长度为 263.3；candidate_ranking 在 30 条样本上平均 latency 为 6.0547 秒，parse_success_rate 为 1.0，平均响应长度为 598.3667；pairwise_preference 在 30 条样本上平均 latency 为 2.7445 秒，parse_success_rate 为 1.0，平均响应长度为 356.7333。三条任务线的 failure_ratio 均为 0，empty_output_ratio 也均为 0。这组数据不应被理解为模型性能结论，而应被理解为语言接口、backend 调用、parser 约束和 prediction schema 在真实推理链上的一次稳定性验收。

从方法视角看，今天最关键的进展不是“多跑了两个脚本”，而是项目的任务对象正式改变了。到今天结束，pointwise 已经不再是整个项目唯一的决策任务，而是被稳定保留为 diagnostic layer；pairwise preference 不再只是 prompt 练习，而是已经形成可正式推理的 preference-mechanism 任务；candidate ranking 也不再停留在样本构造与 parser 自测层，而是已经形成能够直接落地产生排序预测文件的主决策任务。这意味着 version pony 的 Part5 三层结构第一次在同一套工程里形成了可执行闭环：pointwise 负责不确定性观测，pairwise 负责局部偏好机制，candidate ranking 负责列表决策输出。这个变化对后续 week6 极其关键，因为没有今天这一步，所谓“多任务 compare”就只是概念对比，而不是可复现的实验比较。

我今天也同步更新了 README，但刻意保持了科研项目应有的克制写法。README 现在已经明确说明，项目正从单一 pointwise pipeline 过渡到 unified multi-decision framework，并且在高层结构上解释了 main_infer.py、main_rank.py、main_pairwise.py 三个入口的职责划分，同时指出仓库现在可以稳定产出 pointwise、ranking、pairwise 三种 prediction artifacts。这里我没有把 README 写成按 day1/day2/day3 罗列的项目日志，因为那样一方面不适合作为公开科研仓库的主页说明，另一方面也容易把工程实现细节过度暴露。README 的职责仍然是概念框架与高层结构说明，详细过程留给 pony2 日报与内部 summary 文件。

当然，今天的工作并不代表多任务系统已经“完成”，我也不打算把轻量闭环包装成最终结果。当前仍然有几个后续要严肃推进的点。第一，今天三条任务都只是完成了 inference 落盘，还没有进入统一 evaluation 层，因此 week5 第四天仍然需要把 ranking 与 pairwise 的指标接口接起来，形成真正的 compare 基础。第二，candidate ranking 当前的平均 latency 明显高于 pointwise 与 pairwise，这是合理现象，因为 prompt 更长、结构约束更复杂，但这也提示后面做更大规模实验时需要关注吞吐与成本。第三，今天的验证规模是 30 样本级别的工程闭环验收，不是全量实验，因此不能把 parse_success=1.0 误读为“大规模稳定性已经完全解决”；后面仍然需要在更长响应、不同模型和不同 domain 上继续观察格式漂移。第四，pairwise 与 ranking 目前虽然已经能推理，但评价口径和与 uncertainty 的连接方式尚未正式打通，这正是 week6 之后真正的研究重点。

基于今天的结果，我对第三天的验收结论是明确成立的：我的项目已经不再只有单一 pointwise inference，而是具备 pointwise、pairwise、candidate ranking 三条并行推理主线；三条主线都能在真实 backend 上完成 prompt 构造、API 调用、parser 解析、prediction 落盘与 summary 聚合；项目从“多任务概念设计”正式进入“多任务执行系统”阶段。换句话说，week5 到今天为止，version pony 已经完成了从单任务样本构造系统到多任务推理系统的实质升级，后续可以进入统一评价层与 uncertainty 跨任务迁移层，而不再停留在结构设想层面。
